# AutoRecruit 项目面试问答

这份文档用于面试中介绍 AutoRecruit 项目经验。建议回答时遵循一个固定结构：

```text
业务问题 -> 技术方案 -> 关键取舍 -> 结果和复盘
```

不要把项目说成通用 ATS 或完整 SaaS 平台。这个项目的定位更准确：CLI-first、本地优先、面向招聘侧重复执行场景的自动化工具，重点解决多招聘平台候选人搜索、简历抓取、JD 匹配评分、报告交付，以及基于 JD 和已确认历史对话的候选人 RAG 问答。

## 一分钟项目介绍

### Q：请简单介绍一下这个项目。

短答：

AutoRecruit 是一个 TypeScript CLI 招聘自动化工具，支持 51job、猎聘和智联三个招聘平台。它把平台搜索、候选人抓取、简历解析、JD 匹配评分、报告导出、邮件发送，以及后续的 RAG 候选人问答串成一套可复跑、可审计、按平台隔离的数据流程。

展开答法：

这个项目一开始解决的是招聘侧重复操作问题：同一个职位要在多个平台搜索候选人，打开简历，提取信息，再根据 JD 做匹配评分和报告。后来扩展了 RAG 能力，让候选人可以围绕职位提问，系统只基于当前职位的 JD 和已确认的招聘方历史答复回答，避免模型自由发挥。

整体上我把它拆成几层：

- 平台适配层：处理 51job、Liepin、Zhilian 的页面差异和候选人提取规则。
- 编排层：处理 CLI 参数、单职位、多平台、批量职位、搜索订阅等流程。
- 存储层：把 JD、简历、评分、运行结果和 RAG 事实源保存在本地 JSON/JSONL。
- AI 层：包括 JD 解析、简历评分和 RAG 问答。
- 运营诊断层：包括 RAG inspect、doctor、review、metrics、ops 和 regression。

## 项目价值和业务背景

### Q：这个项目解决了什么实际问题？

短答：

它把招聘过程中的重复浏览器操作和重复判断自动化，同时保留本地可审计数据，减少人工重复劳动，并让后续候选人问答可以复用已确认的职位事实。

展开答法：

招聘场景里有几个痛点：

- 多个平台入口不一致，筛选条件、已查看状态和简历详情页行为都不同。
- 候选人是否已经处理过，需要跨运行去重，否则容易重复打开和重复评分。
- JD 和简历匹配判断有大量重复性，但直接让模型评分又需要结构化校验和失败兜底。
- 候选人常问的问题高度重复，比如薪资、地点、学历、语言、福利，如果每次人工回答效率低。
- 历史对话里有些事实只有招聘方确认后才可信，不能把候选人猜测或未确认答复当成事实。

所以项目把流程做成本地 CLI，而不是直接做 Web 平台。这样更适合当前场景：可控、可复跑、易调试，并且可以保留每个职位目录下的完整产物。

### Q：为什么选择 CLI-first、本地优先，而不是一开始做 Web 系统？

短答：

因为核心风险在平台适配、浏览器自动化稳定性、数据闭环和 RAG 事实治理，不在前端交互。CLI-first 能更快验证稳定流程，同时本地数据更容易审计和重建。

展开答法：

这个项目的复杂度主要来自招聘平台页面变化、登录态、筛选条件、候选人详情打开和简历解析。如果一开始做 Web 系统，会把权限、用户管理、部署、前端状态也引入进来，反而分散重点。

本地优先有几个好处：

- 数据可审计：每个职位目录都有 `jd.json`、`resumes/`、`scores/`、`results/` 和 `rag/`。
- 失败可复盘：有文本快照、DOM 快照和失败评分产物。
- 索引可重建：Qdrant 只是索引，本地 JSONL 才是事实源。
- 运行可控：适合有人值守的招聘平台自动化，尤其是登录和平台风控相关流程。

## 架构设计

### Q：项目整体架构怎么拆分？

短答：

按职责拆成 CLI 编排、平台适配、浏览器工具、搜索筛选、解析评分、RAG、存储和报告几个模块。平台差异只放在 `src/platforms/`，共享流程放在 `src/index.ts` 和公共 helper 里。

展开答法：

核心边界是：

- `src/index.ts`：CLI 参数解析和主流程编排。
- `src/platforms/`：51job、Liepin、Zhilian 平台适配。
- `src/browser/`：浏览器会话、登录态、候选列表和详情页公共逻辑。
- `src/search/`：筛选发现、筛选输入校验、搜索订阅。
- `src/parsers/`：JD 解析。
- `src/scoring/`：简历评分和 Zod 校验。
- `src/rag/`：RAG 事实源、chunk、embedding、Qdrant、混合检索、问答、诊断、评测、运营指标。
- `src/storage/`：本地 JSON 持久化。

这个拆分的重点是保持平台 adapter 边界。比如 Zhilian 的 Vue props 提取、Liepin 的找人入口、51job 的订阅搜索入口，都不应该污染主流程。

### Q：为什么要保持平台 adapter 边界？

短答：

因为三个平台的页面结构、搜索入口、筛选状态和简历详情行为差异很大。把平台差异隔离后，主流程可以保持统一，新增平台或修复某个平台不会影响其他平台。

展开答法：

项目里有很多平台特有规则：

- 51job 保存入口从订阅卡片进入，候选卡片要用 `no_interested_<id>` 附近结构定位，但它不是详情入口。
- Liepin 需要进入招聘方找人页面，默认隐藏已查看，且交互节奏更慢。
- Zhilian 在 `/app/search` 上使用保存标签或直接关键词，详情是 modal，还需要复制转给同事的分享链接。

如果这些逻辑都写在主流程里，后期维护会非常困难。现在主流程只关心“打开搜索、提取候选人、打开详情、解析简历、评分、导出”，具体怎么做交给平台 adapter。

### Q：`--platform all` 为什么要求固定顺序并且失败就停止？

短答：

这是为了让批量运行具备可预测性。固定顺序是 51job、Liepin、Zhilian；如果其中一个失败立即停止，避免后续平台在错误上下文里继续产生不完整结果。

展开答法：

招聘自动化不是纯离线任务，里面有浏览器登录态、平台页面状态和运行产物。如果一个平台失败还继续跑，最终结果可能让人误以为所有平台都完成了。固定顺序和 fail-fast 可以让运行结果更明确，出错后也更容易复盘。

## 浏览器自动化和平台适配

### Q：这个项目里浏览器自动化最大的难点是什么？

短答：

难点不是简单点击页面，而是保证每个平台在不同历史状态下都能进入正确搜索条件，并且把空结果、详情失败、提取失败、评分失败区分清楚。

展开答法：

招聘平台页面有很多不稳定因素：

- 登录态可能失效。
- 搜索页可能保留上一次筛选条件。
- 保存搜索标签和直接输入关键词的行为不同。
- 已查看筛选状态会影响候选人列表。
- 候选卡片不一定有稳定链接或 ID。
- 详情可能是新页、弹窗或 modal。
- 空结果不是失败，明确空结果应该算成功的 0 候选人运行。

所以项目里用了几个策略：

- 平台级 storage state，避免多个平台混用登录态。
- 每个平台显式确认搜索条件，例如可见 `关键词：<keyword>`。
- 默认排除已查看候选人，`--include-viewed true` 时显式取消对应筛选。
- 只把成功保存简历的候选人标记 seen，详情打开失败保持可重试。
- 对详情打开使用总 deadline，而不是多个固定等待叠加。

### Q：如何处理候选人去重？

短答：

每个职位目录有 `seen-ids.json`。只有成功捕获并保存简历的候选人才会标记 seen，详情打开失败或提取失败不会标记，保证下次可重试。

展开答法：

这个设计避免了两个问题：

- 如果一打开候选卡片就标记 seen，后面详情页失败会导致候选人永久丢失。
- 如果评分失败就回滚 seen，又会导致同一个简历重复抓取。

项目的语义是：成功保存简历后先 seen，再评分。评分失败只保存 `status: failed` 的评分产物，不影响简历已捕获的事实。

### Q：为什么要支持 saved 和 direct 两种搜索来源？

短答：

saved 复用平台已有订阅或快捷搜索，适合日常稳定运行；direct 直接输入关键词和筛选条件，适合临时或批量配置化搜索。

展开答法：

saved 模式依赖平台已经保存的搜索入口，优点是和招聘方日常使用习惯一致。direct 模式则绕过保存标签，打开搜索页后清空旧条件，输入关键词，应用外部 JSON 筛选配置，再开始抓取。

direct 模式的关键是不能在筛选条件部分失败后继续抓候选人，否则会从错误候选池里抓数据。所以项目规定任何筛选条件跳过或应用失败都是 run error。

### Q：搜索筛选发现功能的价值是什么？

短答：

它把平台上的筛选项和可选值导出成 catalog，让后续 direct 搜索可以用结构化 JSON 配置筛选条件，减少硬编码和人工试错。

展开答法：

招聘平台的筛选项很多，而且有下拉框、级联城市、行业树、薪资范围、年龄范围等不同控件。项目通过 discovery 命令把可见筛选项、选项值和层级结构导出来，再用输入校验命令检查用户提供的 filter input 是否能被平台支持。

这相当于给平台筛选做了一层本地 schema 和 catalog，使批量搜索更可控。

## LLM 解析和评分

### Q：项目里哪些地方用了大模型？

短答：

主要三处：JD 解析、简历评分、RAG 最终回答生成。RAG embedding 当前默认不用 OpenAI，而是本地 FastEmbed 服务。

展开答法：

- JD 解析：把原始 JD 解析成结构化字段，如职位、薪资、职责、硬性要求、优先条件。
- 简历评分：根据 JD 和结构化简历输出评分 JSON。
- RAG 回答：在检索到可信上下文后，让模型生成候选人可读的自然语言回答。

这三处都不是让模型随便输出。JD 和评分会用 schema 校验，RAG 会先检索可信事实，没命中可信资料时直接拒答，不让模型猜。

### Q：如何降低模型输出不稳定的风险？

短答：

用结构化 schema 校验、失败产物持久化、事实检索约束和 no-answer 机制。

展开答法：

项目里的策略包括：

- JD 解析和评分用 Zod schema 校验。
- 模型调用失败或输出不符合 schema 时，保存失败评分产物，而不是吞掉错误。
- RAG 回答前先检索 JD 和 verified 历史答复。
- 如果没有可信资料或分数低于阈值，不调用模型编造，直接返回未说明。
- 回答日志会记录 confidence、sources、noAnswerReason，后续可以 review 和 feedback。

## RAG 方案

### Q：为什么要给招聘项目加 RAG？

短答：

因为候选人会反复问围绕职位的细节问题，而答案应该来自当前职位 JD 和已确认历史答复，不应该让模型凭常识回答。

展开答法：

传统做法是每次让招聘方人工查 JD 或历史聊天记录。RAG 的价值是把“可信资料”整理成可检索事实库。候选人问问题时，系统先查当前职位资料，再让模型基于资料回答。

这个项目的 RAG 不追求通用知识问答，而是职位级、平台级隔离的问答：

```text
当前平台 + 当前职位 + 当前有效 JD/verified 对话
```

### Q：当前 RAG 的完整链路是什么？

短答：

资料入库 -> chunk -> embedding -> 写入 Qdrant -> 提问时 hybrid 检索 -> 融合重排 -> 置信度判断 -> 生成答案或拒答 -> 写 answer log。

展开答法：

链路如下：

1. 保存职位 JD，或导入历史对话。
2. 生成 RAG source，JD 是事实源；历史对话只有 `role=recruiter` 且 `verified=true` 才进入事实 chunk。
3. 把 source 切成 chunks。
4. 调用 embedding provider，把 chunk 变成向量，并缓存到 `rag/embeddings.jsonl`。
5. 把向量和 metadata 写入 Qdrant collection。
6. 候选人提问时，用 metadata 过滤当前 platform、jobKey 和 active chunk。
7. 同时做 dense 向量召回和本地 BM25 关键词召回。
8. 对两路结果融合、轻量重排，选出 topK 上下文。
9. 如果上下文不够可信，直接 no-answer。
10. 如果可信，再调用回答模型生成中文候选人回答。
11. 对保存职位写入 `answer-logs.jsonl`，用于审计、反馈和评测。

### Q：为什么选择 Qdrant + 本地 embedding？

短答：

Qdrant 适合持久化向量检索和 metadata filtering，本地 embedding 降低外部依赖和成本，也更适合中文职位文本的本地化运行。

展开答法：

选择 Qdrant 是因为它提供稳定的向量集合、payload metadata 和过滤能力。对本项目来说，metadata filter 很关键，因为必须保证只查当前平台和当前职位，不能串到其他职位。

embedding 方面，当前默认是本地 HTTP 服务，模型是 `BAAI/bge-small-zh-v1.5`。这样有几个好处：

- 不需要每次 embedding 都调用外部 API。
- 对本地批量重建索引更稳定。
- 可以通过 `rag/embeddings.jsonl` 缓存向量，减少重复计算。
- 如果以后要切回 OpenAI embedding，也保留了 provider 接口。

### Q：Qdrant 是事实源吗？

短答：

不是。Qdrant 只是可重建索引，本地 JSONL 才是事实源。

展开答法：

项目明确把本地文件作为 source of truth：

- `rag/sources.jsonl`
- `rag/chunks.jsonl`
- `rag/conversations/`
- `rag/embeddings.jsonl`
- `rag/index-manifest.json`
- `rag/answer-logs.jsonl`

Qdrant 可以删掉后用 `rag:rebuild` 从本地 chunks 重建。这样设计是为了避免向量库成为唯一数据副本，也方便审计和迁移。

### Q：什么是 metadata filtering？为什么重要？

短答：

metadata filtering 是检索时按 `platform`、`jobKey`、source 类型、active 状态等字段过滤，只允许当前职位的资料参与回答。

展开答法：

招聘问答里最危险的问题之一是串职位。例如候选人问 A 职位薪资，系统却引用了 B 职位的待遇。metadata filtering 可以在向量召回之前就限制搜索范围，保证召回池只包含当前职位资料。

这和 hybrid 检索不冲突。dense 召回和 keyword 召回都必须在同一个 metadata 范围里执行，最后再融合排序。

### Q：混合检索解决了什么问题？

短答：

向量检索擅长语义相近问题，关键词检索擅长精确数字、地点、福利名词。混合检索把两者结合起来，降低漏召回。

展开答法：

招聘问题里有很多精确事实：

- `8-12k`
- `13薪`
- `世博天地`
- `大专`
- `3年以上`

纯向量检索可能能理解语义，但对数字和短词不一定稳定。BM25 关键词检索能补足这类精确匹配。所以默认使用 `RAG_RETRIEVAL_MODE=hybrid`，先分别召回，再融合和重排。

### Q：重新排序是怎么理解的？

短答：

重新排序不是再调用一个大模型 reranker，而是基于轻量特征把更可能回答问题的 chunk 排前面。

展开答法：

当前项目的 rerank 是轻量排序，重点考虑：

- dense 召回分。
- keyword 命中情况。
- 来源可信度，JD 和 verified 对话。
- 问题和 chunk 的词面重合。
- 是否命中关键事实短语。

这样做的好处是成本低、可控、易测试。如果后续质量要求更高，可以把 rerank provider 抽象出来，接入专门的 rerank 模型。

### Q：RAG 如何避免编造？

短答：

如果没有可信 JD 或 verified 对话 chunk 命中，或者最高可信分低于阈值，系统不调用回答模型，直接返回未说明。

展开答法：

防编造分三层：

1. 事实源治理：只有 JD 和 verified recruiter 对话进入事实检索。
2. 检索阈值：没有足够相关的 chunk 时直接 no-answer。
3. Prompt 约束：回答模型只能基于上下文，不允许补充上下文没有的信息。

这比单纯在 prompt 里写“不要编造”更可靠，因为系统层面会阻止低可信问题进入生成阶段。

### Q：为什么只有 verified 的招聘方回复能进入事实库？

短答：

因为候选人发言可能是猜测，未确认招聘方答复也可能不准确。只有被业务确认的招聘方事实才能影响未来候选人回答。

展开答法：

历史对话会完整保存，便于审计。但进入检索事实库的条件是：

```text
role = recruiter
verified = true
```

这样可以同时满足两个需求：

- 保留完整上下文。
- 控制哪些内容能成为未来回答依据。

### Q：如何支持一个职位持续加入新对话？

短答：

按 conversationId 和 turn id 增量导入，对同一 conversation 的 turn 去重合并，然后只重建受影响的 conversation sources、chunks 和 verified fact vectors。

展开答法：

批量导入时支持 JSON、items wrapper 和 JSONL。每条 item 有 platform、jobKey 或 keyword、conversationId，以及 turns。系统会根据稳定 turn id 合并，如果没有 id，会用 role、createdAt、content 生成稳定 id。

增量导入后，只把 verified recruiter turns 转成事实 chunk。这样一个职位随着沟通增加，可以持续补充事实，而不会重新解析 JD 或污染其他职位。

## 质量保障和运维

### Q：RAG 功能如何做测试？

短答：

有单元测试、离线基线、召回评测、答案评测、回归套件和 doctor 诊断。

展开答法：

关键命令包括：

- `test:rag:offline`：不依赖 OpenAI 和 Qdrant，用内存向量库和确定性假 embedding 跑离线召回基线。
- `rag:eval`：检查某些问题能否召回预期资料。
- `rag:answer-eval`：真实走问答链路，检查答案包含预期事实且不包含禁止内容。
- `rag:regression`：把召回评测和答案评测串成一键回归。
- `rag:doctor`：检查单职位 manifest、source、chunk、Qdrant、embedding 配置和检索命中。
- `rag:ops`：汇总 doctor、review 和 metrics，适合日常运营巡检。

### Q：为什么需要 answer log、review、feedback 和 metrics？

短答：

因为 RAG 不是一次写完就结束，需要持续验证真实候选人问题的回答质量。

展开答法：

每次保存职位的 RAG 问答会写入 `answer-logs.jsonl`，记录问题、答案、confidence、sources 和 noAnswerReason。人工可以用 `rag:feedback` 标记正确或错误，并给错误类型，例如 wrong_fact、bad_source、low_relevance。

这些反馈可以进入：

- `rag:review`：找出未审核、低置信、拒答或错误回答。
- `rag:metrics`：统计审核率、正确率、错误率、拒答率、低置信率。
- `rag:export-answer-eval`：把已审核正确的日志导成回归用例。

这形成了从真实问题到回归测试的闭环。

### Q：如何排查一个职位 RAG 回答不准？

短答：

先看 `rag:doctor`，再用 `rag:inspect --question` 看 dense、keyword、hybrid 各自命中了什么，最后检查 sources/chunks 和 answer log。

展开答法：

排查顺序：

1. `rag:doctor` 看索引、manifest、embedding provider、Qdrant 是否健康。
2. `rag:inspect --question` 看检索有没有召回正确 chunk。
3. 如果 dense 没命中但 keyword 命中，考虑 hybrid 参数或 chunk。
4. 如果都没命中，检查 JD 或 verified 对话里是否真的有答案。
5. 如果检索命中了但回答错，检查 prompt 或答案生成逻辑。
6. 如果是低置信拒答，检查阈值和 chunk 质量。

### Q：本地 embedding 服务如何运行和排障？

短答：

开发时可以用 `npm run rag:embedding:local`，本机长期运行用 LaunchAgent `com.autorecruit.embedding`。健康检查看 `http://127.0.0.1:8011/health`。

展开答法：

当前本机运行路径是：

```text
~/.local/share/autorecruit/embedding/
~/.local/bin/autorecruit-local-embedding
~/Library/LaunchAgents/com.autorecruit.embedding.plist
~/.local/var/log/autorecruit/local-embedding.log
~/.local/var/log/autorecruit/local-embedding.err.log
```

之所以放在 `~/.local/share`，是因为 macOS LaunchAgent 对 `~/Documents` 目录可能有隐私权限限制，后台服务从独立运行目录启动更稳定。

常用检查：

```bash
rtk launchctl list | rtk rg 'com\\.autorecruit\\.(qdrant|embedding)'
rtk curl -sSf http://127.0.0.1:6333/collections
rtk curl -sSf http://127.0.0.1:8011/health
```

## 产品化接口

### Q：`rag:api` 是完整网关吗？

短答：

不是。它是内部产品化 HTTP 接口，可以做 API key 校验，但不是完整多租户网关。

展开答法：

当前 API 提供：

- `GET /health`
- `POST /v1/rag/answer`
- `POST /v1/rag/conversations`

它复用 CLI 的 RAG 问答和对话导入逻辑，保证 jobKey、verified 规则、metadata filtering、自动索引和 answer log 行为一致。

但它不做完整 SaaS 能力：

- 多租户。
- RBAC。
- 限流。
- 集中审计。
- 监控告警。
- 前端管理后台。

生产接入时，应该放在内网或业务网关后面，由上游系统处理身份、权限和限流。

## 设计取舍

### Q：为什么不用数据库保存所有数据，而是用本地 JSON/JSONL？

短答：

当前项目是本地优先的 CLI 工具，JSON/JSONL 更容易审计、迁移、diff 和离线复盘。Qdrant 只是可重建索引。

展开答法：

本地文件有几个优势：

- 每个职位目录天然隔离。
- 出错时可以直接打开文件查看。
- 易于做离线脚本、回归 fixture 和数据迁移。
- 不依赖中心数据库服务。

如果后续产品化成多人系统，可以把本地 store 抽象替换成数据库，但当前阶段 JSON/JSONL 更符合使用场景。

### Q：如果后续要做成 SaaS，你会怎么演进？

短答：

保留现有领域逻辑，把存储、权限、任务调度、审计和前端管理独立产品化。

展开答法：

演进方向：

- 把 `JobStore` 从本地 JSON 抽象到数据库。
- 把 Qdrant 从本机服务迁移到托管或集群部署。
- 增加多租户和 RBAC。
- 给 `rag:api` 前面加网关、限流、审计和监控。
- 增加任务队列，把浏览器抓取和 RAG 索引构建异步化。
- 提供前端管理后台，展示职位、问答日志、审核和指标。
- 对平台适配层增加更明确的契约测试和回放测试。

### Q：如果要新增一个招聘平台，怎么做？

短答：

实现新的 platform adapter，保证它满足统一的搜索、候选提取、详情打开和简历解析契约，同时不要改主流程语义。

展开答法：

步骤：

1. 增加平台类型和 registry。
2. 定义平台登录态文件和入口 URL。
3. 实现 saved/direct 搜索准备逻辑。
4. 明确已查看筛选的默认状态和 include-viewed 行为。
5. 实现候选人 DOM/API 提取。
6. 实现详情打开和简历文本/DOM 快照提取。
7. 添加平台特有测试和 smoke 命令。
8. 确认 `--platform all` 顺序和失败语义不被破坏。

## 可能被追问的技术细节

### Q：RAG 的 embedding 模型需要下载到本地吗？

短答：

当前默认本地 embedding 需要下载 FastEmbed 模型到本机缓存；第一次启动服务会下载，之后复用缓存。回答模型不下载到本地，仍走 OpenAI 兼容 API。

展开答法：

当前 embedding provider 是 `local-http`，服务使用 `BAAI/bge-small-zh-v1.5`。它把文本转成向量后返回给 TypeScript 侧。RAG 的最终回答仍然由 `OPENAI_MODEL` 指定的模型生成，`RAG_MODEL` 只是可选覆盖。

### Q：为什么 `OPENAI_MODEL` 和 `RAG_MODEL` 要分开？

短答：

为了允许 RAG 问答单独使用不同模型。但当前本机配置通常只设置 `OPENAI_MODEL`，让 RAG 默认继承它。

展开答法：

有些场景中 JD 解析、评分和 RAG 问答可能需要不同模型。例如评分需要更稳定 JSON，问答需要更自然表达。保留 `RAG_MODEL` 可以单独覆盖。但为了降低配置复杂度，当前推荐只配 `OPENAI_MODEL`。

### Q：如何保证临时 JD 问答不污染持久化 RAG？

短答：

`--jd-question` 如果使用临时 `--jd` 或 `--jd-file`，只从本次 JD 做轻量检索回答，不创建职位记录，不写持久化索引，也不追加 answer log。

展开答法：

这很重要，因为候选人可能临时测试一段 JD。如果把临时 JD 写入职位目录，会污染正式数据。项目区分了保存职位路径和临时 JD 路径：已有 jobKey 走持久化 RAG，临时 JD 只在内存里检索和回答。

### Q：为什么离线基线不用真实 embedding？

短答：

为了让 CI 或本地快速冒烟不依赖外部服务。它用确定性假 embedding 和内存向量库，只验证 chunk、索引和召回逻辑。

展开答法：

真实 embedding 和 Qdrant 更接近生产，但也带来服务依赖和不稳定性。离线基线的目标是快速发现代码层退化，不是评估真实语义效果。真实链路可以用 `rag:baseline` 和 `rag:regression` 跑。

## 行为面试答法

### Q：这个项目里你最有挑战的部分是什么？

建议答法：

最有挑战的是把“浏览器自动化的不确定性”和“RAG 问答的事实可靠性”都控制住。浏览器侧要处理平台状态残留、登录态、空结果、详情打开失败和候选人去重；RAG 侧要处理事实源治理、metadata 隔离、混合召回、低置信拒答和质量反馈闭环。

我不是只把流程跑通，而是把失败语义和可恢复路径明确下来。例如只有成功保存简历才 seen；Qdrant 不是唯一事实源；未验证对话不进入事实库；没有可信上下文时不调用模型编造。这些设计让系统后续可维护。

### Q：你在项目里的主要贡献怎么表达？

建议答法：

我主要负责把项目从单纯招聘自动化扩展成具备 RAG 问答和运营闭环的系统。具体包括：

- 设计职位级 RAG 事实库和 Qdrant 可重建索引。
- 接入本地 embedding 服务并保留 provider 扩展。
- 实现 hybrid 检索、metadata filtering、轻量 rerank 和 no-answer 机制。
- 建立对话导入规则，只让 verified 招聘方事实进入检索。
- 增加 answer log、feedback、review、metrics、ops 和 regression。
- 完善文档和 agent 指引，保证后续维护者知道运行和排障方式。

### Q：项目有什么不足？

建议答法：

当前不足主要在产品化层面，而不是核心流程：

- 还是 CLI-first，没有完整 Web 管理后台。
- `rag:api` 是内部接口，不是完整多租户网关。
- 浏览器自动化仍然依赖招聘平台页面稳定性。
- Rerank 目前是轻量规则，没有接入专门 rerank 模型。
- 指标和告警还需要接入外部监控系统。

如果继续演进，我会优先做管理后台、任务队列、统一审计、告警，以及更系统的在线评测。
