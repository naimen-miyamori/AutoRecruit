**目标**

在前端新增一个“CLI 助手/智能助手”页面，让用户用中文和大模型对话，描述想执行的招聘自动化操作。系统将对话内容转换成受控的 CLI 功能草稿，用户确认后再通过现有后端任务队列执行。

核心原则：不让大模型直接执行任意 shell 命令，而是让它生成结构化任务参数，再复用现有 `resume-capture`、`batch`、`search-subscription`、`login-refresh`、`rag-ops` 等已校验任务入口。

**改动范围**

后端：

- 新增 `src/server/cli-assistant.ts`
- 扩展 `src/server/types.ts`
- 调整 `src/server/routes.ts`
- 必要时把现有 task normalizer 从 `routes.ts` 拆到 `src/server/task-normalizers.ts`
- 复用 `src/server/task-queue.ts`
- 使用现有 OpenAI 配置：`OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_MODEL`

前端：

- 新增“智能助手”页面
- 修改 `frontend/src/api.ts`
- 修改 `frontend/src/App.tsx` 导航和路由
- 可新增组件，例如：
  - `frontend/src/components/AssistantChat.tsx`
  - `frontend/src/components/AssistantDraftPanel.tsx`
  - `frontend/src/components/AssistantTaskPreview.tsx`

测试与验证：

- 增加后端 API 测试
- 增加前端构建验证
- 用浏览器验证页面交互和错误显示

**功能设计**

1. 前端提供中文对话框

   用户可以输入：

   - “帮我在猎聘搜索 Java 后端，筛选本科以上，3-5 年经验”
   - “刷新智联登录”
   - “跑一下 51job 的搜索订阅”
   - “用这个岗位 JD 执行三平台搜索”
   - “检查 RAG 运维指标”
   - “问一下这个岗位是否接受远程办公”

2. 后端调用大模型解析意图

   大模型只允许输出结构化 JSON，例如：

   ```ts
   type AssistantDraft =
     | {
         kind: "resume-capture";
         input: ResumeCaptureTaskInput;
         missingFields: string[];
         warnings: string[];
         argvPreview: string[];
       }
     | {
         kind: "batch";
         input: BatchTaskInput;
         missingFields: string[];
         warnings: string[];
         argvPreview: string[];
       }
     | {
         kind: "search-subscription";
         input: SearchSubscriptionTaskInput;
         missingFields: string[];
         warnings: string[];
         argvPreview: string[];
       }
     | {
         kind: "login-refresh";
         input: LoginRefreshTaskInput;
         missingFields: string[];
         warnings: string[];
         argvPreview: string[];
       }
     | {
         kind: "rag-ops";
         input: RagOpsTaskInput;
         missingFields: string[];
         warnings: string[];
         argvPreview: string[];
       }
     | {
         kind: "rag-answer";
         input: RagAnswerInput;
         missingFields: string[];
         warnings: string[];
       };
   ```

3. 用户确认前不执行

   聊天只生成“待执行草稿”。前端展示：

   - 操作类型
   - 平台
   - 关键词
   - JD 输入方式
   - 筛选条件
   - 是否包含已查看
   - 是否发送邮件
   - 预计 CLI 参数
   - 风险提醒
   - 缺失字段

4. 用户点击“确认执行”后才提交

   确认后：

   - 普通任务调用现有 `api.submitTask(kind, body)`
   - RAG 问答调用现有或新增 `api.askRag()`
   - 后端再次运行 normalizer 校验
   - 校验通过后进入 `TaskQueue`
   - 前端跳转或展示任务详情链接

**实施步骤**

1. 梳理现有任务校验

   - 阅读 `src/server/routes.ts`
   - 找出现有 normalize 函数：
     - `normalizeResumeCaptureTask`
     - `normalizeBatchTask`
     - `normalizeSearchSubscriptionTask`
     - `normalizeLoginRefreshTask`
     - `normalizeRagOpsTask`
   - 如果它们目前是 route 内部私有函数，拆到 `src/server/task-normalizers.ts`
   - `routes.ts` 和新的助手确认接口共同复用这些校验逻辑

2. 新增助手类型

   在 `src/server/types.ts` 增加：

   - `AssistantMessage`
   - `AssistantChatRequest`
   - `AssistantChatResponse`
   - `AssistantDraft`
   - `AssistantConfirmRequest`
   - `AssistantConfirmResponse`
   - `AssistantActionKind`

   允许的 action 只包含现有受控功能：

   - `resume-capture`
   - `batch`
   - `search-subscription`
   - `login-refresh`
   - `rag-ops`
   - `rag-answer`

3. 新增后端助手模块

   新建 `src/server/cli-assistant.ts`，负责：

   - 组装 system prompt
   - 调用 OpenAI 兼容接口
   - 要求模型输出严格 JSON
   - 用 Zod 校验模型输出
   - 对缺失字段生成澄清问题
   - 生成 `argvPreview`
   - 标记风险项

   明确禁止模型输出或执行：

   - 任意 shell 命令
   - `rm`、`git reset` 等破坏性命令
   - 未在 allowlist 的 npm script
   - 直接写入未知文件
   - 绕过 task normalizer 的 CLI 参数

4. 新增 API 路由

   在 `src/server/routes.ts` 增加：

   - `POST /api/assistant/chat`

     输入用户消息和历史上下文，返回 assistant 回复和任务草稿。

   - `POST /api/assistant/confirm`

     输入已确认草稿，再次经过 normalizer 校验，然后：

     - enqueue 现有任务
     - 或执行 RAG 问答
     - 返回 taskId 或 answer

5. 前端 API 封装

   在 `frontend/src/api.ts` 增加：

   - `chatWithAssistant(payload)`
   - `confirmAssistantDraft(payload)`

6. 新增前端页面

   新增“智能助手”页面，布局建议：

   - 左侧：聊天区
   - 右侧：任务草稿预览
   - 顶部：快捷操作入口
   - 底部：确认执行按钮

   页面文案全部使用中文。

   快捷入口包括：

   - 执行单岗位搜索
   - 执行批量任务
   - 执行搜索订阅
   - 刷新平台登录
   - RAG 问答
   - RAG 运维检查

7. 草稿编辑能力

   生成草稿后允许用户修改关键字段：

   - 平台
   - 关键词
   - JD 文本或 JD 文件路径
   - 搜索来源 saved/direct
   - 筛选输入文件
   - 是否包含已查看
   - 邮件收件人
   - 猎聘转发联系人
   - 搜索订阅文件
   - RAG 运维文件和策略文件

   修改后前端重新提交草稿校验，不直接执行。

8. 风险确认

   以下操作需要额外确认提示：

   - `platform=all`
   - `includeViewed=true`
   - 发送邮件
   - 猎聘转发简历
   - 批量任务
   - RAG rebuild 或 fail-on-issue
   - 搜索订阅保存

9. 保持 CLI 约束

   特别保证：

   - 搜索订阅模式不允许解析 JD、创建岗位、抓简历、评分、发邮件
   - `applicationFilterInputFile` 只用于 direct 普通简历抓取
   - `--platform all` 顺序仍是 `51job -> liepin -> zhilian`
   - JD/RAG 问答不打开浏览器、不抓简历、不写生产日志
   - 不让模型生成绕过现有 CLI 规则的参数

10. 前端 UI 优化

   新页面风格和现有后台保持一致：

   - 中文导航名称：“智能助手”
   - 少用大面积空框
   - 使用紧凑的操作面板
   - 任务草稿用结构化信息块展示
   - 错误、警告、缺失字段清晰显示
   - 确认按钮只有在校验通过后可用

**交付结果**

完成后应具备：

- 前端可和大模型中文对话
- 大模型可把用户意图转换成 CLI 功能草稿
- 用户确认后可直接触发现有 CLI 后台任务
- 所有执行仍走现有 task queue 和 normalizer
- 不支持任意 shell，避免破坏项目状态或绕过业务约束
