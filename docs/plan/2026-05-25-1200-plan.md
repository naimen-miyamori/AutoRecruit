**目标**

降低“候选人详情页打开后等待很久”的体感和实际耗时：

1. 将简历抓取阶段和模型评分阶段解耦：先打开所有新候选人详情页、提取并保存简历，全部完成后再逐份评分。
2. 评分失败不影响已抓取简历的持久化和去重；模型连接失败时仍生成失败评分 artifact，并继续导出/发送邮件。
3. 将同一详情页打开过程中的多个串行等待改成单个总等待预算，减少 popup/current-page/modal 判断导致的重复等待。

**改动范围**

主要涉及：

- `src/index.ts`
  - 拆分候选人处理流程：抓取保存简历、统一评分、生成 run result。
- `src/browser/resume-detail.ts`
  - 优化 51job 详情页打开等待。
- `src/platforms/liepin-adapter.ts`
  - 优化猎聘详情页 popup/current page 等待。
- `src/platforms/zhilian-adapter.ts`
  - 优化智联 modal 详情页等待和重复 ready 检查。
- `src/config.ts`
  - 可选新增详情页总超时配置，例如 `PLAYWRIGHT_RESUME_DETAIL_TIMEOUT_MS`。
- `src/scripts/test-scoring-run-semantics.ts`
  - 增加/调整 orchestration 测试。
- 可能涉及 `src/scripts/test-experimental:*` 相关测试断言，但尽量不改公共数据结构。

**实施步骤**

1. 重构候选人处理函数

   当前 `processCandidate()` 同时做：
   - 打开详情页
   - 提取简历
   - 保存简历
   - 调模型评分
   - 保存评分 artifact

   计划拆成两个阶段：

   - `captureCandidateResume(...)`
     - 只负责打开候选人详情页、解析简历、保存 resume/snapshot/domSnapshot。
     - 成功保存简历后返回 `{ candidateId, captured: true, markAsSeen: true }`。
     - 打开或提取失败返回 `{ candidateId, captured: false, markAsSeen: false, failureReason }`。
     - `finally` 中关闭详情页，确保评分阶段不会继续停留在候选人详情页。

   - `scoreCapturedResumes(...)`
     - 在所有新候选人的抓取循环结束后执行。
     - 从 `store.listStoredResumes(platform, jobKey)` 读取已保存简历，并按本次抓取成功的 candidateId 过滤。
     - 每份简历单独调用 `scoreResumeAgainstJobRef.fn(job, resume)`。
     - 成功保存 `status: 'success'` score artifact。
     - 失败保存 `status: 'failed'` score artifact，error 写入模型连接/超时/解析失败原因。
     - 默认串行评分，先不引入并发，避免触发模型限流和增加恢复复杂度。

2. 调整 `runResumeCaptureFlow(...)`

   新流程：

   - 打开搜索页并提取候选人列表。
   - 计算 `newCandidates`。
   - 对 `newCandidates` 逐个执行 `captureCandidateResume(...)`。
   - 抓取阶段结束后，立即根据抓取成功的候选人更新 `seen-ids.json`。
   - 再调用 `scoreCapturedResumes(...)` 对本次抓取成功的简历评分。
   - 最后生成 `RunResult`：
     - `newCandidateIds`: 本次发现的新候选人 ID。
     - `scoredCandidates`: 评分成功的候选人 ID。
     - `failedCandidates`: 抓取失败 + 评分失败。
       - 抓取失败不生成 score artifact，保持 retryable。
       - 评分失败生成 failed score artifact，因此导出/邮件可以展示失败原因。
   - 保存 run result。
   - 返回现有结构，尽量保持外部调用兼容。

3. 保持邮件和导出行为

   `runSinglePlatform(...)` 中仍然在 `runResumeCaptureFlow(...)` 返回后执行：

   - `exportJobResults(...)`
   - `sendJobReport(...)`

   预期行为：

   - 没有新候选人：继续走 no-new-candidates 邮件。
   - 有新候选人且评分成功：正常报告。
   - 有新候选人但模型连接失败：每个已抓取候选人都有 failed score artifact，报告/邮件显示评分失败原因。
   - 候选人详情页打开失败：不 mark seen，下次可重试；如果本次没有任何评分 artifact，导出/邮件仍可能失败，这与当前 extraction failure 语义一致。

4. 优化 51job 详情页等待

   当前问题点：

   - 先等 `#no_interested_<id>` 最长 10 秒。
   - 然后依次尝试姓名、职位、公司、整卡点击。
   - 每次点击后都先等 popup 3 秒，再检查当前页/正文。
   - 多个 fallback 串行叠加，失败路径耗时明显。

   计划：

   - 新增详情页总预算，例如默认 20 秒：
     - `PLAYWRIGHT_RESUME_DETAIL_TIMEOUT_MS`
     - fallback 默认值 `20000`
   - `waitForResumeDetailContent(page, candidateId)` 改为支持 `timeoutMs` 或 `deadline`。
   - 每次点击后并发观察：
     - popup 是否出现
     - 当前页 URL 是否变化
     - 当前页正文是否已出现简历内容
   - 不再固定先等 popup 3 秒后才看当前页。
   - 所有 locator click/content wait 都使用同一个剩余时间预算。

5. 优化猎聘详情页等待

   当前问题点：

   - 点击候选人链接后先等 popup 最长 10 秒。
   - 如果实际在当前页跳转，会白等 popup timeout。

   计划：

   - 点击后使用同一个详情页 deadline 并发等待：
     - popup page
     - 当前页 URL 变化
     - 当前页 ready
   - 任一分支成功后立即进入对应页面 ready 检查。
   - `waitForLiepinPageReady()` 接收可选 timeout/deadline，避免不同层重复使用固定 15 秒等待。

6. 优化智联 modal 等待

   当前问题点：

   - 打开详情前关闭旧 modal 可能等待 5 秒。
   - `openResumeDetail()` 等一次 `waitForZhilianResumeDetailReady()`。
   - `parseResumeDetail()` 又等一次同样 ready 条件。

   计划：

   - `closeExistingZhilianResumeModal()` 先做快速存在性检查；没有 modal 时不走 5 秒 click timeout。
   - 详情打开阶段使用总 deadline 等待 modal + 正文标记。
   - `parseResumeDetail()` 改为短确认或直接读取 modal 文本，避免重复完整 15 秒等待。
   - 保留登录态检查，防止 session 失效时误读搜索列表。

7. 测试计划

   增加或调整测试，重点覆盖 orchestration，而不是只测单个函数：

   - `runResumeCaptureFlow` 会先抓完所有简历，再开始评分。
     - 用两个候选人 mock。
     - 记录调用顺序，断言两个 `saveCandidateResume` 都发生在第一次 `scoreResumeAgainstJob` 之前。
   - 评分失败不会影响简历保存和 seen ids。
     - mock 模型抛错。
     - 断言 resume 已保存、seen ids 已更新、failed score artifact 已保存。
   - 抓取失败仍不 mark seen。
     - mock `openResumeDetail` 抛错。
     - 断言不保存 resume、不保存 score artifact、不写 seen id。
   - 有评分失败 artifact 时，导出和邮件仍可执行。
     - 覆盖模型连接失败后的 report path。
   - 多平台语义不变。
     - 51job 仍走 Crawl4AI/源码提取。
     - Liepin/Zhilian 正常 CLI 抓取仍走 adapter parser。

8. 验证步骤

   修改完成后按顺序运行：

   - `npm run typecheck`
   - `npm run test:scoring`
   - `npm run test:export`
   - `npm run test:maintenance`
   - `npm run test:experimental:liepin`
   - `npm run test:experimental:zhilian`
   - `npm run test`

   如需做真实页面验证，再手动运行已有 smoke 命令：

   - `npm run smoke:liepin -- --keyword "<keyword>"`
   - `npm run smoke:zhilian -- --keyword "优衣库"`
   - 51job 用正常 `npm run dev -- --platform 51job ...`

**预期结果**

- 候选人详情页只承担打开、解析、保存职责，不再被模型 API 慢响应拖住。
- 模型失败会集中体现在评分阶段和报告中，不会让每个详情页看起来都卡住。
- 详情页打开失败路径的等待时间受单个总 timeout 控制，不再由多个 fallback timeout 叠加。
- 已成功抓取的简历会被标记 seen，避免模型失败导致下次重复打开同一批候选人详情页。
