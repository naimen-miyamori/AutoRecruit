**目标**

调整 Boss 自动聊天分流：

- 已经聊过：不打开简历、不判断 JD、不转发、不发送话术、不换电话；仅提取本次红点对应的候选人新回复，写入运行结果和总结邮件。
- 此前未聊过：继续执行现有简历解析、JD 判断、转发、常用语和换电话流程。
- “新回复”定义为点击前未读快照对应的候选人消息。文本保留原意并规范空白；图片、附件等记录为明确的类型占位符。

**改动范围**

- `src/platforms/boss-chat.ts`：识别消息发送方并提取未读候选人回复。
- `src/index.ts`：在打开会话后立即分流，确保已聊过分支不进入 JD 流程。
- `src/types/job.ts`：增加新回复结构、审查状态和汇总计数。
- `src/storage/job-store.ts`：调整红点会话去重和失败重试逻辑。
- `src/reporting/boss-chat-summary.ts`：增加已聊过候选人的新回复章节。
- `src/server/task-queue.ts`：暴露新汇总计数。
- Boss 测试、编排测试、AGENTS、项目说明及计划文档同步更新。

**实施步骤**

1. 扩展未读会话快照，区分真正有红点的会话和从失败记录恢复的会话，保留点击前的 `unreadCount`。

2. 打开会话后等待以下状态同时就绪：
   - `.chat-conversation` 对应目标会话。
   - `.base-info-single-container` 已完成 hydration。
   - `.chat-message-list` 已加载当前会话消息。

3. 统一实现消息方向识别：
   - 优先读取 Boss Vue 消息数据中的发送方字段。
   - 使用已验证的招聘方消息 class/属性作为 DOM 兜底。
   - 排除系统提示和时间分隔元素。
   - 不持久化完整聊天历史。

4. 从候选人消息中按时间顺序选取最后 `unreadCount` 条，形成 `newCandidateReplies`：
   - 文本消息保存规范化文本。
   - 图片、简历、附件、语音等保存类型和可见描述。
   - 保存可用的消息 ID、时间和类型，缺失时不伪造。
   - 无法可靠提取时记录明确错误，不能生成空白“新回复”。

5. 扩展审查结构：
   - 新增状态 `follow_up_reply`。
   - `BossChatReviewItem` 增加 `newCandidateReplies`。
   - 运行汇总增加 `followUpConversations` 和 `newReplyMessages`。
   - 新字段保持可选，兼容已有历史 JSON。

6. 重排 `runBossAutoChat`：
   - 先打开红点会话，写入候选人身份、`previousChat` 和新回复。
   - `previouslyChatted=true` 时直接设置 `follow_up_reply`，停止该候选人的后续处理。
   - 该分支不得读取或评分简历、判断硬要求、转发、发送消息或申请换电话。
   - `previouslyChatted=false` 时才读取岗位 JD、转发配置并继续现有流程。

7. 调整缺少 JD/转发配置的处理：
   - 当前“缺少配置时保持会话未打开”与“打开后才能判断是否聊过”存在顺序冲突。
   - 本方案以新需求为准：先打开并判断。
   - 若属于首次沟通但缺少 JD 或转发配置，记录为可重试失败，不加入已审查 ID；后续配置补齐后从失败记录恢复。
   - 已聊过候选人不要求存在 JD 或转发配置，也能进入总结。

8. 修正跨运行去重：
   - 新红点不能因为会话 ID 曾经被审查过而直接跳过。
   - 真正有新红点的会话始终按本次未读快照处理。
   - `reviewedConversationIds` 只防止无新红点的失败恢复项被重复执行。
   - 已聊过回复成功写入后标记本次事件完成；未来再次出现红点仍可处理。

9. 调整总结邮件：
   - 新增“已聊过候选人的新回复”独立章节。
   - 展示候选人姓名、ID、岗位、回复条数及每条回复内容。
   - 符合与不符合章节只包含首次沟通且完成 JD 判断的候选人。
   - 汇总区展示首次沟通数、此前已聊过数、跟进回复会话数和新回复消息数。
   - Markdown 内容进行换行和特殊字符处理，避免候选人文本破坏邮件结构。

10. 同步项目约束，明确历史沟通判断从“仅供记录”升级为流程分流依据，并记录缺少配置时新的可重试语义。

**验证步骤**

1. DOM 测试覆盖招聘方消息、候选人消息、系统消息、连续多条未读消息和非文本消息。

2. 验证 `unreadCount=2` 时只提取最后两条候选人新消息，不包含招聘方历史话术。

3. 验证已聊过分支：
   - 总结包含新回复原文。
   - 不调用简历详情、JD 评分、硬要求判断、转发、聊天或换电话。
   - 即使没有岗位 JD 或转发配置也能完成回复汇总。

4. 验证首次沟通分支完整保持现有物业电工规则，包括上海籍确认逻辑。

5. 验证同一会话未来再次出现红点时不会被历史 `reviewedConversationIds` 错误跳过。

6. 验证首次沟通但缺少配置时保持可重试，补齐配置后能够恢复。

7. 验证旧审查 JSON 没有新字段时仍能读取和生成总结。

   - `rtk npm test`
   - `rtk npm run build`
   - `rtk npm run web:build`
