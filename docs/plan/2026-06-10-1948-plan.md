### 2. Secret 处理

1. 先轮换 Stitch API key。
2. 替换 `.env` 中的旧值，不要打印 `.env` 内容：

   ```bash
   rtk sh -c 'if rg -q "^STITCH_API_KEY=" .env; then echo stitch-key-line-present; else echo stitch-key-line-missing; fi'
   ```

3. 用户提供新 key 后，仅更新 `.env` 的 `STITCH_API_KEY=` 行。
4. 确认 `.env` 仍被 git ignore：

   ```bash
   rtk git check-ignore -v .env
   ```

5. 确认真实 key 没有写入仓库文件：

   ```bash
   rtk sh -c 'if rg -q "AQ\\\\." . --glob "!.env"; then echo possible-secret-marker; else echo no-secret-marker-outside-env; fi'
   ```

   如果出现 `possible-secret-marker`，必须定位并清理，但不要在回复中打印真实 key。

### 6. 低风险端到端试跑

先不要跑真实批量任务。选择一个低风险路径：

1. API 参数校验试跑：
   - 提交一个会被拒绝的非法组合，确认任务不会进入执行队列。
   - 例如 `applicationFilterInputFile` 未配 `searchSource=direct`。

2. Search-subscription 低风险试跑：
   - 使用 standalone search-subscription 模式。
   - 确认不会解析 JD、不会捕获简历、不会评分、不会发邮件。

3. RAG temporary JD 试跑：
   - 在 RAG 页面输入临时 JD 和问题。
   - 确认不会创建 job、不会写 persistent RAG index、不会 append `answer-logs.jsonl`。

4. 如需真实 resume-capture，只选择一个平台和一个低风险关键词，避免 `platform all` 直接跑三平台。

## 风险和注意事项

- 不要把 Stitch key 提交到仓库。
- 不要重新全局安装 Stitch skills。
- 不要把平台自动化逻辑搬到前端。
- `platform all` 顺序必须继续依赖现有 CLI registry：
  1. `51job`
  2. `liepin`
  3. `zhilian`
- 任务队列必须保持单并发，避免多个浏览器自动化流程污染状态。
- RAG no-answer 不能调用模型编造答案。
- temporary JD 问答不能创建 job、不能写 persistent RAG index、不能 append `answer-logs.jsonl`。
- `.agents/skills/` 是否提交是产品/团队协作决策；技术上它已经实现“当前项目范围生效”。
