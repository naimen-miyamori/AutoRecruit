目标：
- 将 51job、Liepin、Zhilian 三个平台的“打开搜索页 + 候选列表 ready”统一到同一套 search wait contract。
- 三个平台都从主流程接收同一个总 deadline，避免各自内部多个等待阶段串行叠加。
- 保留平台差异：51job 以 DOM 卡片为主，Liepin/Zhilian 可使用 API fallback，但 API 不能拖慢已可用的 DOM 结果。
- 避免 51job 因列表容器先出现而提前误判 0 候选。

改动范围：
- `src/config.ts`
  - 增加搜索页总等待配置。
- `src/platforms/types.ts`
  - 增加统一搜索等待参数类型。
  - 更新 adapter 方法签名。
- `src/index.ts`
  - 在主流程创建一次 search deadline，并传给搜索打开和候选列表提取。
- `src/platforms/51job-adapter.ts`
  - 转发 deadline 到 51job 浏览器模块。
- `src/browser/subscribe-search.ts`
  - 51job 搜索入口打开使用共享 deadline。
- `src/browser/candidate-list.ts`
  - 51job 候选列表 ready 使用共享 deadline，并加入空列表稳定窗口。
- `src/platforms/liepin-adapter.ts`
  - Liepin 搜索入口、搜索壳页、候选列表 DOM/API 等待使用共享 deadline。
- `src/platforms/zhilian-adapter.ts`
  - Zhilian 接入同一 deadline 参数，保持 DOM-first，不让 API fallback 拖慢 DOM 结果。
- 测试脚本
  - `src/scripts/test-liepin-adapter.ts`
  - `src/scripts/test-zhilian-adapter.ts`
  - 51job 候选列表相关测试，如现有测试不足则在主流程/候选列表语义测试中补充。
- 文档
  - `AGENTS.md`
  - `项目说明文档.md`

统一语义：
1. `openSubscribeSearch + extractCandidateList` 共用同一个总 deadline。
2. DOM 候选卡出现：立即 ready，返回候选。
3. 明确空结果文案出现：立即 ready，返回 0。
4. 只有列表容器可见但候选卡为 0：不能立即返回 0，必须满足稳定窗口。
5. API fallback 只用于：
   - DOM 尚未出现候选时提供候选来源；
   - DOM 候选缺少安全详情 URL 时补充 URL。
6. DOM 已能提供完整候选结果时，不等待 API。
7. deadline 耗尽时输出 diagnostics，不再进入额外固定 10s/15s 等待。

实施步骤：

1. 增加搜索等待配置
   - 在 `src/config.ts` 的 `playwright` 配置中增加：
     - `searchPageTimeoutMs: getOptionalNumberEnv('PLAYWRIGHT_SEARCH_PAGE_TIMEOUT_MS', 20000)`
     - `emptyResultsStableMs: getOptionalNumberEnv('PLAYWRIGHT_EMPTY_RESULTS_STABLE_MS', 2000)`
     - `apiFallbackTimeoutMs: getOptionalNumberEnv('PLAYWRIGHT_API_FALLBACK_TIMEOUT_MS', 3000)`
   - `PLAYWRIGHT_SEARCH_PAGE_TIMEOUT_MS` 表示从开始打开平台搜索入口到候选列表 ready 的总预算。
   - `PLAYWRIGHT_EMPTY_RESULTS_STABLE_MS` 只用于“容器为空”的稳定窗口。
   - `PLAYWRIGHT_API_FALLBACK_TIMEOUT_MS` 只用于 DOM 已有部分结果但需要 API 补充时的短等待。

2. 定义统一接口
   - 在 `src/platforms/types.ts` 增加：
     - `SearchWaitOptions`
     - 字段：
       - `deadline?: number`
   - 更新 `PlatformAdapter`：
     - `openSubscribeSearch(page, keyword, options?: SearchWaitOptions): Promise<Page>`
     - `extractCandidateList(page, options?: SearchWaitOptions): Promise<{ candidates: CandidateListItem[] }>`
   - 保持参数可选，避免破坏 smoke/debug 脚本和既有测试。

3. 主流程创建并传递 search deadline
   - 在 `src/index.ts` 的 `runResumeCaptureFlow()` 中：
     - 在调用 `platformAdapter.openSubscribeSearch()` 前创建：
       - `const searchDeadline = Date.now() + config.playwright.searchPageTimeoutMs`
     - 调用：
       - `platformAdapter.openSubscribeSearch(session.page, searchKeyword, { deadline: searchDeadline })`
       - `extractCandidateListRef.fn(searchPage, { deadline: searchDeadline })` 或 `platformAdapter.extractCandidateList(searchPage, { deadline: searchDeadline })`
   - 确保 51job 的 extraction boundary 能接收可选参数，或在 51job adapter 内直接调用支持 options 的 `collectCandidateList()`。

4. 51job 搜索入口打开使用共享 deadline
   - 在 `src/browser/subscribe-search.ts`：
     - 给 `openSubscribeSearch()` 增加 `options?: SearchWaitOptions` 或本地等价类型。
     - 内部使用 `deadline = options.deadline ?? Date.now() + config.playwright.searchPageTimeoutMs`。
     - 将以下固定 10s 改成 `remainingTime(deadline)`：
       - popup wait
       - current page URL navigation wait
       - search link fallback `page.goto()`
     - `findSubscriptionCard()`、`waitForAuthenticatedSubscribeReady()`、`clickSearchTrigger()` 接收并使用同一个 deadline。
   - 保留订阅卡片 settle delay，但必须受剩余时间约束；剩余时间不足时跳过或缩短。

5. 51job 列表 ready 使用共享 deadline
   - 在 `src/browser/candidate-list.ts`：
     - `waitForCandidateResultsReady(page, options?)`
     - `collectCandidateList(page, options?)`
   - 改造当前 `waitForKnownCandidateListState()`：
     - 不再固定 `firstCandidateCard.waitFor(... timeout: 10000)`。
     - 改成循环或 `waitForFunction` 风格，用剩余 deadline 检查状态。
   - 状态判断：
     - `candidateCardSelector` 数量大于 0：`candidates`
     - body 命中 `emptyResultsPattern`：`empty`
     - `.virtual_list` 可见且 loading 不可见且候选卡持续为 0 超过 `emptyResultsStableMs`：`empty-list`
   - 删除或弱化当前 catch 中 `resultListVisible && candidateCardCount === 0` 直接成功返回的逻辑，避免 deadline 后把未稳定空列表误判成 0。
   - diagnostics 增加：
     - `emptyTextMatched`
     - `stableEmptyListObservedMs`
     - `deadlineRemainingMs`

6. Liepin 搜索入口和列表使用共享 deadline
   - 在 `src/platforms/liepin-adapter.ts`：
     - `openSubscribeSearch(page, keyword, options?)`
     - `extractCandidateList(page, options?)`
   - `waitForLiepinPageReady()` 接收 deadline，不再默认创建多个局部 15s 窗口。
   - 搜索页 ready 改为：
     - 必须完成登录态校验。
     - 搜索 shell ready 可通过 DOM 文案或 initial-data API 任一证明。
     - 不强制 initial-data 和 search shell 串行都等待满 15s。
   - `clickLiepinQuickSearchTag()` 使用剩余 deadline。
   - `waitForLiepinSearchResumesApi()` 增加 timeout 参数：
     - 默认使用剩余 deadline；
     - DOM 已有候选但只需补 URL 时，使用 `min(remainingTime(deadline), apiFallbackTimeoutMs)`。

7. Liepin 候选提取 DOM-first
   - `extractCandidateList(page, { deadline })` 逻辑调整为：
     - attach API observer。
     - 启动 API promise，但不立即 await。
     - 立即尝试 `collectLiepinCards(page)`。
     - DOM candidates > 0 且 URL 安全：立即返回。
     - DOM candidates > 0 但 URL 不安全：短等 API fallback 补 URL；短等结束仍返回 DOM candidates。
     - DOM candidates = 0：在剩余 deadline 内等待 DOM/API/明确空结果任一状态。
     - API candidates > 0：返回 API candidates。
     - 明确空结果：返回空数组。
     - deadline 耗尽：返回当前 candidates 或抛 diagnostics，按现有“可见空列表是成功 0”的项目语义处理，但不能隐式等待额外 15s。

8. Zhilian 接入统一 deadline
   - 在 `src/platforms/zhilian-adapter.ts`：
     - `openSubscribeSearch(page, keyword, options?)`
     - `extractCandidateList(page, options?)`
   - `openZhilianRecruiterHome()`、`waitForZhilianRecruiterShell()`、`clickSavedZhilianQuickSearchTag()` 使用剩余 deadline。
   - 保持现有 DOM-first：
     - `extractCandidateList()` attach API observer。
     - 启动 API promise，但先等 recruiter shell 和 DOM collect。
     - DOM candidates > 0：立即返回，不 await API。
     - DOM candidates = 0：只在剩余 deadline 内等待 API fallback 或明确空结果。
   - API fallback timeout 使用共享 deadline，不再有独立长等待。

9. 明确三平台差异但统一外部行为
   - 51job：
     - 无 API fallback。
     - 依赖候选卡、明确空文案、稳定空列表。
   - Liepin：
     - DOM 和 API 双来源。
     - API 主要补详情 URL 或兜底候选。
   - Zhilian：
     - DOM-first，API fallback 仅在 DOM 没候选时使用。
   - 外部行为统一：
     - 主流程只知道同一个 search deadline。
     - `extractCandidateList()` 要么返回候选数组，要么返回空数组，要么抛带 diagnostics 的错误。

10. 测试补充
   - 51job：
     - 搜索入口 popup/navigation 使用传入 deadline 的剩余时间。
     - `.virtual_list` 先出现，候选卡稍后出现：返回候选，不判定 0。
     - 明确空文案出现：快速返回 0。
     - 空列表稳定窗口满足：返回 0。
     - deadline 耗尽且无稳定空列表：抛 diagnostics。
   - Liepin：
     - `openSubscribeSearch + extractCandidateList` 使用同一 deadline。
     - DOM candidates 完整时，不等待 `search-resumes` API。
     - DOM candidates URL 不安全时，只短等 API fallback。
     - API 先返回 candidates 时可返回 API。
     - 不再出现 initial-data、shell、search-resumes 多个 15s 串行等待。
   - Zhilian：
     - DOM candidates > 0 时不等待 API。
     - DOM candidates = 0 时在 deadline 内等待 API fallback。
     - quick-search tag 和 recruiter shell 使用传入 deadline。
   - 主流程：
     - `runResumeCaptureFlow()` 创建一次 deadline，并传给 open/extract 两个阶段。
