**目标**

实现一个只读的“三平台筛选项发现器”，用 Playwright 进入 51job、liepin、zhilian 的搜索条件页，自动展开高级筛选，逐项交互已发现控件，提取可见选项、级联子选项、输入框类型和失败原因，输出平台独立的 filter catalog JSON。

不修改现有候选人抓取、评分、导出、邮件流程。Crawl4AI 不作为主流程依赖，只可作为后续辅助 HTML 文本解析；筛选发现主力使用 Playwright。

**改动范围**

新增共享模块：

- `src/search/filter-catalog.ts`
  定义 catalog 类型、控件类型、探测状态、选项结构。
- `src/search/filter-discovery.ts`
  通用 Playwright 发现引擎：扫描、入队、交互、diff、抽取选项、恢复状态。
- `src/search/filter-dom.ts`
  DOM 指纹、可见元素扫描、变化容器识别、选项节点抽取等纯工具。
- `src/scripts/discover-search-filters.ts`
  CLI 入口，用于生成筛选项 catalog。

新增平台模块：

- `src/platforms/51job-filters.ts`
- `src/platforms/liepin-filters.ts`
- `src/platforms/zhilian-filters.ts`

这些模块只放平台特定的：
- 如何进入搜索条件页
- 如何展开更多筛选
- 哪些区域应忽略
- 哪些文案代表筛选容器
- 特殊弹窗/级联/输入框处理规则

扩展 adapter 契约：

- 在 `src/platforms/types.ts` 增加可选方法：
  `discoverSearchFilters?(page, options): Promise<SearchFilterCatalog>`

新增 npm script：

- `discover:filters`
  示例：
  `rtk npm run discover:filters -- --platform zhilian --keyword "优衣库"`
  `rtk npm run discover:filters -- --platform all --keyword "优衣库"`

输出路径：

- `data/<platform>/filter-catalog/latest.json`
- `data/<platform>/filter-catalog/<timestamp>.json`

**Catalog 类型设计**

核心结构：

```ts
export interface SearchFilterCatalog {
  platform: SupportedPlatform;
  keyword: string;
  capturedAt: string;
  pageUrl: string;
  filters: SearchFilterDefinition[];
  failures: SearchFilterDiscoveryFailure[];
  stats: {
    discoveredControls: number;
    inspectedControls: number;
    optionsExtracted: number;
    failedControls: number;
    unknownControls: number;
  };
}

export interface SearchFilterDefinition {
  key: string;
  label: string;
  controlType:
    | 'singleSelect'
    | 'multiSelect'
    | 'cascadeSelect'
    | 'rangeInput'
    | 'textInput'
    | 'remoteSuggest'
    | 'datePreset'
    | 'toggle'
    | 'unknown';
  valueShape: 'string' | 'string[]' | 'range' | 'boolean' | 'object';
  status:
    | 'discovered'
    | 'inspected'
    | 'optionsExtracted'
    | 'noOptions'
    | 'failed'
    | 'unknownControl';
  options?: SearchFilterOption[];
  selectorHints: SearchFilterSelectorHint[];
  inputPlaceholder?: string;
  childrenLazy?: boolean;
  message?: string;
}
```

每个控件必须有 `status`。无法识别或无法交互的控件不能静默跳过，必须进入 `failures` 或标记为 `unknownControl`。

**实施步骤**

1. 建立只读 catalog 类型
   - 新增 `src/search/filter-catalog.ts`
   - 定义 filter、option、failure、stats、control fingerprint 类型。
   - 不接入现有主流程。

2. 实现 DOM 扫描工具
   - 新增 `src/search/filter-dom.ts`
   - 扫描可见可交互元素：
     `button`、`input`、`select`、`textarea`、`[role=combobox]`、`[role=listbox]`、`[role=menuitem]`、`[aria-haspopup]`、`[aria-expanded]`、常见 checkbox/radio。
   - 为每个元素生成指纹：
     近邻标题、文本、placeholder、role、aria、tagName、坐标、简化 DOM path。
   - 过滤明显非筛选区：
     导航、候选人卡片、分页、广告、用户菜单、结果列表操作按钮。

3. 实现通用交互状态机
   - 新增 `src/search/filter-discovery.ts`
   - 逻辑：
     1. 进入平台搜索条件页。
     2. 展开高级筛选。
     3. 扫描控件并入队。
     4. 对每个控件保存 before snapshot。
     5. 点击/聚焦/打开控件。
     6. 等待 UI 稳定。
     7. 计算 after snapshot。
     8. 只从新增或变化容器中抽取选项。
     9. 判断控件类型。
     10. 恢复现场。
     11. 记录 status。
   - 设置保护参数：
     最大控件数、最大级联深度、每级最大选项数、单控件超时、全局 deadline。

4. 实现级联菜单探测
   - 对识别为 `cascadeSelect` 的控件做 BFS。
   - 默认限制：
     - `maxDepth = 3`
     - `maxOptionsPerLevel = 50`
   - 每点击一个父级，只抓其变化出来的子级容器。
   - 记录 `childrenLazy: true`，避免宣称穷举远程数据。

5. 实现输入框和远程建议识别
   - 对普通输入框记录 `textInput`。
   - 对输入后触发建议列表的控件记录 `remoteSuggest`。
   - 不穷举学校、公司、专业、技能等无限集合。
   - 可用少量 probe 值做类型确认，例如：
     - 城市类：`上海`
     - 学校类：`上海`
     - 技能类：`销售`
   - probe 值必须可配置，默认只做安全探测，不提交搜索。

6. 实现平台适配
   - 51job：
     复用 `prepare51jobSearchConditionPage` 进入搜索页。
     增加 51job 专属展开和忽略规则。
   - liepin：
     复用 `prepareLiepinSearchConditionPage`。
     避免破坏 `隐藏已查看` 的现有候选人抓取约束；筛选发现独立运行，不进入候选人提取。
   - zhilian：
     复用 `prepareZhilianSearchConditionPage`。
     特别处理高级搜索弹窗、城市/职能/行业类多级选择。

7. 新增 CLI 脚本
   - 新增 `src/scripts/discover-search-filters.ts`
   - 参数：
     - `--platform <51job|liepin|zhilian|all>`
     - `--keyword "<关键词>"`
     - `--output <path>` 可选
     - `--max-depth <n>` 可选
     - `--max-options-per-level <n>` 可选
     - `--include-remote-probes true|false` 可选，默认 `false`
   - `--platform all` 顺序保持：
     1. `51job`
     2. `liepin`
     3. `zhilian`

8. 保存结果
   - 新增或扩展 storage 方法，把 catalog 写入：
     `data/<platform>/filter-catalog/latest.json`
     `data/<platform>/filter-catalog/<timestamp>.json`
   - catalog 只存结构化筛选元数据，不存完整网页 HTML，不存候选人数据。

9. 测试
   - 单元测试：
     - DOM 扫描能识别 button/input/select/role 控件。
     - diff 只返回新增或变化容器。
     - option extraction 不把候选人卡片、分页、导航误判为筛选项。
     - unknown 控件会记录 `unknownControl` 或 failure。
   - 平台 adapter 测试：
     - 51job/liepin/zhilian 都能挂载 `discoverSearchFilters`。
     - `--platform all` 顺序正确。
     - 单个平台失败时 all 流程立即停止并抛错。
   - CLI 测试：
     - 缺少 `--keyword` 报错。
     - 输出 JSON 包含 `filters`、`failures`、`stats`。
     - 不调用 JD parsing、resume capture、score、export、email。

10. 现场验证命令
   - 类型检查：
     `rtk npm run typecheck`
   - 单元测试：
     `rtk npm run test`
   - 平台专项测试：
     `rtk npm run test:experimental:liepin`
     `rtk npm run test:experimental:zhilian`
   - 真实发现流程：
     `rtk npm run discover:filters -- --platform 51job --keyword "优衣库"`
     `rtk npm run discover:filters -- --platform liepin --keyword "优衣库"`
     `rtk npm run discover:filters -- --platform zhilian --keyword "优衣库"`
     `rtk npm run discover:filters -- --platform all --keyword "优衣库"`

**验收标准**

- 三个平台都能生成 `filter-catalog/latest.json`。
- 每个发现到的可交互控件都有明确状态，不允许静默丢弃。
- 弹窗、下拉、级联、输入框、远程建议至少能分类。
- 对级联控件能抓到至少一级选项，并在限制内继续探测子级。
- 对远程建议控件不宣称完整枚举，只记录为 `remoteSuggest`。
- 发现流程不创建 job record，不解析 JD，不抓候选人，不评分，不导出报告，不发邮件。
- 现有全平台候选人流程行为不变。
