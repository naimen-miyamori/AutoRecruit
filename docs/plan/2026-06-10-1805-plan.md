**目标**

为 `autorecruit` 增加内部前端运营控制台，用于提交招聘任务、查看任务状态、浏览岗位/候选人/评分结果、使用 RAG 问答和进入运营诊断工具。Google Stitch 参与设计系统、页面方案、variants 和 React 草案生成，但不作为生产运行时依赖。

**Stitch 配置与密钥交接**

- Stitch 文档入口：`https://stitch.withgoogle.com/docs`
- Stitch 实际文档资源：`https://app-companion-430619.appspot.com/docs/.../index.html`
- 访问代理：`http://127.0.0.1:7897`
- API key 获取方式：
  - 执行本计划的新对话开始时，用户需要重新提供 Stitch API key，或确认本机已配置 `STITCH_API_KEY`。
  - 计划中不明文保存真实 key。
  - 执行时使用：
    ```bash
    export STITCH_API_KEY="<user-provided-stitch-api-key>"
    export HTTPS_PROXY="http://127.0.0.1:7897"
    ```
  - 前端代码、提交记录、设计文档、测试 fixture、计划文本中不得出现真实 key。
  - 因真实 key 已在当前对话中出现过，正式实施前建议轮换一次 key。

**Stitch 已确认能力**

- Stitch 本体：
  - prompt 生成 UI
  - App/Web 设备类型
  - design modes
  - 1-5 个 variants
  - 局部编辑 screen
  - theme 编辑
  - prototype 预览
  - 下载 HTML code 和 image
- MCP：
  - remote MCP
  - API key 或 OAuth 认证
  - 工具包括 `create_project`、`list_projects`、`get_project`、`list_screens`、`get_screen`、`generate_screen_from_text`、`edit_screens`、`generate_variants`、`create_design_system`、`update_design_system`、`list_design_systems`、`apply_design_system`、`upload_design_md`、`create_design_system_from_design_md`
- SDK：
  - 包名：`@google/stitch-sdk`
  - 环境变量：`STITCH_API_KEY`、`STITCH_HOST`
  - 支持项目、screen、variants、artifact 下载、图片/HTML 上传、design system 管理
- DESIGN.md：
  - YAML front matter 存 design tokens
  - Markdown body 存设计说明
  - 覆盖 colors、typography、spacing、rounded、components、do/don’t
  - 可用 `@google/design.md` 做 lint、contrast 检查、token export
- Agent Skills：
  - 官方仓库：`google-labs-code/stitch-skills`
  - 重点 skills：`enhance-prompt`、`taste-design`、`generate-design`、`manage-design-system`、`code-to-design`、`extract-design-md`、`extract-static-html`、`upload-to-stitch`、`react-components`、`stitch-loop`

**改动范围**

- 新增后端产品 API：`src/server/`
- 新增任务队列和任务状态持久化：`data/runtime/tasks/`
- 复用并适度拆分 `src/index.ts` 的 CLI 运行逻辑
- 新增前端应用：`frontend/`
- 新增设计资料：`docs/design/`
- 新增 Stitch 设计 brief、DESIGN.md、mock 数据、设计评审记录
- 不修改平台 adapter 边界
- 不让前端直接写 `data/` 业务文件
- 不把 51job、Liepin、Zhilian 自动化逻辑搬到前端

**前端产品范围**

1. **Dashboard**
   - 最近任务
   - 平台状态
   - 最近运行结果
   - 成功、失败、零候选统计

2. **Run Job**
   - 单任务提交
   - 支持 `platform`：`51job`、`liepin`、`zhilian`、`all`
   - 支持 `keyword`
   - 支持 `jd` / `jdFile`
   - 支持 `includeViewed`
   - 支持 `searchSource`：`saved` / `direct`
   - 支持 `applicationFilterInputFile`
   - 支持 `email` / `cc`
   - 支持 `liepinForwardContact`

3. **Tasks**
   - 任务列表
   - 任务详情
   - 运行日志
   - 错误原因
   - 任务输入摘要
   - 任务输出 summary

4. **Jobs**
   - 按平台浏览历史 job
   - 查看 JD、normalized job、运行记录

5. **Job Detail**
   - 岗位信息
   - run results
   - candidate list
   - export path
   - email delivery 状态

6. **Candidate Detail**
   - 简历结构
   - 评分维度
   - 失败评分 artifact
   - Zhilian share link
   - 原始 snapshot 只读入口

7. **RAG**
   - 选择 platform/jobKey
   - 提问
   - 展示 answer、confidence、sources、no-answer reason
   - 支持 conversation ingest 的后续入口

8. **Ops**
   - RAG doctor
   - RAG review
   - RAG metrics
   - filter catalog 查看
   - session/login 诊断入口

**实施步骤**

1. **接入 Stitch**
   - 按 Stitch 文档安装或启用 `google-labs-code/stitch-skills`。
   - 配置 Stitch MCP 或 SDK。
   - 使用 `STITCH_API_KEY` 和 `HTTPS_PROXY`。
   - 先验证 `list_projects`、`list_design_systems`。

2. **建立设计系统**
   - 用 `taste-design` 生成适合招聘运营台的 `DESIGN.md`。
   - 用 DESIGN.md lint 规则检查 token、contrast、typography、section order。
   - 设计原则：高密度、表格优先、中文友好、状态清晰、避免营销页风格。

3. **生成 Stitch 页面方案**
   - 用 `enhance-prompt` 优化每个页面 prompt。
   - 用 `generate-design` / `generate_screen_from_text` 生成核心页面。
   - 用 `generate_variants` 为 Dashboard、Run Job、Candidate Detail、RAG 生成 2-5 个变体。
   - 用 `edit_screens` 做局部迭代。
   - 用 `get_screen` 下载 screenshot 和 HTML 作为实现参考。

4. **实现后端 API**
   - 新增：
     - `src/server/http.ts`
     - `src/server/routes.ts`
     - `src/server/task-queue.ts`
     - `src/server/job-read-model.ts`
   - 从 `src/index.ts` 抽出可复用运行函数，保持 CLI 行为不变。
   - 任务队列初版单并发，避免多个浏览器自动化流程互相污染状态。
   - 任务状态落盘到 `data/runtime/tasks/`。

5. **实现前端 MVP**
   - 新增 `frontend/`
   - 技术栈：
     - Vite
     - React
     - TypeScript
     - `lucide-react`
   - 页面顺序：
     - App shell
     - Dashboard
     - Tasks
     - Jobs
     - Job Detail
     - Candidate Detail
     - Run Job
     - RAG
     - Ops
   - 先用 mock API 开发 UI，再切真实 API。
   - Stitch `react-components` 输出只能作为草案，必须审查、重构并适配本项目组件结构。

6. **接入真实任务提交**
   - `Run Job` 表单映射现有 CLI 参数。
   - `platform all` 顺序必须保持：
     - `51job`
     - `liepin`
     - `zhilian`
   - 任一平台失败立即停止并返回失败任务状态。
   - batch 模式遵守现有规则：
     - `--jobs-file` 是唯一 job 定义来源
     - 不允许和 single-job `keyword`、`jd`、`jd-file` 混用。

7. **接入 RAG**
   - 复用现有 `rag:api` 语义。
   - `POST /api/rag/answer` 不得在 no-answer 时编造事实。
   - temporary JD 问答不得创建 job、不得写 persistent RAG indexes、不得 append `answer-logs.jsonl`。

**验证步骤**

1. Stitch 验证：
   - 能通过代理访问 Stitch 文档。
   - 能通过 `STITCH_API_KEY` 连接 MCP 或 SDK。
   - 能执行 `list_projects`。
   - 能生成 mock screen、variant，并下载 HTML/screenshot。
   - 确认真实 key 没有进入仓库。

2. 后端验证：
   - `rtk npm run typecheck`
   - `rtk npm run test`
   - 增加 API/queue 测试，覆盖参数互斥、任务状态、job 读取、`platform all` 顺序。

3. 前端验证：
   - `rtk npm run web:build`
   - Playwright 检查桌面和移动端关键页面。
   - 检查中文长文本、失败原因、候选人字段、RAG sources 不溢出。

4. 集成验证：
   - 启动 API 服务。
   - 启动前端 dev server。
   - 用 mock 或低风险 job 验证任务提交、状态刷新、结果读取。
   - 验证 RAG no-answer 不会调用模型编造答案。
   - 验证前端不能直接写 `data/` 业务文件。
