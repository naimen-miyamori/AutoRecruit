# 项目说明文档

## 项目定位

AutoRecruit 是一个 TypeScript CLI，用于把招聘网站的人才搜索、简历抓取、结构化解析、JD 匹配评分、Markdown 报告导出和邮件发送串成一条可重复运行的本地流程。

当前正式支持的生产浏览器路径包括：

| 平台 | CLI 名称 | 登录态文件 | 主要入口 | 详细说明 |
| --- | --- | --- | --- | --- |
| 前程无忧 | `51job` | `storage-state.json` | 订阅搜索页 | 本文保留主流程说明 |
| 猎聘 | `liepin` | `storage-state.liepin.json` | 快捷搜索标签 | [docs/platforms/liepin.md](docs/platforms/liepin.md) |
| 智联招聘 | `zhilian` | `storage-state.zhilian.json` | 快捷搜索标签 | [docs/platforms/zhilian.md](docs/platforms/zhilian.md) |

项目不是通用 ATS，也不是 Web 平台。它是一个 CLI-first、本地优先的招聘筛选工具，重点是稳定执行、可离线复盘、可按平台隔离数据。

## 主流程

一次完整运行大致如下：

`CLI 参数 / jobs 文件` -> `JD 标准化` -> `JobRecord` -> `平台搜索` -> `候选人列表` -> `新增候选人` -> `简历快照和结构化简历` -> `seen-ids.json` -> `评分产物` -> `RunResult` -> `Markdown 报告` -> `邮件发送`

具体语义：

1. 用户指定 `--platform` 和搜索关键词。首次运行新 `jobKey` 时还需要 `--jd` 或 `--jd-file`。
2. 系统优先根据搜索关键词生成 `jobKey`。如果该平台下已有 `jd.json`，直接复用历史 JD 原文和标准化结果，不再读取或解析本次 JD 输入。
3. Playwright 读取当前平台的 storage state，并验证登录态仍可访问平台已鉴权入口。
4. 进入平台搜索入口，按默认规则排除平台“已查看/已看过”候选人；显式传入 `--include-viewed true` 时取消对应平台的已查看筛选，再提取候选人 ID。明确空结果文案或稳定空列表视为成功的 0 候选人运行。
5. 使用 `seen-ids.json` 过滤历史已处理候选人，只打开新增候选人的简历详情。
6. 成功解析的简历会保存原始文本快照、DOM 快照和结构化 JSON。
7. 成功保存简历的候选人先写入 `seen-ids.json`，再进入评分阶段；详情打开或提取失败的候选人下次仍可重试。
8. 评分从本地简历 JSON 执行。评分失败会写入失败评分产物，不回滚 seen 状态。
9. 生成轻量运行结果，导出最新 Markdown 报告；如配置收件人，则发送邮件。

## 运行模式

### 单职位模式

首次运行一个新职位：

```bash
npm run dev -- --platform <51job|liepin|zhilian> --keyword "<关键词>" --jd "<JD 文本>"
```

或从文件读取 JD：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>" --jd-file ./fixtures/jd.txt
```

重跑已有职位时可以不再提供 JD：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>"
```

默认情况下，主流程会尽量排除平台已查看候选人。需要复跑已查看候选人时，显式打开 `--include-viewed`：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>" --include-viewed true
```

平台行为：

- `51job`：取消搜索页的 `我已看`。
- `liepin`：取消搜索页的 `隐藏已查看`；默认不传时会保持勾选。
- `zhilian`：只取消可见的 `未看过`，不会改动 `未聊过`；取消后会确认快捷搜索条件仍显示 `关键词：<关键词>`，必要时重新选择快捷搜索标签再提取候选人。

Liepin 可在新增候选人进入详情页后自动转发给常用联系人。显式传入联系人姓名才会启用；转发成功后才继续解析和标记 seen，转发失败会让该候选人保持可重试：

```bash
npm run dev -- --platform liepin --keyword "<关键词>" --liepin-forward-contact "<常用联系人姓名>"
```

### 全平台模式

`--platform all` 会按固定顺序串行运行：

1. `51job`
2. `liepin`
3. `zhilian`

每个平台使用独立浏览器会话和平台隔离存储。任一平台失败时，整体运行停止并抛出该平台错误。

示例：

```bash
npm run dev -- --platform all --keyword "<关键词>" --jd "<JD 文本>"
```

### 批量模式

`--jobs-file` 接收 JSON 数组，每个条目至少包含 `keyword`，并可选包含 `jd`、`jdFile`、`email`、`cc`。

```json
[
  {
    "keyword": "东南亚 销售",
    "jdFile": "./fixtures/jd.txt",
    "email": "user@example.com",
    "cc": ["manager@example.com"]
  }
]
```

运行：

```bash
npm run dev -- --platform <platform|all> --jobs-file ./jobs.json
```

批量模式不能与单职位参数 `--keyword`、`--jd`、`--jd-file` 混用。与 `--platform all` 组合时，外层顺序是 jobs 文件顺序，内层顺序是 `51job -> liepin -> zhilian`。

批量模式同样支持在 CLI 上传入 `--include-viewed true`，该开关会应用到本次 jobs 文件内的所有职位和平台。

### 搜索订阅模式

`--search-subscription-file` 是独立模式，只负责打开平台搜索条件页、填入关键词、应用支持的搜索条件、读取结果数，并在显式要求时保存搜索条件/订阅。

输入文件示例：

```json
{
  "keyword": "东南亚 销售",
  "savedSearchName": "东南亚 销售订阅",
  "conditions": []
}
```

运行：

```bash
npm run dev -- --platform 51job --search-subscription-file ./search-subscription.json --save-search-subscription true
```

51job 和 Liepin 支持通过 `applicationFilterInputFile` 引用应用层筛选输入文件，例如：

```json
{
  "keyword": "优衣库",
  "savedSearchName": "优衣库筛选输入结果数检查",
  "applicationFilterInputFile": "./51job-filter-input.example.json",
  "conditions": []
}
```

复用浏览器运行示例：

```bash
PLAYWRIGHT_51JOB_REUSE_BROWSER=true npm run dev -- --platform 51job --search-subscription-file fixtures/51job-search-subscription-with-filter-input.example.json
```

Liepin 输入示例：

```json
{
  "keyword": "优衣库",
  "applicationFilterInput": {
    "work_years": "3-5年",
    "education": "本科",
    "school_nature": "211",
    "language": "英语",
    "expected_location": "上海",
    "age": { "min": 25, "max": 35 },
    "engaged_industry": {
      "value": "电子商务",
      "pathLabels": ["AI/互联网/IT", "电子商务"]
    },
    "engaged_function": "店长/卖场管理",
    "expected_industry": {
      "value": "电子商务",
      "pathLabels": ["AI/互联网/IT", "电子商务"]
    },
    "expected_function": "店长/卖场管理",
    "company_name": "迅销",
    "school_name": "辽宁大学",
    "major": "新闻学",
    "expected_salary": { "min": "10万", "max": "30万" },
    "current_salary": { "min": "10万", "max": "30万" },
    "overseas_work_experience": "有海外工作经验",
    "management_experience": "有管理经验"
  },
  "conditions": []
}
```

Liepin 可采集的 application options 多于当前已验证的页面回放字段。搜索订阅回放请先使用已验证字段：`work_years`、`education`、`school_nature`、`recent_activity_time`、`gender`、`language`、`living_location`、`expected_location`、`expected_salary`、`current_salary`、`job_hopping_count`、`job_status`、`resume_language`、`overseas_work_experience`、`management_experience`、`age`、`engaged_industry`、`engaged_function`、`expected_industry`、`expected_function`、`company_name`、`school_name`、`major`。其中 `age` 是数字范围对象，单位为岁；`engaged_industry`、`engaged_function`、`expected_industry` 和 `expected_function` 分别对应页面的“当前行业”“当前职位”“期望行业”“期望职位”。行业父级例如 `AI/互联网/IT`、`消费品`、`生活服务`、`交通/物流/贸易/零售` 是路径节点，不是直接提交值；行业叶子值如果存在跨大类同名，使用 `{ "value": "电子商务", "pathLabels": ["AI/互联网/IT", "电子商务"] }` 指明点击路径。`company_name`、`school_name` 和 `major` 是不限枚举的文本输入。`keyword_title` 由搜索订阅 `keyword` 控制，不重复作为 `applicationFilter`；`recruitment_type` 当前只采集到默认不限值，暂不作为有效回放字段。

Liepin 行业弹窗完整树采集命令：

```bash
npm run discover:liepin-industry-tree -- --keyword "优衣库"
```

可用 `--field engaged_industry,expected_industry` 限定字段，`--output` 额外写一份原始树。默认输出包括 `data/liepin/filter-catalog/industry-tree/latest.json`、合并后的 `data/liepin/filter-catalog/latest.json` 和 `data/liepin/filter-catalog/application-filter-options.latest.json`。

Liepin 全部选项实站验证使用独立脚本，默认只生成计划，不打开浏览器。行业完整树采集完成后，`engaged_industry` 和 `expected_industry` 的验证用例会遍历树上的叶子节点，并生成 `{ "value": "...", "pathLabels": [...] }` 输入；父级行业只作为路径节点，不作为直接验证值：

```bash
npm run verify:liepin-filter-options -- --keyword "优衣库" --limit 10
```

确认计划后再分批实站执行：

```bash
npm run verify:liepin-filter-options -- --keyword "优衣库" --run true --limit 10
```

常用参数：`--field work_years,education` 限定字段；`--offset`/`--limit` 分批；`--include-defaults false` 跳过“不限”；`--include-range-combinations true` 对薪资/年龄跑所有有效上下限组合；`--stop-on-failure true` 遇到首个失败即停。输出默认写入 `data/liepin/filter-catalog/option-verification/`，dry-run 是计划 JSON，实站运行是逐行 JSONL。

该模式不会解析 JD、创建职位记录、抓取候选人、评分、导出报告或发送邮件，也不能与 `--include-viewed` 组合。输出中的 `conditionResults` 会逐项记录 `applied`、`skipped` 或 `failed`，并通过 `allConditionsApplied` 汇总本次条件是否全部落页。

## 运行要求

### 必需条件

- Node.js 24 LTS 或 Node.js 26。仓库 `.nvmrc` 固定为 `24`，`package.json` 声明 `>=24 <27`。
- 已安装依赖：`npm install`
- 可用模型 API 凭据：`OPENAI_API_KEY`
- 可复用的平台登录态：通过 `npm run login:session -- --platform <platform>` 生成或刷新。

### 可选配置

- `DATA_DIR`：调整本地数据目录，默认是 `data/`。
- `OPENAI_BASE_URL`：使用 OpenAI 兼容模型服务。
- `OPENAI_MODEL`：JD 解析和评分的默认模型。
- `JD_PARSING_MODEL`：覆盖 JD 解析模型。
- `SCORING_MODEL`：覆盖评分模型。
- `.venv/bin/python` + `crawl4ai`：启用 Crawl4AI 增强提取；不可用时会自动回退到内置解析。
- SMTP 环境变量：`SMTP_HOST`、`SMTP_PORT`、`SMTP_USER`、`SMTP_PASS`、`SMTP_FROM`。

### 浏览器配置

- `BROWSER_ENGINE=cloakbrowser|playwright`：浏览器启动引擎，默认 `cloakbrowser`。如需回退到 Playwright 自带 Chromium，设置为 `playwright`。
- `PLAYWRIGHT_HEADLESS=true`：启用 headless。默认是 headed；Liepin 始终强制使用有头浏览器，此变量对 Liepin 不生效。
- `PLAYWRIGHT_SEARCH_PAGE_TIMEOUT_MS`：搜索入口到候选列表 ready 的总等待预算，默认 `20000`。
- `PLAYWRIGHT_EMPTY_RESULTS_STABLE_MS`：51job 空列表稳定窗口，默认 `2000`。
- `PLAYWRIGHT_API_FALLBACK_TIMEOUT_MS`：Liepin/Zhilian API fallback 短等待上限，默认 `3000`。
- `PLAYWRIGHT_RESUME_DETAIL_TIMEOUT_MS`：简历详情打开/就绪总等待预算，默认 `20000`。
- `PLAYWRIGHT_ACTION_DELAY_MIN_MS` / `PLAYWRIGHT_ACTION_DELAY_MAX_MS`：全平台点击/输入动作前的随机等待默认值。51job/Zhilian 默认 `0-0`，Liepin 默认 `2000-3000`。
- `PLAYWRIGHT_CANDIDATE_DELAY_MIN_MS` / `PLAYWRIGHT_CANDIDATE_DELAY_MAX_MS`：全平台候选人之间的随机等待默认值。51job/Zhilian 默认 `0-0`，Liepin 默认 `2000-3000`。
- `PLAYWRIGHT_51JOB_ACTION_DELAY_MIN_MS` / `PLAYWRIGHT_51JOB_ACTION_DELAY_MAX_MS`、`PLAYWRIGHT_LIEPIN_ACTION_DELAY_MIN_MS` / `PLAYWRIGHT_LIEPIN_ACTION_DELAY_MAX_MS`、`PLAYWRIGHT_ZHILIAN_ACTION_DELAY_MIN_MS` / `PLAYWRIGHT_ZHILIAN_ACTION_DELAY_MAX_MS`：覆盖单个平台动作等待。
- `PLAYWRIGHT_51JOB_CANDIDATE_DELAY_MIN_MS` / `PLAYWRIGHT_51JOB_CANDIDATE_DELAY_MAX_MS`、`PLAYWRIGHT_LIEPIN_CANDIDATE_DELAY_MIN_MS` / `PLAYWRIGHT_LIEPIN_CANDIDATE_DELAY_MAX_MS`、`PLAYWRIGHT_ZHILIAN_CANDIDATE_DELAY_MIN_MS` / `PLAYWRIGHT_ZHILIAN_CANDIDATE_DELAY_MAX_MS`：覆盖单个平台候选人间隔。
- `PLAYWRIGHT_REUSE_BROWSER=true|false`：全平台可复用浏览器默认开关。默认不启用；Liepin 有平台默认值，会默认启用。
- `PLAYWRIGHT_51JOB_REUSE_BROWSER=true` / `PLAYWRIGHT_ZHILIAN_REUSE_BROWSER=true`：为 51job 或 Zhilian opt-in 可复用浏览器。
- `PLAYWRIGHT_LIEPIN_REUSE_BROWSER=false`：关闭 Liepin 可复用浏览器；默认开启。开启时流程结束会关闭最后的详情页、停留在人才搜索页并保留浏览器，下次 Liepin 运行会优先连接这个窗口。
- `PLAYWRIGHT_51JOB_REUSE_CDP_PORT` / `PLAYWRIGHT_LIEPIN_REUSE_CDP_PORT` / `PLAYWRIGHT_ZHILIAN_REUSE_CDP_PORT`：各平台可复用浏览器的本地 CDP 端口，默认分别是 `19325`、`19327`、`19329`。

CloakBrowser 自带诊断和维护 CLI：

```bash
npx cloakbrowser info        # 查看 CloakBrowser 版本、平台、二进制路径和安装状态
npx cloakbrowser install     # 预下载 CloakBrowser Chromium
npx cloakbrowser update      # 检查并下载新版 Chromium
npx cloakbrowser clear-cache # 清除 CloakBrowser 浏览器缓存
```

正常多平台运行时应保持 `STORAGE_STATE_PATH` 未设置，让系统按平台选择默认 storage state。显式设置时，路径必须与当前平台匹配，避免把一个平台的登录态套用到另一个平台。

## 架构说明

| 层次 | 职责 | 主要位置 |
| --- | --- | --- |
| 入口编排层 | 参数解析、模式分发、平台循环、结果收口 | `src/index.ts` |
| 平台适配层 | 平台登录入口、搜索入口、候选人提取、详情解析 | `src/platforms/*.ts` |
| 浏览器交互层 | Playwright 会话、51job 搜索页、列表和详情 helper | `src/browser/*.ts` |
| JD 解析层 | 模型化 JD 结构化、schema 校验和清洗 | `src/parsers/jd-parser.ts` |
| 简历提取层 | 页面文本/DOM 快照解析和可选 Crawl4AI 适配 | `src/browser/resume-detail.ts`、`src/extraction/` |
| 评分层 | 构造评分提示词、调用模型、校验评分 JSON | `src/scoring/` |

核心源码位置：

- `src/index.ts`：CLI 主入口和总编排。
- `src/config.ts`：环境变量读取与运行时配置。
- `src/types/job.ts`：JD、候选人、简历、评分、运行结果等共享类型。
- `src/platforms/registry.ts`：平台注册、默认平台解析与 `--platform` 校验。
- `src/search/search-subscription.ts`：独立搜索订阅创建编排。
- `src/scripts/*.ts`：登录、迁移、重解析、评分、导出、诊断和 smoke 工具。

## 数据模型与落盘结构

核心类型定义在 `src/types/job.ts`：

- `NormalizedJob`：结构化 JD。
- `JobRecord`：职位持久化记录，包含 `jobKey`、`platform`、`searchKeyword`、JD 原文、标准化 JD 和邮件收件人。
- `CandidateListItem`：搜索结果页候选人卡片。
- `CandidateResume`：结构化简历，包含基础信息、工作经历、项目经历、教育经历、技能、证书和可选 `candidateShareUrl`。
- `CandidateScoreArtifact`：评分产物，可能是成功评分，也可能是失败记录。
- `RunResult`：一次执行的轻量摘要。
- `JobResultsMarkdownExport`：导出的 Markdown 结果视图。

默认数据目录是 `data/`，每个职位位于：

```text
data/<platform>/jobs/<jobKey>/
```

目录内容：

- `jd.json`：职位记录，也是 rerun 复用 JD 载荷和邮件收件人的来源。
- `seen-ids.json`：已成功处理的候选人 ID。
- `resumes/<candidateId>.json`：结构化简历。
- `snapshots/<candidateId>.txt`：原始页面文本快照。
- `snapshots-dom/<candidateId>.json`：DOM 抽取快照。
- `scores/<candidateId>.json`：评分产物。
- `results/<timestamp>.json`：轻量运行摘要。
- `exports/latest.md`：最新导出的 Markdown 报告。

历史平台化改造前的数据可通过 `npm run migrate:platform-storage` 从 `data/jobs/` 迁移到 `data/51job/jobs/`，并回填缺失的 `platform` 字段。迁移脚本会拒绝覆盖已存在的目标目录。

## 邮件与失败语义

收件人和抄送列表按职位维度持久化：

- 首次运行传入 `--email` 或 `--cc` 时，会写入对应 `jd.json`。
- 后续同一 `jobKey` 重跑时，如果未显式传入，会复用已保存配置。
- 再次显式传入时，以本次参数为准并覆盖保存值。

失败处理：

- 评分失败：保存失败评分产物，候选人仍保持已处理。
- 上游详情打开或提取失败：不写入 seen，下次继续重试。
- 导出失败：不让整次运行失败，错误记录到 summary。
- 邮件失败：不让整次运行失败，错误记录到 summary。
- 空结果运行：属于成功运行，可发送“无新增候选人”通知邮件。
- Zhilian 有新增候选人的报告邮件要求当前运行评分产物都带唯一 `candidateShareUrl`；缺失或重复会导致邮件发送失败并记录到 summary。
- Zhilian 搜索页卡片可能不暴露可用详情链接，候选人 ID 提取会优先读取卡片 Vue props 中的 `userMasterId`、`resumeNumber` 等字段，再使用 API fallback。

## 维护建议

- 修改 `src/browser/resume-detail.ts` 这类启发式模块时，优先使用已保存快照离线重解析验证，再考虑 live 流程。
- 修改平台搜索入口、列表提取或详情解析时，保留多重 DOM fallback，避免假设页面结构稳定。
- 修改全平台或批量语义时，同时验证单平台、`--platform all` 和 `--jobs-file` 的顺序与错误传播。
- 修改报告或邮件逻辑时，覆盖有新增候选人、无新增候选人、评分失败和 Zhilian 分享链接校验路径。
