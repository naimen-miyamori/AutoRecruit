## 项目定位

AutoRecruit 是一个 CLI-first、本地优先的招聘自动化工具。它面向招聘侧的重复执行场景，把平台搜索、候选人抓取、简历解析、JD 匹配评分、Markdown 报告和邮件发送串成可复跑、可审计、按平台隔离的数据流程。

项目当前包含六类能力：

- 普通简历抓取主流程：支持单职位、`--platform all`、`--jobs-file` 批量、保存搜索入口和直接筛选输入入口、已查看筛选切换、Liepin 常用联系人转发、Zhilian 分享链接邮件交付。
- 搜索条件维护流程：支持筛选发现、应用层筛选输入导出/校验、搜索订阅结果数检查和显式保存搜索条件；该流程独立于 JD、候选人抓取、评分、报告和邮件。
- 内部运营控制台：提供本地 HTTP API 和 React 前端，用于查看任务、职位、候选人、筛选 catalog、RAG 问答、运营诊断和循环自动运行；任务执行仍复用 CLI 语义、共享队列和本地数据目录。
- 智能助手：在控制台中把自然语言整理成结构化任务草稿，支持普通抓取、批量、搜索订阅、登录刷新、RAG 运营和候选人问答；确认后才进入现有后端执行路径。
- 离线导出工具：支持把已保存的结构化简历按本地 Word 模板导出为 DOCX，并在能确认详情页本人头像时放入模板头像位置。
- Boss 单平台扩展：支持复用当前搜索页抓取/筛选、站内同事或邮件转发，以及在当前沟通页审查未读候选人；符合者先转发简历，再从常用语发送求简历消息并申请换电话；不符合者默认只记录判断，只有显式设置 `--boss-chat-reply-unqualified true` 才从常用语发送拒绝消息；最后发送审查总结邮件。

当前浏览器平台分为三平台主流程和 Boss 单平台扩展。`--platform all` 始终只包含 51job、Liepin、Zhilian；Boss 必须显式使用 `--platform boss`：

| 平台 | CLI 名称 | 登录态文件 | 主要入口 | 核心规则 |
| --- | --- | --- | --- | --- |
| 前程无忧 | `51job` | `storage-state.json` | 订阅搜索页或直接人才搜索页 | 默认有头和可复用浏览器；保存入口通过订阅卡片进入并确认 `关键词：<keyword>`；直接入口清空筛选后输入关键词和应用筛选；默认勾选 `我已看` |
| 猎聘 | `liepin` | `storage-state.liepin.json` | 找人/快捷搜索或直接关键词搜索 | 默认有头和可复用浏览器，且始终强制有头；保存入口进入 `找人` 快捷搜索；直接入口清空筛选后输入关键词和应用筛选；默认勾选 `隐藏已查看`；可选常用联系人转发 |
| 智联招聘 | `zhilian` | `storage-state.zhilian.json` | `/app/search` 快捷搜索标签或直接关键词搜索 | 默认有头和可复用浏览器；保存入口点击快捷搜索标签并确认 `关键词：<keyword>`；直接入口不点快捷标签，清空筛选后输入关键词和应用筛选；默认勾选 `未看过`；自动复制分享链接 |
| Boss 直聘 | `boss` | `storage-state.boss.json` | 当前人才搜索页或 `/web/chat/index` 沟通页 | 默认有头和可复用浏览器；只允许单平台运行，不进入 `all`；复用当前页面避免重复登录和新开标签；支持配置式简历转发和独立自动沟通审查 |

项目不是通用 ATS，也不是完整 SaaS 或平台后台替代品。当前 Web 能力是本地内部运营控制台，重点仍是把已定义的本地运行模式稳定落页、保留可离线复盘的数据产物，并维持四个平台的适配边界以及 Boss 不进入 `all` 的产品约束。

## 主流程

一次完整运行大致如下：

`CLI 参数 / jobs 文件` -> `JD 标准化` -> `JobRecord` -> `平台搜索` -> `候选人列表` -> `新增候选人` -> `简历快照和结构化简历` -> `seen-ids.json` -> `评分产物` -> `RunResult` -> `Markdown 报告` -> `邮件发送`

该链路描述普通抓取。Boss 自动沟通是独立流程，不进入普通抓取、批量或 `all`：`当前沟通页` -> `未读会话快照` -> `打开会话并判断此前是否聊过` -> `已聊过：只汇总本次新回复 / 首次沟通：读取岗位 JD 和转发配置` -> `在线简历` -> `严格条件/评分判断` -> `符合者先转发、发求简历常用语并换电话 / 不符合者记录判断，并按显式开关决定是否发送拒绝话术` -> `审查结果` -> `总结邮件`。

具体语义：

1. 用户指定 `--platform` 和搜索关键词。首次运行新 `jobKey` 时还需要 `--jd` 或 `--jd-file`。
2. 系统优先根据搜索关键词生成 `jobKey`。如果该平台下已有 `jd.json`，直接复用历史 JD 原文和标准化结果，不再读取或解析本次 JD 输入。该文件同时保存岗位搜索来源、direct 筛选输入及标准化条件、报告邮箱和 Boss 转发目标；同一配置只保留一份，不追加重复历史记录，也不重写内容完全相同的文件。
3. Playwright 读取当前平台的 storage state，并验证登录态仍可访问平台已鉴权入口。
4. 进入平台搜索入口。新岗位默认 `--search-source saved` 复用平台已保存订阅/快捷搜索；`--search-source direct` 会清空旧条件、输入本次关键词、应用 `--application-filter-input-file` 的筛选条件，再进入正常简历抓取。岗位首次保存 direct 条件后，后续只传平台和关键词即可复用，不再依赖原筛选 JSON 或 catalog 文件；显式传入新的 `--search-source` 时才替换历史搜索配置。
5. 对齐平台默认已查看筛选：51job 默认勾选 `我已看`，Liepin 默认勾选 `隐藏已查看`，Zhilian 默认勾选 `未看过`；显式传入 `--include-viewed true` 时取消对应平台的已查看筛选，再提取候选人 ID。明确空结果文案或稳定空列表视为成功的 0 候选人运行。
6. 使用 `seen-ids.json` 过滤历史已处理候选人，只打开新增候选人的简历详情。
7. 成功解析的简历会保存原始文本快照、DOM 快照和结构化 JSON。
8. 成功保存简历的候选人先写入 `seen-ids.json`，再进入评分阶段；详情打开或提取失败的候选人下次仍可重试。
9. 评分从本地简历 JSON 执行。评分失败会写入失败评分产物，不回滚 seen 状态。
10. 生成轻量运行结果，导出最新 Markdown 报告；如配置收件人，则发送邮件。

## 平台行为

### 浏览器操作节奏

网页动作节奏集中维护如下。除非表中另有说明，等待发生在导航、点击、输入、按键和候选人切换等浏览器操作之间；页面状态读取、简历解析、模型评分、文件写入和 SMTP 邮件不额外增加用户操作等待。

| 平台 | 动作间隔 | 候选人间隔 | 详情关闭规则 |
| --- | --- | --- | --- |
| 51job | 默认不额外等待 | 默认不额外等待 | 受详情页就绪和关闭流程控制 |
| Liepin | 随机 `2000-3000ms` | 随机 `2000-3000ms` | 成功提取并保存后再等待 `2000-3000ms` 关闭；转发或提取失败时保留详情页 |
| Zhilian | 默认不额外等待 | 默认不额外等待 | 关闭页内 modal 后继续 |
| Boss | 加权 `2000-4000ms`；约 80% 在 `2000-3000ms`、20% 在 `3001-4000ms` | 使用相同加权分布 | 打开详情、转发、关闭详情和进入下一候选人之间都经过共享节奏控制 |

平台级 `PLAYWRIGHT_<PLATFORM>_ACTION_DELAY_*` 和 `PLAYWRIGHT_<PLATFORM>_CANDIDATE_DELAY_*` 可以覆盖默认值。具体实现约束见 [`src/browser/AGENTS.md`](./src/browser/AGENTS.md)，平台动作边界见 [`src/platforms/AGENTS.md`](./src/platforms/AGENTS.md)。

### 51job

51job 通过订阅搜索页进入人才搜索。流程会点击匹配的订阅关键词卡片，确认当前活跃订阅详情面板标题等于本次关键词，再点击该面板内的 `去搜索` 人才搜索入口；进入人才搜索页后还会确认页面显示 `关键词：<keyword>`，避免复用浏览器状态或其它订阅条件串入本次运行。流程保留多组 selector fallback，并把 `没有搜索到相关的人才` 这类明确空结果视为成功的 0 候选人运行。

候选卡片中的 `no_interested_<id>` 只作为候选人锚点，不作为详情入口。详情打开会优先点击候选人姓名、详情行等区域，并在单个点击入口无效时继续尝试其它 fallback，避免一个无效区域耗尽整次详情打开等待预算。

默认会显式勾选搜索页的 `我已看`；传入 `--include-viewed true` 时才会取消 `我已看`。51job 默认启用可复用浏览器；打开人才搜索页后会关闭同一上下文里多余的“我的订阅”页面，避免复跑时订阅页标签堆积。

### Liepin

Liepin 始终使用有头浏览器，`PLAYWRIGHT_HEADLESS=true` 对 Liepin 不生效。浏览器启动引擎默认是 CloakBrowser；如需回退到 Playwright 自带 Chromium，可设置 `BROWSER_ENGINE=playwright`。Liepin 默认启用可复用浏览器：流程结束会关闭最后的详情页、停留并置前招聘端人才搜索页，下次运行优先通过本地 CDP 端口连接这个窗口继续。

搜索流程必须进入招聘端人才搜索面，必要时从首页点击顶部导航 `找人`，再点击本次关键词对应的快捷搜索标签。默认确认 `隐藏已查看` 已勾选；传入 `--include-viewed true` 时才取消该筛选。勾选或取消已查看筛选后，旧的 `search-resumes` 响应不能复用，候选提取以 DOM 为先，只有 DOM 无候选或缺安全详情 URL 时才短等 API fallback。

Liepin 主流程支持 `--liepin-forward-contact "<常用联系人姓名>"`。该开关只处理本地 `seen-ids.json` 过滤后的新增候选人：打开详情页后先点击 `转发`，在弹窗里选择联系人并确认，成功后才解析简历和标记 seen；转发失败时该候选人保持可重试。

Liepin 站内操作、候选人切换和成功详情关闭遵循本节统一的[浏览器操作节奏](#浏览器操作节奏)，点击会尽量先移动鼠标到目标元素。转发或提取失败时保留详情页供检查。

### Zhilian

Zhilian 登录页是 `https://passport.zhaopin.com/org/login`，已鉴权搜索页是 `https://rd6.zhaopin.com/app/search`；`https://rd6.zhaopin.com/desktop` 不是搜索流程目标页。

Zhilian 默认启用可复用浏览器。搜索必须点击已保存的快捷搜索标签，标签文本需要包含本次运行传入的原始 `--keyword`，点击后还必须确认条件中出现 `关键词：<keyword>`。默认会显式勾选 `未看过`；传入 `--include-viewed true` 时只取消可见的 `未看过`，不改动 `未聊过`。如果取消或勾选后快捷搜索条件消失，流程会重新选择该快捷搜索标签，再次设置 `未看过` 并重新确认关键词条件。

候选列表提取 DOM-first。当前搜索卡片可能没有可用详情链接或显式 ID，因此 DOM 提取会优先读取卡片 Vue props 中的 `userMasterId`、`resumeNumber`、`resumeK`、`resumeT` 等字段；DOM 无候选时才等待 API fallback 或明确空结果。

Zhilian 简历详情是 `/app/search` 页内 modal，不是新标签页。解析时只读取 modal 子树，并通过 `转给同事` -> `链接转发` 复制分享链接，保存为 `candidateShareUrl`。Zhilian 有新增候选人的报告邮件必须使用当前运行复制出的唯一分享链接；缺失或重复会作为邮件发送错误记录。

### Boss

Boss 登录入口是 `https://www.zhipin.com/web/user/?ka=header-login`，默认使用 `storage-state.boss.json`、独立浏览器 profile 和本地 CDP 端口 `19331`。运行时优先复用已经打开的 Boss 搜索页或沟通页；不要反复访问登录页，也不要为每次任务创建新 Boss 标签。登录态缺失或失效时，才显式运行登录保存命令并由用户完成登录。

#### 普通抓取

Boss 普通抓取支持当前搜索页准备、direct 条件回放、候选人列表/简历解析和配置式转发。转发参数出现时必须成对提供：`--boss-forward-mode colleague|email` 与 `--boss-forward-recipient`。显式配置会保存到岗位记录，后续同岗位普通抓取可省略并复用。站内同事模式填写姓名并选择唯一匹配项；邮件模式填写收件地址；两种模式的留言都固定使用候选人 ID。转发发生在简历解析和 seen 标记之前，失败候选人保持可重试。

Boss 普通抓取和自动沟通共同遵循本节统一的[浏览器操作节奏](#浏览器操作节奏)，不得引入无间隔的导航、点击、输入、按键、转发或候选人切换动作。

#### 自动沟通

Boss 自动沟通是独立模式。流程只点击左侧“沟通”进入 `https://www.zhipin.com/web/chat/index`，选择“未读”，先快照所有带红点的会话，再按会话 ID 逐个打开。

- 显式转发和总结邮件配置会去重保存到 `data/boss/chat-review/automation-settings.json`。
- 首次沟通候选人的转发目标按“岗位记录优先、Boss 平台默认其次”复用。
- 首次沟通缺少岗位 JD 或转发配置时，在打开并完成历史沟通判断后记录可重试失败，不加入已审查 ID。
- 已聊过候选人不需要岗位 JD 或转发配置。
- 可选 `--boss-sync-jobs-before-review true` 会在读取未读会话前同步职位/JD；任一同步项失败时本轮审查立即停止。
- 会话有 Boss 职位 ID 时必须优先按 ID 找岗位记录；没有 ID 时只允许回退到唯一的标准化同名岗位，同名多岗位必须报歧义错误。

在线简历入口必须使用 Playwright 原生 locator 点击。会话切换后还要等待 Vue 当前会话 ID 和候选人基础信息（包括 `ageDesc`）完成异步水合，再读取摘要；简历详情可合并 iframe `IFRAME_DONE` 的 WASM 解密数据，但原始解密载荷不会落盘。

#### 物业电工严格规则

`--boss-chat-require-all true` 要求简历对六项条件提供明确证据：

1. 年龄严格小于 47 岁。
2. 明确持有高压电工证。
3. 明确持有低压电工证。
4. 明确具有物业行业电工经验。
5. 至少一家公司连续工作 24 个月。
6. 明确是上海人；期望城市或现居地为上海不能替代籍贯证据。

任一项缺少明确证据都判为不符合，不做推测。只有其他五项全部满足、简历没有明确上海或非上海籍贯且教育经历显示在上海学校就读时，才允许关闭简历后询问 `是上海人吗？`。此时状态记为等待确认，不转发、不拒绝、不换电话，也不加入已审查 ID；上海就读经历本身不算上海人证据。

#### 消息与跟进回复

符合候选人的动作顺序固定为：保持简历浮窗打开并先完成转发；立即记录 `forwarded: true`；按统一节奏关闭简历浮窗；从常用语选择 `方便发一份你的简历过来吗？` 并发送；点击“换电话”并确认。

不符合者不转发、不换电话，关闭简历浮窗后记录判断。默认不发送消息；只有显式设置 `--boss-chat-reply-unqualified true`，才从页面常用语选择完整话术 `对不起，看了你的简历以后觉得不太合适，希望你早日找到满意的工作机会` 并发送。符合和不符合候选人的预设消息都禁止直接填写编辑器，具体 selector 由平台适配层维护。

上海籍待确认候选人是受控例外：直接在聊天框输入 `是上海人吗？` 后点击主发送按钮；若编辑器已有其他草稿则禁止覆盖。消息和换电话有幂等保护；转发成功后联系失败会保留转发成功状态，不自动重复转发。

点击红点候选人后，流程同时等待当前会话、候选人基础信息和消息列表加载，再记录 `previousChat`。证据优先级依次为 Boss `bothTalked` 状态、当前可见的招聘方历史消息，以及“可见消息数大于点击前保存的未读数”的兜底判断。

若此前已聊过，只按时间顺序提取最后 `unreadCount` 条候选人消息作为 `newCandidateReplies`，状态记为 `follow_up_reply`，随后立即结束该会话；不会打开简历、读取或判断 JD、评分、转发、发送话术或换电话。文本会规范空白，非文本消息使用明确类型占位符；发送方或消息数量无法可靠确认时记录可重试失败，不生成空白回复。

尚未成功转发的失败会话会从历史审查运行中恢复。真实的新红点事件不会因为会话 ID 已存在于 `reviewedConversationIds` 而跳过；该列表只阻止没有新红点的失败恢复项被重复执行。已聊过候选人的新回复在总结邮件中单独展示，汇总同时统计首次沟通数、此前已聊过数、跟进回复会话数和新回复消息数。运行结束保持当前 Boss 沟通页打开。

#### 推荐、深度搜索与单人打招呼

推荐牛人入口是 `/web/chat/recommend`（候选人列表可能位于 `recommendFrame`），原生深度搜索入口是 `/web/chat/aiform`。两种查询默认只读，并统一输出稳定候选人 ID、卡片摘要、联系状态和来源；不得把可见序号当成执行标识。

深度搜索读取并可精确同步“核心要求”和“加分项”。至少需要一项核心要求；`triggerMatch` 默认 `false`，只有显式确认后才可点击“立即匹配”，且必须先确认剩余次数大于零、按钮可用。匹配结果最多返回最新 20 位候选人。单人打招呼执行前必须按候选人 ID 重新定位卡片，并复核预期姓名、职位名和可用的 Boss 职位 ID；卡片已显示“继续沟通”时视为已联系，不重复点击。

#### 原子会话操作

Boss 会话支持只读操作 `list-conversations`、`open-conversation`、`read-conversation`、`read-history`、`preview-resume`，以及受控变更操作 `send-text`、`remark`、`mark-not-fit`、`request-attachment-resume`、`accept-attachment-resume`、`exchange-phone`、`exchange-wechat`。所有 HTTP 和控制台请求必须进入共享 `TaskQueue`，再复用 Boss 当前浏览器 profile/CDP/标签页；路由层不得直接操作浏览器。

变更操作必须提供精确会话 ID、`confirmed=true` 和唯一 `intentId`。打开后再次校验候选人和职位；通用文本发送发现编辑器已有草稿时必须失败，禁止覆盖。成功回执落在 `data/boss/chat-operations/runs/`，同一 `intentId` 重试返回原回执。原子变更操作不允许加入调度计划。

#### 职位/JD 同步

职位列表入口是 `/web/chat/job/list`。同步包含开放、审核中和已关闭职位，因为未读会话可能引用已关闭岗位；也可显式排除已关闭职位或只同步指定 Boss 职位 ID。岗位目录按“职位名 + Boss 职位 ID”生成稳定 `jobKey`，同名不同 ID 永不合并。

同步结果写入 `data/boss/job-sync/positions.latest.json` 和 `data/boss/job-sync/runs/`，有效 JD 仍写入标准 `data/boss/jobs/<jobKey>/jd.json`。JD 原文先计算 SHA-256；哈希未变时不再次解析或重写岗位记录。新 JD 必须在模型解析成功后才替换旧记录，解析失败不会覆盖已有有效 JD。原有创建时间、转发、报告和搜索配置会保留。

## 筛选发现与搜索订阅

本节只描述搜索筛选项发现、应用层筛选输入和搜索订阅结果数检查。这些能力不解析 JD、不创建职位记录、不抓候选人、不评分、不导出报告、不发送邮件，也不改 seen 状态。

### 筛选发现

筛选发现入口：

```bash
npm run discover:filters -- --platform <51job|liepin|zhilian|boss|all> --keyword "<关键词>"
```

`--platform all` 按 `51job -> liepin -> zhilian` 串行执行，任一平台失败立即停止；Boss 发现必须单独使用 `--platform boss`。发现结果写入：

- `data/<platform>/filter-catalog/latest.json`
- `data/<platform>/filter-catalog/<timestamp>.json`

Liepin 和 Zhilian 默认采用较保守的静态读取；需要打开下拉、弹层和级联树时，显式传入 `--slow-click true`。Zhilian 发现必须停留在 `/app/search`，复用保存的快捷搜索标签并确认搜索条件面板；不要用泛文本点击 `筛选` 或 `使用高级搜索`，这些文字可能命中推荐页或顶部导航。

Liepin 行业完整树单独采集：

```bash
npm run discover:liepin-industry-tree -- --keyword "优衣库"
```

该命令采集 `当前行业`、`期望行业` 的完整大类/叶子树，并合并回 `latest.json` 与 `application-filter-options.latest.json`。行业父级只作为 `pathLabels` 路径节点，叶子才是可提交值。

### 应用层筛选输入

筛选 catalog 可以导出成应用层 options：

```bash
node --import ./scripts/node-ts-hooks.mjs src/scripts/export-application-filter-options.ts <platform>
```

导出文件是 `data/<platform>/filter-catalog/application-filter-options.latest.json`。应用层用 `groups.singleSelect`、`groups.textInput`、`groups.salaryRange`、`groups.numberRange` 分组；`fieldsById.<fieldId>.kind` 决定输入形态。

常见输入形态：

- 普通单选：`"education": "本科"`
- 文本/弹层输入：`"company_name": "迅销"`
- 级联或同名叶子：`{ "value": "电子商务", "pathLabels": ["AI/互联网/IT", "电子商务"] }`
- 数字范围：`"age": { "min": 25, "max": 35 }`
- 薪资范围：`"expected_salary": { "min": "10万", "max": "30万" }`
- 自定义输入：`{ "label": "自定义", "input": { "min": "大专", "max": "本科" } }`

输入校验命令：

```bash
npm run validate:filter-input -- <platform> <input.json>
```

### 平台筛选要点

筛选字段和数量会随平台页面变化，完整可用值以 `data/<platform>/filter-catalog/application-filter-options.latest.json` 为准。本文只保留稳定的回放差异：

| 平台 | 稳定规则 |
| --- | --- |
| 51job | 应用前清理旧筛选和输入 tag；明确空结果按成功的 0 人处理 |
| Liepin | 薪资使用年薪“万”，年龄使用岁；行内文本和 checkbox 必须按真实输入/checked 状态验证；顶层 `keyword` 不重复作为筛选字段 |
| Zhilian | 年限、学历和年龄可使用预设或自定义范围，薪资为月薪；城市、户口、行业和职位等多级字段优先提交 `value` 与 `pathLabels` |
| Boss | direct 和搜索订阅均复用当前搜索页，并逐项确认筛选已落页 |

搜索订阅应用筛选时，输出中的 `conditionResults` 会逐项记录 `applied`、`skipped` 或 `failed`，`allConditionsApplied` 汇总本次条件是否全部落页。只要存在 `skipped` 或 `failed`，显式保存订阅时会拒绝保存半成品条件。

普通简历抓取的 direct 模式复用同一套 application options 和筛选输入格式，但 direct 是正常抓取流程的一部分：应用筛选后会继续候选提取、详情打开、评分、导出和邮件。搜索订阅模式仍然独立，只做条件落页、结果数读取和可选保存。

### 实站验证

Liepin 全部选项实站验证脚本默认只生成计划，不打开浏览器：

```bash
npm run verify:liepin-filter-options -- --keyword "优衣库" --limit 10
```

确认计划后再分批实站执行：

```bash
npm run verify:liepin-filter-options -- --keyword "优衣库" --run true --limit 10
```

常用参数：`--field work_years,education` 限定字段；`--offset`/`--limit` 分批；`--include-defaults false` 跳过“不限”；`--include-range-combinations true` 对薪资/年龄跑所有有效上下限组合；`--stop-on-failure true` 遇到首个失败即停。输出默认写入 `data/liepin/filter-catalog/option-verification/`，dry-run 是计划 JSON，实站运行是逐行 JSONL。

## 内部运营控制台与智能助手

控制台由 `npm run api` 启动本地 HTTP 服务，默认监听 `127.0.0.1:4180`，并从 `frontend/dist` 提供已构建的前端静态文件。开发前端时可单独运行 `npm run web:dev`，Vite 默认监听 `127.0.0.1:5173`，并把 `/api` 代理到 `127.0.0.1:4180`。

```bash
npm run web:build
npm run api

# 开发模式可另开终端：
npm run web:dev
```

可用环境变量：

- `AUTORECRUIT_CONSOLE_HOST`：控制台 API 监听地址，默认 `127.0.0.1`。
- `AUTORECRUIT_CONSOLE_PORT`：控制台 API 端口，默认 `4180`。
- `AUTORECRUIT_CONSOLE_API_KEY`：可选 bearer token；设置后所有 `/api/*` 请求都需要 `Authorization: Bearer <token>`。
- `AUTORECRUIT_CONSOLE_MAX_BODY_BYTES`：请求体上限，默认 2 MiB。
- `AUTORECRUIT_CONSOLE_FRONTEND_DIR`：静态前端目录，默认 `frontend/dist`。
- `VITE_API_BASE_URL`：前端 API 基址，默认 `/api`。

控制台后端使用单进程 `TaskQueue`，普通抓取、批量、搜索订阅、登录刷新、Boss 人才发现/单人打招呼/原子会话操作/职位同步/自动沟通和 RAG 运营任务都会串行入队，任务记录落在 `data/runtime/tasks/`。已保存职位、候选人、评分、运行结果、筛选 catalog 和 RAG 文件仍读取原有 `data/<platform>/...` 结构。

### 循环自动运行

控制台“自动运行”页可以创建持久化计划，把 `resume-capture`、`batch`、`search-subscription`、`boss-job-sync` 或 `boss-auto-chat` 任务组成串行轮次。典型 Boss 轮次按 `boss-job-sync -> boss-auto-chat` 排列；人才匹配、打招呼和原子会话变更不可调度。计划受指定时区和每日时间窗口约束；新一轮从上一轮全部任务完成后计算间隔，`delaySeconds` 为 `0` 时会立即尝试下一轮。计划任务与手工任务共享同一个 `TaskQueue`，不会并发控制浏览器。

调度器支持失败后间隔、整轮遇错停止或继续，以及连续失败达到阈值后暂停。计划和轮次分别持久化到 `data/runtime/schedules/` 与 `data/runtime/schedule-runs/`，服务重启后仍可恢复。停止计划采用“当前任务结束后停止”：正在运行的单个任务完成后，取消本轮余下任务并停止后续轮次；`--platform all`、一次 batch 和一次 Boss 自动沟通各自都视为不可从内部强制中断的单个任务。

控制台提供启动、暂停、立即运行和“当前任务结束后停止”操作；命令行控制入口见 [`README.md`](./README.md#循环自动运行)。调度器必须复用任务 normalizer 和全局队列，详细代码约束见 [`src/server/AGENTS.md`](./src/server/AGENTS.md)。

### API 边界

API 按职责分组，具体路由清单以 [`src/server/routes.ts`](./src/server/routes.ts) 和 [`src/server/http.ts`](./src/server/http.ts) 为准：

| 路由组 | 代表入口 | 职责 |
| --- | --- | --- |
| 健康与仪表盘 | `/api/health`、`/api/dashboard/health` | 服务与依赖状态 |
| 任务 | `/api/tasks/*` | 提交和读取共享队列任务 |
| 自动运行 | `/api/schedules/*` | 计划、轮次和运行控制 |
| 职位与候选人 | `/api/jobs/*` | 读取职位、运行和候选人数据 |
| RAG | `/api/rag/*` | 候选人问答和历史对话导入 |
| 运营 | `/api/ops/*` | 筛选 catalog、筛选输入和运营数据 |
| 智能助手 | `/api/assistant/*` | 草稿生成、校验和确认 |

控制台的 `智能助手` 页面对应三组 API：

- `POST /api/assistant/chat`：根据对话生成或更新结构化 `AssistantDraft`。
- `POST /api/assistant/validate`：重新清洗并校验 draft，返回缺失字段、警告和可确认预览。
- `POST /api/assistant/confirm`：确认 draft。除 `rag-answer` 外，都会复用 shared normalizer 后进入 `TaskQueue`；`rag-answer` 直接调用 RAG 问答并返回答案。

控制台的 `配置` 页面提供 `中转地址`、`模型名称` 和 `API Key` 三项请求级模型覆盖：

- 只随 `POST /api/assistant/chat` 和 `POST /api/rag/answer` 发送，用于本次助手草稿生成或控制台 RAG 问答。
- 不写入任务草稿、`TaskQueue`、任务记录、RAG answer log 或后端配置文件。
- 中转地址和模型名称保存在浏览器本地，API Key 只保存在当前浏览器会话；留空时使用后端环境变量。
- 确认后的 JD 解析、评分、抓取、导出、邮件和 RAG 运维仍使用各自运行环境，不受控制台请求级配置影响。

智能助手只负责把自然语言整理成受控草稿，不是命令执行器。模型输出会按任务类型过滤字段，任意 shell、npm script、文件写入或绕过 normalizer 的请求应被拒绝或作为风险提示；最终确认仍必须通过 `src/server/task-normalizers.ts` 的同一套校验。`argvPreview` 只是给用户核对的预览，不是执行来源。

助手支持的 draft 类型包括 `resume-capture`、`batch`、`search-subscription`、`login-refresh`、`boss-auto-chat`、`boss-talent-search`、`boss-greet`、`boss-chat-operation`、`boss-job-sync`、`rag-ops` 和 `rag-answer`。Boss 草稿和确认仍必须通过共享 normalizer，平台固定为 Boss；立即匹配、打招呼或会话变更必须明确展示风险并同时通过 draft 内的确认字段和最终风险确认。`rag-answer` 保持候选人问答语义：不会创建任务、不会打开浏览器、不会抓候选人、不会评分、不会导出报告或发送邮件；已保存职位和临时 JD 的索引/日志行为与 `--jd-question` / `--rag-question` 保持一致。

## 运行模式

### 单职位模式

首次运行一个新职位：

```bash
npm run dev -- --platform <51job|liepin|zhilian|boss> --keyword "<关键词>" --jd "<JD 文本>"
```

或从文件读取 JD：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>" --jd-file ./fixtures/jd.txt
```

重跑已有职位时可以不再提供 JD：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>"
```

根据已保存或临时 JD 回答候选人问题，不会打开浏览器、抓候选人、评分、导出或发邮件：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>" --jd-question "这个岗位薪资范围是多少？"
npm run dev -- --platform <platform> --keyword "<关键词>" --jd-file ./fixtures/jd.txt --jd-question "这个岗位需要出差吗？"
```

`--rag-question` 是 `--jd-question` 的别名，两者遵循相同隔离规则：

- 已保存职位优先复用 `data/<platform>/jobs/<jobKey>/jd.json` 和持久化 RAG；缺少索引时自动索引当前 JD，不重新解析 JD。
- 找不到已保存 JD 时才读取本次 `--jd` 或 `--jd-file`，使用临时轻量检索回答，不创建职位记录、持久化索引或生产问答日志。
- 检索组合 Qdrant 向量、本地 BM25、融合排序和轻量重排。输出包含 `answer`、`answered`、`confidence`、`noAnswerReason` 和 `sources`。
- 没有可信 JD/已确认招聘方事实，或最高可信片段低于 `RAG_MIN_CONFIDENCE_SCORE` 时，直接返回明确的无答案结果，不调用模型编造。

RAG 运维入口按用途分组；完整参数和日常处理顺序见 [`docs/rag运营手册.md`](./docs/rag运营手册.md)，全部脚本名以 [`package.json`](./package.json) 为准：

```bash
npm run rag:index -- --platform <platform> --keyword "<关键词>"
npm run rag:ask -- --platform <platform> --keyword "<关键词>" --question "这个岗位需要出差吗？"
npm run rag:doctor -- --platform <platform> --keyword "<关键词>" [--question "这个岗位需要出差吗？"]
npm run rag:ingest-conversations -- --file ./conversations.jsonl [--dry-run true] [--fail-on-error false] [--doctor true]
npm run rag:ops -- --file ./rag-review-jobs.json [--question "这个岗位需要出差吗？"] [--policy ./rag-metrics-policy.json] [--output ./rag-ops.md] [--fail-on-issue true]
```

RAG 的稳定行为按能力分组如下：

| 能力 | 入口 | 稳定语义 |
| --- | --- | --- |
| 只读诊断 | `rag:inspect`、`rag:doctor`、`rag:doctor:batch` | 检查本地事实、索引、provider/model、Qdrant 和检索命中，不生成回答或改写索引 |
| 对话导入 | `rag:ingest-conversation`、`rag:ingest-conversations` | 按稳定 turn ID 增量、幂等合并；只把已确认的招聘方信息建成回答事实 |
| 人工审核 | `rag:feedback`、`rag:review`、`rag:review:batch` | 反馈写回职位 `answer-logs.jsonl`；批量审核只汇总指定职位 |
| 质量运营 | `rag:metrics`、`rag:ops` | 汇总健康、待审核和质量阈值；可用失败码接入 CI 或定时巡检 |
| 回归评测 | `rag:eval`、`rag:answer-eval`、`rag:regression` | 分别验证召回、最终回答和完整 suite；失败 case 返回非 0 |
| 内部 HTTP 接口 | `rag:api` | 复用 CLI 的职位隔离、事实信任、自动索引和日志语义；API key 只是轻量保护，不提供多租户或 RBAC |

默认基线使用本地 embedding HTTP 服务和 Qdrant；`npm run test:rag:offline` 则使用临时数据、内存向量库和确定性假 embedding，不调用 OpenAI 或连接 Qdrant。Fixture 必须脱敏，只有已确认招聘方事实可标记 `verified: true`，并应包含资料未说明或未确认的负例。

完整参数、质量 policy、模板文件、基线维护和本机服务排障见 [`docs/rag运营手册.md`](./docs/rag运营手册.md)；RAG 的事实信任、职位隔离和无答案约束见 [`src/rag/AGENTS.md`](./src/rag/AGENTS.md)。

默认 `--search-source saved` 会使用平台已保存的订阅/快捷搜索入口。需要临时主动输入筛选条件、不依赖保存标签时，使用 direct 搜索来源：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>" --jd-file ./fixtures/jd.txt --search-source direct --application-filter-input-file ./filter-input.json
```

direct 模式会打开平台搜索页、清理旧筛选、输入本次关键词、应用 `data/<platform>/filter-catalog/application-filter-options.latest.json` 支持的应用层筛选条件，然后继续同一套候选抽取、详情打开、评分、报告和邮件流程。`--application-filter-input-file` 只在 `--search-source direct` 下有效；筛选条件出现 `skipped` 或 `failed` 时会中止本次抓取，避免从半应用条件里抓候选人。

默认情况下，主流程会按平台默认状态设置已查看筛选。需要切换到包含已查看候选人的复跑时，显式打开 `--include-viewed`：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>" --include-viewed true
```

平台行为：

- `51job`：默认勾选搜索页的 `我已看`；`--include-viewed true` 时取消它。
- `liepin`：默认勾选搜索页的 `隐藏已查看`；`--include-viewed true` 时取消它。
- `zhilian`：默认勾选可见的 `未看过`；`--include-viewed true` 时只取消可见的 `未看过`，不会改动 `未聊过`。设置后会确认快捷搜索条件仍显示 `关键词：<关键词>`，必要时重新选择快捷搜索标签再提取候选人。

Liepin 可在新增候选人进入详情页后自动转发给常用联系人。显式传入联系人姓名才会启用；转发成功后才继续解析和标记 seen，转发失败会让该候选人保持可重试：

```bash
npm run dev -- --platform liepin --keyword "<关键词>" --liepin-forward-contact "<常用联系人姓名>"
```

`--platform all` 的单职位运行也可以传入 `--include-viewed true` 和 `--liepin-forward-contact "<常用联系人姓名>"`。`--include-viewed true` 会应用到三个平台；`--liepin-forward-contact` 只在 Liepin 阶段生效。

direct 单职位模式同样支持这些开关。Zhilian 仍会在新增候选人详情中复制 `转给同事` -> `链接转发` 分享链接；Liepin 仍只对新增候选人执行常用联系人转发。

### Boss 自动沟通模式

Boss 自动沟通只允许 `--platform boss`，且必须显式配置转发方式和收件目标。物业电工严格模式示例：

```bash
PLAYWRIGHT_HEADLESS=false npm run dev -- \
  --platform boss \
  --boss-auto-chat true \
  --boss-chat-require-all true \
  --boss-forward-mode email \
  --boss-forward-recipient "resume@example.com" \
  --boss-chat-summary-email "summary@example.com" \
  --boss-chat-summary-cc "audit@example.com"
```

参数说明：

- `--boss-auto-chat true`：进入独立未读会话审查模式。
- `--boss-chat-require-all true`：使用当前已配置岗位的全部硬性条件；物业电工会走六项明确证据判断。
- `--boss-chat-score-threshold 0..100`：不启用严格模式时使用的模型评分阈值，默认 70。
- `--boss-chat-reply-unqualified true`：可选；启用后才向不符合者发送固定拒绝话术，默认关闭。
- `--boss-forward-mode colleague|email`：符合者的转发方式。
- `--boss-forward-recipient`：站内同事姓名或简历收件邮箱。
- `--boss-chat-summary-email`：整批完成后的总结邮件收件人。
- `--boss-chat-summary-cc`：可选逗号分隔抄送列表，必须与总结收件人一起使用。

该模式不能和 `--keyword`、`--jobs-file`、`--jd`、`--jd-file`、普通 `--email`/`--cc`、`--include-viewed`、搜索来源、搜索订阅或 JD/RAG 问答参数组合。运行前应先确认岗位 JD 已保存在 Boss 平台目录；例如 `data/boss/jobs/物业电工/jd.json`。

总结邮件列出符合与不符合候选人的姓名、ID、理由和动作状态。若符合者已成功转发但后续消息或换电话失败，会明确显示“已转发，但联系动作未完成”。SMTP 发送失败不会删除已落盘的审查结果，但命令会失败，避免误报邮件已送达。

### 全平台模式

`--platform all` 会按固定顺序串行运行：

1. `51job`
2. `liepin`
3. `zhilian`

每个平台使用独立浏览器会话和平台隔离存储。任一平台失败时，整体运行停止并抛出该平台错误。

示例：

```bash
npm run dev -- --platform all --keyword "<关键词>" --jd "<JD 文本>"
```

全平台普通简历抓取支持 `--include-viewed true` 和 `--liepin-forward-contact "<常用联系人姓名>"`；转发开关只在 Liepin 阶段使用。

全平台也支持 direct 搜索来源：

```bash
npm run dev -- --platform all --keyword "<关键词>" --jd-file ./fixtures/jd.txt --search-source direct --application-filter-input-file ./filter-input.json
```

此时三个平台仍按 `51job -> liepin -> zhilian` 串行运行。每个平台会用自己的 application options catalog 解释同一个筛选输入文件；输入文件必须只包含目标平台支持的字段。任一平台筛选应用、候选抽取、转发、评分、导出或邮件失败时，`--platform all` 立即停止并抛出该平台错误。

### 批量模式

`--jobs-file` 接收 JSON 数组，每个条目至少包含 `keyword`，并可选包含 `jd`、`jdFile`、`email`、`cc`、`searchSource`、`applicationFilterInputFile`。

```json
[
  {
    "keyword": "东南亚 销售",
    "jdFile": "./fixtures/jd.txt",
    "email": "user@example.com",
    "cc": ["manager@example.com"],
    "searchSource": "direct",
    "applicationFilterInputFile": "./filter-input.json"
  }
]
```

运行：

```bash
npm run dev -- --platform <platform|all> --jobs-file ./jobs.json
```

批量模式不能与单职位参数 `--keyword`、`--jd`、`--jd-file` 混用。与 `--platform all` 组合时，外层顺序是 jobs 文件顺序，内层顺序是 `51job -> liepin -> zhilian`。

批量模式同样支持在 CLI 上传入 `--include-viewed true`，该开关会应用到本次 jobs 文件内的所有职位和平台。批量普通简历抓取也支持 `--liepin-forward-contact "<常用联系人姓名>"`，但只允许与 `--platform liepin` 或 `--platform all` 组合；`--platform all --jobs-file` 时只在每个职位的 Liepin 阶段转发，51job 和 Zhilian 不使用该开关。

批量模式可在 CLI 上传入 `--search-source direct --application-filter-input-file ./filter-input.json` 作为所有职位的默认 direct 筛选输入；jobs 文件内的 `searchSource` 和 `applicationFilterInputFile` 会覆盖 CLI 默认值。jobs 文件里的相对 `applicationFilterInputFile` 路径按 jobs 文件所在目录解析。

### 搜索订阅模式

`--search-subscription-file` 是独立模式，只负责打开平台搜索条件页、填入关键词、应用支持的搜索条件、读取结果数，并在显式要求时保存搜索条件/订阅。


## 运行要求

### 必需条件

- Node.js 24 LTS 或 Node.js 26。仓库 `.nvmrc` 固定为 `24`，`package.json` 声明 `>=24 <27`。
- 已安装依赖：`npm install`
- 可用模型 API 凭据：`OPENAI_API_KEY`
- 可复用的平台登录态：通过 `npm run login:session -- --platform <platform>` 生成或刷新。
- 使用内部控制台前，先运行 `npm run web:build` 生成 `frontend/dist`，再启动 `npm run api`；开发前端时可改用 `npm run web:dev`。

### 可选配置

完整变量、示例值和注释以 [`.env.example`](./.env.example) 为准，运行时默认值以 [`src/config.ts`](./src/config.ts) 为准。本文只保留影响架构或运行边界的配置组：

| 配置组 | 主要变量 | 说明 |
| --- | --- | --- |
| 数据目录 | `DATA_DIR` | 默认使用 `data/`，所有平台职位数据继续按平台隔离 |
| 模型路由 | `OPENAI_BASE_URL`、`OPENAI_MODEL`、`JD_PARSING_MODEL`、`SCORING_MODEL`、`RAG_MODEL` | 分别控制兼容服务和 JD、评分、RAG 模型覆盖 |
| RAG 检索 | `RAG_EMBEDDING_*`、`RAG_*_TOP_K`、`RAG_RETRIEVAL_MODE`、`RAG_MIN_CONFIDENCE_SCORE` | 默认使用本地 embedding 与 hybrid 检索；低于可信阈值时明确拒答 |
| 向量索引 | `QDRANT_URL`、`QDRANT_API_KEY`、`RAG_VECTOR_COLLECTION` | Qdrant 是可重建索引，不是唯一事实源 |
| 控制台 | `AUTORECRUIT_CONSOLE_*`、`VITE_API_BASE_URL` | 控制监听、轻量 API 保护、请求体和前端目录；不构成完整认证网关 |
| 简历增强提取 | `.venv/bin/python` 与 `crawl4ai` | 不可用时自动回退到内置解析 |
| 邮件 | `SMTP_HOST`、`SMTP_PORT`、`SMTP_USER`、`SMTP_PASS`、`SMTP_FROM` | 用于报告和 Boss 审查总结交付 |

### 浏览器配置

- `BROWSER_ENGINE=cloakbrowser|playwright`：浏览器启动引擎，默认 `cloakbrowser`。如需回退到 Playwright 自带 Chromium，设置为 `playwright`。
- `PLAYWRIGHT_HEADLESS=true`：启用 headless。默认是 headed；Liepin 始终强制使用有头浏览器，此变量对 Liepin 不生效。
- `PLAYWRIGHT_SEARCH_PAGE_TIMEOUT_MS`：搜索入口到候选列表 ready 的总等待预算，默认 `20000`。
- `PLAYWRIGHT_EMPTY_RESULTS_STABLE_MS`：51job 空列表稳定窗口，默认 `2000`。
- `PLAYWRIGHT_API_FALLBACK_TIMEOUT_MS`：Liepin/Zhilian API fallback 短等待上限，默认 `3000`。
- `PLAYWRIGHT_RESUME_DETAIL_TIMEOUT_MS`：简历详情打开/就绪总等待预算，默认 `20000`。
- `PLAYWRIGHT_{ACTION|CANDIDATE}_DELAY_{MIN|MAX}_MS`：全局操作和候选人等待覆盖。
- `PLAYWRIGHT_<PLATFORM>_{ACTION|CANDIDATE}_DELAY_{MIN|MAX}_MS`：平台级覆盖；默认值和加权规则统一见[浏览器操作节奏](#浏览器操作节奏)。
- `PLAYWRIGHT_REUSE_BROWSER=true|false`：全平台可复用浏览器默认开关。默认不启用；51job、Liepin、Zhilian 和 Boss 有平台默认值，会默认启用。
- `PLAYWRIGHT_51JOB_REUSE_BROWSER=false` / `PLAYWRIGHT_ZHILIAN_REUSE_BROWSER=false` / `PLAYWRIGHT_BOSS_REUSE_BROWSER=false`：关闭对应平台可复用浏览器。
- `PLAYWRIGHT_LIEPIN_REUSE_BROWSER=false`：关闭 Liepin 可复用浏览器；默认开启。开启时流程结束会关闭最后的详情页、停留在人才搜索页并保留浏览器，下次 Liepin 运行会优先连接这个窗口。
- `PLAYWRIGHT_51JOB_REUSE_CDP_PORT` / `PLAYWRIGHT_LIEPIN_REUSE_CDP_PORT` / `PLAYWRIGHT_ZHILIAN_REUSE_CDP_PORT` / `PLAYWRIGHT_BOSS_REUSE_CDP_PORT`：各平台可复用浏览器的本地 CDP 端口，默认分别是 `19325`、`19327`、`19329`、`19331`。

CloakBrowser 自带诊断和维护 CLI：

```bash
npx cloakbrowser info        # 查看 CloakBrowser 版本、平台、二进制路径和安装状态
npx cloakbrowser install     # 预下载 CloakBrowser Chromium
npx cloakbrowser update      # 检查并下载新版 Chromium
npx cloakbrowser clear-cache # 清除 CloakBrowser 浏览器缓存
```

正常多平台运行时应保持 `STORAGE_STATE_PATH` 未设置，让系统按平台选择默认 storage state。显式设置时，路径必须与当前平台匹配，避免把一个平台的登录态套用到另一个平台。

```bash
npm run validate:resumes
npm run capture:resume-dom -- --platform <platform> <jobKey> <searchKeyword> <candidateId>
npm run capture:resume-dom -- --platform <platform> <jobKey> <candidateId> <resumeUrl>
```

`export:resume-docx` 默认使用 `/Users/Admin/Downloads/简历模板.docx`。按平台、职位和候选人 ID 导出时，默认写入 `data/<platform>/jobs/<jobKey>/exports/resumes/<candidateId>-<name>.docx`；也可以通过 `--output` 或 `--output-dir` 指定输出位置。导出会尝试从已保存的详情页结构化内容中识别候选人本人头像并嵌入模板头像位；默认头像、学校图、页面图标、SVG 资源和相似人才头像都会被排除。如果无法确认或下载本人头像，则保留无头像导出，不放入模板示例头像或他人头像。

平台专项回归命令名保留 `experimental` 前缀以兼容已有脚本：

```bash
npm run test:experimental:zhilian
```

## 数据模型与落盘结构

核心类型定义在 `src/types/job.ts`：

- `NormalizedJob`：结构化 JD。
- `JobRecord`：职位持久化记录，包含 `jobKey`、`platform`、`searchKeyword`、JD 原文、标准化 JD 和邮件收件人。
- `CandidateListItem`：搜索结果页候选人卡片。
- `CandidateResume`：结构化简历，包含基础信息、工作经历、项目经历、教育经历、技能、证书和可选 `candidateShareUrl`。
- `CandidateScoreArtifact`：评分产物，可能是成功评分，也可能是失败记录。
- `RunResult`：一次执行的轻量摘要。
- `BossChatReviewRun`：一次 Boss 未读会话审查的候选人判断、动作状态和错误摘要。
- `JobResultsMarkdownExport`：导出的 Markdown 结果视图。

默认数据目录是 `data/`，每个职位位于：

```text
data/<platform>/jobs/<jobKey>/
```

目录内容：

- `jd.json`：职位记录，也是 rerun 复用 JD 载荷和邮件收件人的来源。
- `seen-ids.json`：已成功处理的候选人 ID。
- `resumes/<candidateId>.json`：结构化简历。
- `snapshots/<candidateId>.txt`：原始页面文本快照。
- `snapshots-dom/<candidateId>.json`：DOM 抽取快照。
- `scores/<candidateId>.json`：评分产物。
- `results/<timestamp>.json`：轻量运行摘要。
- `exports/latest.md`：最新导出的 Markdown 报告。
- `rag/sources.jsonl`：RAG 事实源记录，包含 JD、历史对话等来源。
- `rag/chunks.jsonl`：RAG chunk 事实源；Qdrant 只是可重建索引。
- `rag/embeddings.jsonl`：RAG embedding 缓存，按 provider、model 和内容 hash 复用向量。
- `rag/conversations/<conversationId>.jsonl`：导入的职位历史对话，按 turn id 去重合并。
- `rag/index-manifest.json`：最近一次 RAG 索引构建摘要。
- `rag/answer-logs.jsonl`：已保存职位的 RAG 问答审计日志，记录 `logId`、问题、答案、confidence、拒答原因、引用 sources 和人工反馈；错误反馈可包含 `errorType`；临时 JD 问答和离线评测不写入。
- `exports/resumes/<candidateId>-<name>.docx`：按 Word 模板离线导出的候选人 DOCX 简历。

Boss 自动沟通另有独立运行状态：

- `data/boss/chat-review/reviewed-conversation-ids.json`：已完成处理或已成功转发的会话 ID。去重按事件处理：当前仍有新红点的会话即使 ID 已存在也可再次处理；该列表只阻止没有新红点的失败恢复项被重复执行。
- `data/boss/chat-review/runs/<timestamp>.json`：每次审查的候选人 ID、岗位、硬性条件证据、匹配结论、转发/聊天/换电话状态和错误。最新状态仍为 `failed` 且尚未转发的会话可作为重试来源。

控制台运行状态独立于职位数据：

- `data/runtime/tasks/`：共享队列的任务记录。
- `data/runtime/schedules/`：自动运行计划定义和当前状态。
- `data/runtime/schedule-runs/`：计划每一轮的任务顺序、完成、取消和失败记录。

历史平台化改造前的数据可通过 `npm run migrate:platform-storage` 从 `data/jobs/` 迁移到 `data/51job/jobs/`，并回填缺失的 `platform` 字段。迁移脚本会拒绝覆盖已存在的目标目录。

## 邮件与失败语义

收件人和抄送列表按职位维度持久化：

- 首次运行传入 `--email` 或 `--cc` 时，会写入对应 `jd.json`。
- 后续同一 `jobKey` 重跑时，如果未显式传入，会复用已保存配置。
- 再次显式传入时，以本次参数为准并覆盖保存值。

失败处理：

- 评分失败：保存失败评分产物，候选人仍保持已处理。
- 上游详情打开或提取失败：不写入 seen，下次继续重试。
- 导出失败：不让整次运行失败，错误记录到 summary。
- 邮件失败：不让整次运行失败，错误记录到 summary。
- 空结果运行：属于成功运行，可发送“无新增候选人”通知邮件。
- Zhilian 有新增候选人的报告邮件要求当前运行评分产物都带唯一 `candidateShareUrl`；缺失或重复会导致邮件发送失败并记录到 summary。
- Zhilian 搜索页卡片可能不暴露可用详情链接，候选人 ID 提取会优先读取卡片 Vue props 中的 `userMasterId`、`resumeNumber` 等字段，再使用 API fallback。
- Boss 简历打开或解析失败且尚未转发：记录失败并保留重试来源；即使红点已消失，也可从历史运行恢复当前列表中仍可见的会话。
- Boss 已转发但联系失败：保留 `forwarded: true`，不自动重复转发；总结邮件明确列出联系未完成原因。
- Boss 总结邮件失败：审查运行结果已经落盘，但命令返回失败，避免把未送达误报为成功。

## 维护建议

- 修改 `src/browser/resume-detail.ts` 这类启发式模块时，优先使用已保存快照离线重解析验证，再考虑 live 流程。
- 修改平台搜索入口、列表提取或详情解析时，保留多重 DOM fallback，避免假设页面结构稳定。
- 修改全平台或批量语义时，同时验证单平台、`--platform all` 和 `--jobs-file` 的顺序与错误传播。
- 修改报告或邮件逻辑时，覆盖有新增候选人、无新增候选人、评分失败和 Zhilian 分享链接校验路径。
- 修改 DOCX 简历导出时，保留模板正文、页眉和头像关系的兼容性；头像只能来自候选人本人详情页证据，无法确认时应省略头像而不是使用默认头像、相似人才头像或模板示例头像。
