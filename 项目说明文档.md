# 项目说明文档

## 项目定位

AutoRecruit 是一个 CLI-first、本地优先的招聘自动化工具。它面向招聘侧的重复执行场景，把平台搜索、候选人抓取、简历解析、JD 匹配评分、Markdown 报告和邮件发送串成可复跑、可审计、按平台隔离的数据流程。

项目当前包含六类能力：

- 普通简历抓取主流程：支持单职位、`--platform all`、`--jobs-file` 批量、保存搜索入口和直接筛选输入入口、已查看筛选切换、Liepin 常用联系人转发、Zhilian 分享链接邮件交付。
- 搜索条件维护流程：支持筛选发现、应用层筛选输入导出/校验、搜索订阅结果数检查和显式保存搜索条件；该流程独立于 JD、候选人抓取、评分、报告和邮件。
- 内部运营控制台：提供本地 HTTP API 和 React 前端，用于查看任务、职位、候选人、筛选 catalog、RAG 问答和运营诊断入口；任务执行仍复用 CLI 语义和本地数据目录。
- 智能助手：在控制台中把自然语言整理成结构化任务草稿，支持普通抓取、批量、搜索订阅、登录刷新、RAG 运营和候选人问答；确认后才进入现有后端执行路径。
- 离线导出工具：支持把已保存的结构化简历按本地 Word 模板导出为 DOCX，并在能确认详情页本人头像时放入模板头像位置。
- Boss 单平台扩展：支持复用当前搜索页抓取/筛选、站内同事或邮件转发，以及在当前沟通页审查未读候选人；符合者先转发简历，再从常用语发送求简历消息并申请换电话；不符合者从常用语发送不合适消息；最后发送审查总结邮件。

当前浏览器平台分为三平台主流程和 Boss 单平台扩展。`--platform all` 始终只包含 51job、Liepin、Zhilian；Boss 必须显式使用 `--platform boss`：

| 平台 | CLI 名称 | 登录态文件 | 主要入口 | 核心规则 |
| --- | --- | --- | --- | --- |
| 前程无忧 | `51job` | `storage-state.json` | 订阅搜索页或直接人才搜索页 | 默认有头和可复用浏览器；保存入口通过订阅卡片进入并确认 `关键词：<keyword>`；直接入口清空筛选后输入关键词和应用筛选；默认勾选 `我已看` |
| 猎聘 | `liepin` | `storage-state.liepin.json` | 找人/快捷搜索或直接关键词搜索 | 默认有头和可复用浏览器，且始终强制有头；保存入口进入 `找人` 快捷搜索；直接入口清空筛选后输入关键词和应用筛选；默认勾选 `隐藏已查看`；可选常用联系人转发 |
| 智联招聘 | `zhilian` | `storage-state.zhilian.json` | `/app/search` 快捷搜索标签或直接关键词搜索 | 默认有头和可复用浏览器；保存入口点击快捷搜索标签并确认 `关键词：<keyword>`；直接入口不点快捷标签，清空筛选后输入关键词和应用筛选；默认勾选 `未看过`；自动复制分享链接 |
| Boss 直聘 | `boss` | `storage-state.boss.json` | 当前人才搜索页或 `/web/chat/index` 沟通页 | 默认有头和可复用浏览器；只允许单平台运行，不进入 `all`；复用当前页面避免重复登录和新开标签；支持配置式简历转发和独立自动沟通审查 |

项目不是通用 ATS，也不是完整 SaaS 或平台后台替代品。当前 Web 能力是本地内部运营控制台，重点仍是把已定义的本地运行模式稳定落页、保留可离线复盘的数据产物，并维持四个平台的适配边界以及 Boss 不进入 `all` 的产品约束。

## 主流程

一次完整运行大致如下：

`CLI 参数 / jobs 文件` -> `JD 标准化` -> `JobRecord` -> `平台搜索` -> `候选人列表` -> `新增候选人` -> `简历快照和结构化简历` -> `seen-ids.json` -> `评分产物` -> `RunResult` -> `Markdown 报告` -> `邮件发送`

该链路描述普通抓取。Boss 自动沟通是独立流程，不进入普通抓取、批量或 `all`：`当前沟通页` -> `未读会话快照` -> `岗位已保存 JD` -> `在线简历` -> `严格条件/评分判断` -> `符合者先转发、发求简历常用语并换电话 / 不符合者发不合适常用语` -> `审查结果` -> `总结邮件`。

具体语义：

1. 用户指定 `--platform` 和搜索关键词。首次运行新 `jobKey` 时还需要 `--jd` 或 `--jd-file`。
2. 系统优先根据搜索关键词生成 `jobKey`。如果该平台下已有 `jd.json`，直接复用历史 JD 原文和标准化结果，不再读取或解析本次 JD 输入。该文件同时保存岗位搜索来源、direct 筛选输入及标准化条件、报告邮箱和 Boss 转发目标；同一配置只保留一份，不追加重复历史记录，也不重写内容完全相同的文件。
3. Playwright 读取当前平台的 storage state，并验证登录态仍可访问平台已鉴权入口。
4. 进入平台搜索入口。新岗位默认 `--search-source saved` 复用平台已保存订阅/快捷搜索；`--search-source direct` 会清空旧条件、输入本次关键词、应用 `--application-filter-input-file` 的筛选条件，再进入正常简历抓取。岗位首次保存 direct 条件后，后续只传平台和关键词即可复用，不再依赖原筛选 JSON 或 catalog 文件；显式传入新的 `--search-source` 时才替换历史搜索配置。
5. 对齐平台默认已查看筛选：51job 默认勾选 `我已看`，Liepin 默认勾选 `隐藏已查看`，Zhilian 默认勾选 `未看过`；显式传入 `--include-viewed true` 时取消对应平台的已查看筛选，再提取候选人 ID。明确空结果文案或稳定空列表视为成功的 0 候选人运行。
6. 使用 `seen-ids.json` 过滤历史已处理候选人，只打开新增候选人的简历详情。
7. 成功解析的简历会保存原始文本快照、DOM 快照和结构化 JSON。
8. 成功保存简历的候选人先写入 `seen-ids.json`，再进入评分阶段；详情打开或提取失败的候选人下次仍可重试。
9. 评分从本地简历 JSON 执行。评分失败会写入失败评分产物，不回滚 seen 状态。
10. 生成轻量运行结果，导出最新 Markdown 报告；如配置收件人，则发送邮件。

## 平台行为

### 51job

51job 通过订阅搜索页进入人才搜索。流程会点击匹配的订阅关键词卡片，确认当前活跃订阅详情面板标题等于本次关键词，再点击该面板内的 `去搜索` 人才搜索入口；进入人才搜索页后还会确认页面显示 `关键词：<keyword>`，避免复用浏览器状态或其它订阅条件串入本次运行。流程保留多组 selector fallback，并把 `没有搜索到相关的人才` 这类明确空结果视为成功的 0 候选人运行。

候选卡片中的 `no_interested_<id>` 只作为候选人锚点，不作为详情入口。详情打开会优先点击候选人姓名、详情行等区域，并在单个点击入口无效时继续尝试其它 fallback，避免一个无效区域耗尽整次详情打开等待预算。

默认会显式勾选搜索页的 `我已看`；传入 `--include-viewed true` 时才会取消 `我已看`。51job 默认启用可复用浏览器；打开人才搜索页后会关闭同一上下文里多余的“我的订阅”页面，避免复跑时订阅页标签堆积。

### Liepin

Liepin 始终使用有头浏览器，`PLAYWRIGHT_HEADLESS=true` 对 Liepin 不生效。浏览器启动引擎默认是 CloakBrowser；如需回退到 Playwright 自带 Chromium，可设置 `BROWSER_ENGINE=playwright`。Liepin 默认启用可复用浏览器：流程结束会关闭最后的详情页、停留并置前招聘端人才搜索页，下次运行优先通过本地 CDP 端口连接这个窗口继续。

搜索流程必须进入招聘端人才搜索面，必要时从首页点击顶部导航 `找人`，再点击本次关键词对应的快捷搜索标签。默认确认 `隐藏已查看` 已勾选；传入 `--include-viewed true` 时才取消该筛选。勾选或取消已查看筛选后，旧的 `search-resumes` 响应不能复用，候选提取以 DOM 为先，只有 DOM 无候选或缺安全详情 URL 时才短等 API fallback。

Liepin 主流程支持 `--liepin-forward-contact "<常用联系人姓名>"`。该开关只处理本地 `seen-ids.json` 过滤后的新增候选人：打开详情页后先点击 `转发`，在弹窗里选择联系人并确认，成功后才解析简历和标记 seen；转发失败时该候选人保持可重试。

Liepin 站内点击、输入和候选人间隔默认使用随机 `2000-3000ms` 节奏，点击会尽量先移动鼠标到目标元素。可通过 `PLAYWRIGHT_LIEPIN_ACTION_DELAY_*` 和 `PLAYWRIGHT_LIEPIN_CANDIDATE_DELAY_*` 覆盖。

### Zhilian

Zhilian 登录页是 `https://passport.zhaopin.com/org/login`，已鉴权搜索页是 `https://rd6.zhaopin.com/app/search`；`https://rd6.zhaopin.com/desktop` 不是搜索流程目标页。

Zhilian 默认启用可复用浏览器。搜索必须点击已保存的快捷搜索标签，标签文本需要包含本次运行传入的原始 `--keyword`，点击后还必须确认条件中出现 `关键词：<keyword>`。默认会显式勾选 `未看过`；传入 `--include-viewed true` 时只取消可见的 `未看过`，不改动 `未聊过`。如果取消或勾选后快捷搜索条件消失，流程会重新选择该快捷搜索标签，再次设置 `未看过` 并重新确认关键词条件。

候选列表提取 DOM-first。当前搜索卡片可能没有可用详情链接或显式 ID，因此 DOM 提取会优先读取卡片 Vue props 中的 `userMasterId`、`resumeNumber`、`resumeK`、`resumeT` 等字段；DOM 无候选时才等待 API fallback 或明确空结果。

Zhilian 简历详情是 `/app/search` 页内 modal，不是新标签页。解析时只读取 modal 子树，并通过 `转给同事` -> `链接转发` 复制分享链接，保存为 `candidateShareUrl`。Zhilian 有新增候选人的报告邮件必须使用当前运行复制出的唯一分享链接；缺失或重复会作为邮件发送错误记录。

### Boss

Boss 登录入口是 `https://www.zhipin.com/web/user/?ka=header-login`，默认使用 `storage-state.boss.json`、独立浏览器 profile 和本地 CDP 端口 `19331`。运行时优先复用已经打开的 Boss 搜索页或沟通页；不要反复访问登录页，也不要为每次任务创建新 Boss 标签。登录态缺失或失效时，才显式运行登录保存命令并由用户完成登录。

Boss 普通抓取支持当前搜索页准备、direct 条件回放、候选人列表/简历解析和配置式转发。转发参数出现时必须成对提供：`--boss-forward-mode colleague|email` 与 `--boss-forward-recipient`。显式配置会保存到岗位记录，后续同岗位普通抓取可省略并复用。站内同事模式填写姓名并选择唯一匹配项；邮件模式填写收件地址；两种模式的留言都固定使用候选人 ID。转发发生在简历解析和 seen 标记之前，失败候选人保持可重试。

Boss 自动沟通是独立模式。流程只点击左侧“沟通”进入 `https://www.zhipin.com/web/chat/index`，选择“未读”，先快照所有带红点的会话，再按会话 ID 逐个处理。岗位名用于读取 `data/boss/jobs/<jobKey>/jd.json`；缺少 JD 时不打开会话，保持未读。显式转发和总结邮件配置会去重保存到 `data/boss/chat-review/automation-settings.json`。后续省略参数时，转发目标按“岗位记录优先、Boss 平台默认其次”复用，总结邮件复用平台配置；没有转发配置的会话不会被打开。

在线简历入口必须使用 Playwright 原生 locator 点击。会话切换后还要等待 Vue 当前会话 ID 和候选人基础信息（包括 `ageDesc`）完成异步水合，再读取摘要；简历详情可合并 iframe `IFRAME_DONE` 的 WASM 解密数据，但原始解密载荷不会落盘。

`--boss-chat-require-all true` 当前为“物业电工”提供严格证据规则：年龄严格小于 47 岁、明确持有高压电工证、明确持有低压电工证、明确具有物业行业电工经验、至少一家公司连续工作 24 个月，并且简历明确说明候选人是上海人。期望城市或现居地为上海不能替代籍贯证据。任一项缺少明确证据都判为不符合，不做推测，但上海籍确认有一个受控例外：其他五项全部满足、简历没有明确上海或非上海籍贯、且教育经历显示在上海学校就读时，关闭简历后在聊天框输入 `是上海人吗？` 并发送，状态记为等待确认；此时不转发、不拒绝、不换电话，也不把会话加入已审查 ID，等待候选人回复后再次审查。上海就读经历本身不算上海人证据。

符合和不符合候选人的聊天消息禁止直接填写编辑器，必须点击 `.toolbar-icon.changyongyu` 打开常用语浮窗，并从 `.phrase-content li` 选择页面现有话术后点击主发送按钮。符合候选人的动作顺序固定为：保持简历浮窗打开并先完成转发；立即记录 `forwarded: true`；关闭简历浮窗；选择 `方便发一份你的简历过来吗？` 并发送；点击“换电话”并确认。不符合者关闭简历浮窗后选择页面完整话术 `对不起，看了你的简历以后觉得不太合适，希望你早日找到满意的工作机会` 并发送，不转发、不换电话。上海籍待确认候选人是唯一例外：直接在聊天框输入 `是上海人吗？` 后点击主发送按钮；若编辑器已有其他草稿则禁止覆盖。消息和换电话有幂等保护；转发成功后联系失败会保留转发成功状态，不自动重复转发。

尚未成功转发的失败会话会从历史审查运行中恢复。即使首次打开已清除红点，只要会话仍在当前未读列表中就可重试。运行结束保持当前 Boss 沟通页打开。

## 筛选发现与搜索订阅

本节只描述搜索筛选项发现、应用层筛选输入和搜索订阅结果数检查。这些能力不解析 JD、不创建职位记录、不抓候选人、不评分、不导出报告、不发送邮件，也不改 seen 状态。

### 筛选发现

筛选发现入口：

```bash
npm run discover:filters -- --platform <51job|liepin|zhilian|boss|all> --keyword "<关键词>"
```

`--platform all` 按 `51job -> liepin -> zhilian` 串行执行，任一平台失败立即停止；Boss 发现必须单独使用 `--platform boss`。发现结果写入：

- `data/<platform>/filter-catalog/latest.json`
- `data/<platform>/filter-catalog/<timestamp>.json`

Liepin 和 Zhilian 默认采用较保守的静态读取；需要打开下拉、弹层和级联树时，显式传入 `--slow-click true`。Zhilian 发现必须停留在 `/app/search`，复用保存的快捷搜索标签并确认搜索条件面板；不要用泛文本点击 `筛选` 或 `使用高级搜索`，这些文字可能命中推荐页或顶部导航。

Liepin 行业完整树单独采集：

```bash
npm run discover:liepin-industry-tree -- --keyword "优衣库"
```

该命令采集 `当前行业`、`期望行业` 的完整大类/叶子树，并合并回 `latest.json` 与 `application-filter-options.latest.json`。行业父级只作为 `pathLabels` 路径节点，叶子才是可提交值。

### 应用层筛选输入

筛选 catalog 可以导出成应用层 options：

```bash
node --import ./scripts/node-ts-hooks.mjs src/scripts/export-application-filter-options.ts <platform>
```

导出文件是 `data/<platform>/filter-catalog/application-filter-options.latest.json`。应用层用 `groups.singleSelect`、`groups.textInput`、`groups.salaryRange`、`groups.numberRange` 分组；`fieldsById.<fieldId>.kind` 决定输入形态。

常见输入形态：

- 普通单选：`"education": "本科"`
- 文本/弹层输入：`"company_name": "迅销"`
- 级联或同名叶子：`{ "value": "电子商务", "pathLabels": ["AI/互联网/IT", "电子商务"] }`
- 数字范围：`"age": { "min": 25, "max": 35 }`
- 薪资范围：`"expected_salary": { "min": "10万", "max": "30万" }`
- 自定义输入：`{ "label": "自定义", "input": { "min": "大专", "max": "本科" } }`

输入校验命令：

```bash
npm run validate:filter-input -- <platform> <input.json>
```

### 平台筛选要点

51job 当前导出 20 个字段，搜索订阅回放覆盖这些字段：`work_years`、`education`、`school_nature`、`recent_activity_time`、`gender`、`living_location`、`expected_location`、`expected_salary`、`job_hopping_count`、`job_status`、`language`、`engaged_industry`、`expected_industry`、`engaged_function`、`expected_function`、`age`、`graduation_year`、`work_type`、`company_nature`、`major`。这些字段覆盖单选、动态文本输入、多级菜单、期望月薪和自定义输入。搜索订阅页会先尝试清空已有筛选；文本输入和弹层输入在应用新值前会清理已有 tag，避免可复用浏览器复跑时残留旧条件。页面出现 `没有搜索到相关的人才` 时按结果数 0 处理。

Liepin 当前导出 25 个招聘端可见字段。已验证回放字段包括 `work_years`、`education`、`school_nature`、`recent_activity_time`、`gender`、`language`、`living_location`、`expected_location`、`expected_salary`、`current_salary`、`job_hopping_count`、`job_status`、`resume_language`、`overseas_work_experience`、`management_experience`、`age`、`engaged_industry`、`engaged_function`、`expected_industry`、`expected_function`、`company_name`、`school_name`、`major`。薪资单位是年薪“万”；年龄单位是岁；`company_name`、`school_name`、`major` 是不限枚举的行内文本输入。`overseas_work_experience` 和 `management_experience` 是 checkbox，实站验证应读取真实 checked 状态，不只看已选条件文字。`keyword_title` 由搜索订阅 `keyword` 控制，不重复作为 `applicationFilter`。

Zhilian 当前导出 19 个字段。已验证回放字段包括 `education`、`work_years`、`school_nature`、`recent_activity_time`、`gender`、`job_status`、`language`、`talent_type`、`talent_photo`、`resume_language`、`job_hopping_count`、`living_location`、`hukou_location`、`engaged_industry`、`expected_industry`、`engaged_function`、`expected_function`、`age`、`expected_salary`。`education` 和 `work_years` 支持预设值，也支持页面“自定义”下拉范围；`age` 会优先使用页面预设范围，非预设范围会改用自定义下拉；`expected_salary` 是智联月薪。城市、户口、行业和职位级联字段建议提交 `{ "value": "...", "pathLabels": [...] }`。`expected_location` 当前不是 Zhilian 可见 catalog 字段。

Boss 当前导出 10 个字段：`work_years`、`education`、`school_nature`、`recent_activity_time`、`gender`、`job_status`、`job_hopping_count`、`candidate_position_requirement`、`age`、`expected_salary`。当前分组为 8 个单选、1 个年龄范围和 1 个薪资范围。Boss direct 和搜索订阅回放都必须复用当前搜索页，逐项确认 `applied`；任一条件 `skipped` 或 `failed` 时中止，避免从半应用条件中抓取候选人。

搜索订阅应用筛选时，输出中的 `conditionResults` 会逐项记录 `applied`、`skipped` 或 `failed`，`allConditionsApplied` 汇总本次条件是否全部落页。只要存在 `skipped` 或 `failed`，显式保存订阅时会拒绝保存半成品条件。

普通简历抓取的 direct 模式复用同一套 application options 和筛选输入格式，但 direct 是正常抓取流程的一部分：应用筛选后会继续候选提取、详情打开、评分、导出和邮件。搜索订阅模式仍然独立，只做条件落页、结果数读取和可选保存。

## 内部运营控制台与智能助手

控制台由 `npm run api` 启动本地 HTTP 服务，默认监听 `127.0.0.1:4180`，并从 `frontend/dist` 提供已构建的前端静态文件。开发前端时可单独运行 `npm run web:dev`，Vite 默认监听 `127.0.0.1:5173`，并把 `/api` 代理到 `127.0.0.1:4180`。

```bash
npm run web:build
npm run api

# 开发模式可另开终端：
npm run web:dev
```

可用环境变量：

- `AUTORECRUIT_CONSOLE_HOST`：控制台 API 监听地址，默认 `127.0.0.1`。
- `AUTORECRUIT_CONSOLE_PORT`：控制台 API 端口，默认 `4180`。
- `AUTORECRUIT_CONSOLE_API_KEY`：可选 bearer token；设置后所有 `/api/*` 请求都需要 `Authorization: Bearer <token>`。
- `AUTORECRUIT_CONSOLE_MAX_BODY_BYTES`：请求体上限，默认 2 MiB。
- `AUTORECRUIT_CONSOLE_FRONTEND_DIR`：静态前端目录，默认 `frontend/dist`。
- `VITE_API_BASE_URL`：前端 API 基址，默认 `/api`。

控制台后端使用单进程 `TaskQueue`，普通抓取、批量、搜索订阅、登录刷新、Boss 自动沟通和 RAG 运营任务都会串行入队，任务记录落在 `data/runtime/tasks/`。已保存职位、候选人、评分、运行结果、筛选 catalog 和 RAG 文件仍读取原有 `data/<platform>/...` 结构。

核心 API 包括：

- `GET /api/health`
- `GET /api/dashboard/health`
- `GET /api/tasks`、`GET /api/tasks/<taskId>`
- `POST /api/tasks/resume-capture`
- `POST /api/tasks/batch`
- `POST /api/tasks/search-subscription`
- `POST /api/tasks/login-refresh`
- `POST /api/tasks/boss-auto-chat`
- `POST /api/tasks/rag-ops`
- `GET /api/jobs`、`GET /api/jobs/<platform>/<jobKey>`
- `GET /api/jobs/<platform>/<jobKey>/runs`
- `GET /api/jobs/<platform>/<jobKey>/candidates`
- `GET /api/jobs/<platform>/<jobKey>/candidates/<candidateId>`
- `POST /api/rag/answer`
- `POST /api/rag/conversations`
- `GET /api/ops/filter-catalogs`
- `GET /api/ops/application-filter-options?platform=<platform>`
- `POST /api/ops/filter-inputs`

控制台的 `智能助手` 页面对应三组 API：

- `POST /api/assistant/chat`：根据对话生成或更新结构化 `AssistantDraft`。
- `POST /api/assistant/validate`：重新清洗并校验 draft，返回缺失字段、警告和可确认预览。
- `POST /api/assistant/confirm`：确认 draft。除 `rag-answer` 外，都会复用 shared normalizer 后进入 `TaskQueue`；`rag-answer` 直接调用 RAG 问答并返回答案。

控制台的 `配置` 页面提供模型调用配置入口：`中转地址`、`模型名称` 和 `API Key`。这些配置会作为请求级覆盖随 `POST /api/assistant/chat` 和 `POST /api/rag/answer` 发送，用于本次智能助手草稿生成和控制台 RAG 问答；不写入任务草稿、不进入 `TaskQueue`、不写入任务记录、RAG answer log 或后端配置文件。中转地址和模型名称会保存在浏览器本地以便下次复用，API Key 只保存在当前浏览器会话。留空时后端继续使用 `OPENAI_BASE_URL`、`OPENAI_MODEL`、`RAG_MODEL` 和 `OPENAI_API_KEY`。确认执行后的 JD 解析、评分、抓取、导出、邮件和 RAG 运维任务仍按各自原有环境变量运行，不受这个控制台配置影响。

智能助手只负责把自然语言整理成受控草稿，不是命令执行器。模型输出会按任务类型过滤字段，任意 shell、npm script、文件写入或绕过 normalizer 的请求应被拒绝或作为风险提示；最终确认仍必须通过 `src/server/task-normalizers.ts` 的同一套校验。`argvPreview` 只是给用户核对的预览，不是执行来源。

助手支持的 draft 类型包括 `resume-capture`、`batch`、`search-subscription`、`login-refresh`、`boss-auto-chat`、`rag-ops` 和 `rag-answer`。Boss 草稿和确认仍必须通过共享 normalizer，平台固定为 Boss，并明确展示打开未读会话、转发和联系候选人的风险。`rag-answer` 保持候选人问答语义：不会创建任务、不会打开浏览器、不会抓候选人、不会评分、不会导出报告或发送邮件；已保存职位和临时 JD 的索引/日志行为与 `--jd-question` / `--rag-question` 保持一致。

## 运行模式

### 单职位模式

首次运行一个新职位：

```bash
npm run dev -- --platform <51job|liepin|zhilian|boss> --keyword "<关键词>" --jd "<JD 文本>"
```

或从文件读取 JD：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>" --jd-file ./fixtures/jd.txt
```

重跑已有职位时可以不再提供 JD：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>"
```

根据已保存或临时 JD 回答候选人问题，不会打开浏览器、抓候选人、评分、导出或发邮件：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>" --jd-question "这个岗位薪资范围是多少？"
npm run dev -- --platform <platform> --keyword "<关键词>" --jd-file ./fixtures/jd.txt --jd-question "这个岗位需要出差吗？"
```

`--jd-question` 会优先复用 `data/<platform>/jobs/<jobKey>/jd.json` 中的 JD 原文和结构化 JD，并通过本地 RAG 事实源、Qdrant 向量检索、本地 BM25 关键词检索、融合排序和轻量重新排序回答；如果该职位尚未建立 RAG 索引，会自动索引当前 JD。找不到已保存 JD 时才读取本次 `--jd` 或 `--jd-file`，并使用临时轻量检索回答，不创建职位记录。`--rag-question` 是同一功能的别名。输出 JSON 会包含 `answer`、`answered`、`confidence`、`noAnswerReason` 和 `sources`；没有可信 JD/已确认历史对话，或最高可信片段低于 `RAG_MIN_CONFIDENCE_SCORE` 时，`answered` 为 `false`，系统直接回答资料未说明，不调用模型编造。

也可以显式维护职位 RAG 索引和历史对话：

```bash
npm run rag:index -- --platform <platform> --keyword "<关键词>"
npm run rag:api -- --host 127.0.0.1 --port 3978
npm run rag:ask -- --platform <platform> --keyword "<关键词>" --question "这个岗位需要出差吗？"
npm run rag:baseline
npm run test:rag:offline
npm run rag:answer-eval -- --platform <platform> --keyword "<关键词>" --eval-file ./rag-answer-eval.json
npm run rag:doctor -- --platform <platform> --keyword "<关键词>" [--question "这个岗位需要出差吗？"]
npm run rag:doctor:batch -- --file ./rag-doctor-jobs.json [--question "这个岗位需要出差吗？"] [--fail-on-issue true]
npm run rag:eval -- --platform <platform> --keyword "<关键词>" --eval-file ./rag-eval.json
npm run rag:export-answer-eval -- --platform <platform> --keyword "<关键词>" --output ./rag-answer-eval.json
npm run rag:feedback -- --platform <platform> --keyword "<关键词>" --log-id <logId> --correct true [--note "回答准确"] [--reviewer "reviewer"]
npm run rag:feedback -- --platform <platform> --keyword "<关键词>" --log-id <logId> --correct false --error-type wrong_fact [--note "错误原因"] [--reviewer "reviewer"]
npm run rag:ingest-conversation -- --platform <platform> --keyword "<关键词>" --conversation-id conv-001 --conversation-file ./conversation.jsonl
npm run rag:ingest-conversations -- --file ./conversations.jsonl [--dry-run true] [--fail-on-error false] [--doctor true]
npm run rag:inspect -- --platform <platform> --keyword "<关键词>" [--question "这个岗位需要出差吗？"]
npm run rag:metrics -- --file ./rag-review-jobs.json [--output ./rag-metrics.json]
npm run rag:ops -- --file ./rag-review-jobs.json [--question "这个岗位需要出差吗？"] [--policy ./rag-metrics-policy.json] [--output ./rag-ops.md] [--fail-on-issue true]
npm run rag:regression -- --suite-file ./fixtures/rag/regression.json
npm run rag:review -- --platform <platform> --keyword "<关键词>" [--output ./rag-review.md]
npm run rag:review:batch -- --file ./rag-review-jobs.json [--output ./rag-review.md] [--fail-on-issue true]
npm run rag:rebuild -- --platform <platform> --keyword "<关键词>"
npm run rag:seed-fixtures -- --index true
```

`rag:inspect` 是只读诊断命令：不生成回答、不写入索引。无 `--question` 时只汇总本地 RAG 文件；带 `--question` 时会展示 Qdrant 向量检索、本地关键词检索、融合重排后的命中片段，方便排查候选人问题为什么命中或漏掉某些资料。

`rag:api` 是产品化 RAG HTTP 接口，默认监听 `127.0.0.1:3978`，可用 `--host`、`--port`、`RAG_API_HOST`、`RAG_API_PORT` 覆盖。设置 `RAG_API_KEY` 或 `--api-key` 后，请求必须带 `Authorization: Bearer <token>`。当前提供 `GET /health`、`POST /v1/rag/answer` 和 `POST /v1/rag/conversations`：问答接口复用 `answerQuestionWithRag`，对话导入接口复用 `ingestConversation`，因此和 CLI 保持同样的 jobKey/keyword 规则、verified 招聘方事实规则、元数据过滤、自动索引和 answer log 行为。生产部署建议放在内网或网关后，并限制 `RAG_API_MAX_BODY_BYTES`。该接口是内部产品化服务入口，不内置多租户、RBAC、限流、集中监控告警或前端管理后台；这些能力应由上游业务系统、网关或后续管理后台提供。

`rag:doctor` 是只读排障命令：基于 `rag:inspect` 的统计和可选问题检索结果，输出 `ok`、`warning` 或 `error` 状态、风险 code 和建议动作。它会检查缺失 manifest/source/chunk、无 active JD、历史对话没有 verified 招聘方事实、manifest 计数不一致、embedding provider/model 与当前配置不一致、Qdrant 配置缺失或不可连接，以及带问题时 dense/keyword/hybrid 是否无命中。

`rag:doctor:batch` 是多职位只读巡检命令：读取 JSON 数组、`{ "items": [...] }` 或 JSONL 文件，每个 item 包含 `platform` 和 `jobKey` 或 `keyword`，可选 `question`。它会逐个运行 `rag:doctor`，汇总 `ok/warning/error/failed` 数量、常见 issue code 和去重建议；单个职位失败不会阻断后续职位。`--question` 可作为全局默认问题，item 自己的 `question` 优先；`--fail-on-issue true` 适合 CI 或定时巡检，只要有 warning、error 或执行失败就返回失败码。

`rag:ingest-conversation` 是单段历史对话导入命令；`rag:ingest-conversations` 是批量导入命令，支持 JSON 数组、`{ "items": [...] }` 或 JSONL。批量 item 需要 `platform`、`conversationId`、`jobKey` 或 `keyword`，并可用 `turns[]` 表示整段对话，或用顶层 `id`、`role`、`content` 表示单个 turn。导入是增量、幂等的：同一 `conversationId` 下的新 turn 会按 `id` 和已保存 turn 合并，同 id 新内容覆盖旧内容；随后只重建该 conversation 的 sources、chunks 和 verified 招聘方事实向量。业务系统应尽量提供稳定 turn id；未提供时系统会用 `role`、`createdAt` 和 `content` 生成稳定 id。批量导入可先用 `--dry-run true` 校验；默认遇到首个失败 item 停止，`--fail-on-error false` 可让单行失败不影响后续 item。加 `--doctor true` 时，导入结束后会对本次成功导入影响到的唯一职位运行 `rag:doctor`，结果放在 `doctorSummary`；`--doctor-question` 可提供诊断问题，`--fail-on-doctor-issue true` 会在 doctor 状态非 `ok` 时返回失败码。

`rag:eval` 是批量召回评测命令：读取评测 JSON，逐题检查 hybrid 最终候选片段是否包含预期文本、来源类型、chunkId 或 conversationId；默认有失败 case 时退出码为 1，方便接入回归检查。

`rag:answer-eval` 是批量答案评测命令：真实调用职位 RAG 问答，检查最终回答是否包含预期事实、是否出现禁止内容、是否引用预期来源；默认有失败 case 时退出码为 1。

`rag:export-answer-eval` 会从 `rag/answer-logs.jsonl` 生成 `rag:answer-eval` 用例文件；默认只导出 `feedback.correct=true` 的已审核日志，`--only-feedback false` 可导出未审核草稿，`--include-no-answer true` 可把拒答日志导成 `expectNoAnswer` 用例。默认 `--expected-text-mode answer` 会把完整回答写入 `expectedAnswerIncludes`；`source` 会从当时引用的 JD 或 verified 历史答复中提取与回答匹配的关键事实短语；`hybrid` 优先用 source，提取不到再回退到完整回答。导出的 case 会带 `metadata.draft=true` 和 `metadata.expectedTextNeedsReview=true`，建议人工确认关键事实后再放入回归套件。

`rag:feedback` 会把人工审核结果写回 `rag/answer-logs.jsonl`。推荐用 `--log-id` 精确定位，也支持 `--created-at` 或唯一的 `--question`；同一个选择器匹配多条日志时会拒绝写入，避免误标。错误回答可用 `--error-type wrong_fact|unsupported_claim|missing_context|bad_source|low_relevance|wording_issue|other` 标记错误原因；`--error-type` 只用于 `--correct false`。

`rag:review` 会从 `rag/answer-logs.jsonl` 生成 Markdown 或 JSON 审核报告，默认列出未审核、拒答、低置信、无来源和已标错的回答，并给出可复制的 `rag:feedback` 命令；旧标错日志缺少 `feedback.errorType` 时会计入 `Missing error types` 并给出补标命令；`--include-reviewed true` 可包含已确认正确的日志。

`rag:review:batch` 是多职位审核汇总命令：读取 JSON 数组、`{ "items": [...] }` 或 JSONL 文件，每个 item 包含 `platform` 和 `jobKey` 或 `keyword`。它会逐个生成职位 Review，并汇总总问答数、待审核数、已标错数、缺失错误类型数、拒答数、低置信数和无来源数；`--fail-on-issue true` 适合定时任务或 CI。

`rag:metrics` 是多职位 RAG 质量指标导出命令，复用 `rag:review:batch` 的职位列表格式，只读 `answer-logs.jsonl`。它输出整体、按平台、按职位、按日期的总问答数、审核率、人工正确率、错误率、拒答率、低置信率、无来源率、平均 confidence 和错误类型分布；错误类型来自 `feedback.errorType`，旧标错日志缺失类型时归入 `unspecified`。支持 `--since` / `--until` 做时间窗口统计，也支持 `--policy ./rag-metrics-policy.json` 读取质量阈值。policy 可包含 `minReviewRate`、`minCorrectRate`、`maxIncorrectRate`、`maxNoAnswerRate`、`maxLowConfidenceRate`、`maxMissingSourcesRate` 和 `maxErrorTypeRates`，CLI 同名 kebab-case 参数优先级更高；输出会包含 `thresholds`、`thresholdViolations` 和 `recommendations`，配合 `--fail-on-threshold true` 可接入 CI 或定时任务。

`rag:ops` 是日常运营汇总命令，会按同一份职位列表依次运行批量 Doctor、批量 Review 和 Metrics，并生成一份 Markdown 或 JSON 报告。报告包含整体 `status`、统一建议、Doctor 索引/环境健康摘要、Review 待审核明细和 Metrics 质量阈值结果；`--policy`、`--since`、`--until`、`--low-confidence-threshold`、`--include-reviewed`、`--reviewer` 等参数会透传到对应环节。`--fail-on-issue true` 会在 `status` 不是 `ok` 时返回失败码，适合每天定时巡检或 CI。

`rag:regression` 是一键回归命令：读取 suite JSON，按职位顺序运行召回评测和答案评测，并汇总失败 case；默认有任何失败时退出码为 1。

仓库内提供了第一批 RAG 回归基线：`fixtures/rag/regression.json`。它当前覆盖 51job `优衣库` 店长 JD 的薪资、地点、学历、经验、语言、职责、排班、已确认历史对话里的住宿补贴、未确认历史对话不能作为事实的负例，以及 JD 未说明问题。新环境可直接执行：

```bash
npm run rag:baseline
```

`rag:baseline` 会自动 seed fixture JD 和对话 fixture、构建/刷新 RAG 索引、运行回归 suite，并在任一评测失败时返回非 0 退出码。它会预检 embedding 和 Qdrant 配置；默认使用本地 embedding HTTP 服务 `RAG_EMBEDDING_PROVIDER=local-http`，默认地址是 `http://127.0.0.1:8011`，向量库需要 `QDRANT_URL`，并会在写入 fixture 前检查 Qdrant `/collections` 可访问。如果显式切回 OpenAI embedding，则需要 `OPENAI_API_KEY`。底层的 `rag:seed-fixtures` 仍可单独运行，它会把 `fixtures/rag/jobs/` 下的脱敏职位 JD 写入本地 `data/`，并读取 `fixtures/rag/conversations/<platform>/<jobKey>/<conversationId>.json` 下的脱敏对话；默认跳过已有职位，需要覆盖时显式传 `--overwrite true`。新增基线时，把脱敏 JD 放入 `fixtures/rag/jobs/<platform>/<jobKey>/jd.json`，需要历史对话时把对话放入 `fixtures/rag/conversations/<platform>/<jobKey>/<conversationId>.json` 并只给已确认的招聘方事实标记 `verified: true`，把召回用例写入 `*.retrieval.json`，把最终回答用例写入 `*.answer.json`，再加入 suite。每个问题都应来自候选人真实会问的内容，并同时包含一个 JD/历史对话未说明或未确认的负例，防止回答编造。

本地 embedding 服务可以用 `npm run rag:embedding:local` 从仓库启动；本机长期运行时推荐使用 LaunchAgent `com.autorecruit.embedding`，运行目录在 `~/.local/share/autorecruit/embedding/`，日志在 `~/.local/var/log/autorecruit/local-embedding.log` 和 `~/.local/var/log/autorecruit/local-embedding.err.log`。该运行目录独立于仓库目录，是为了避免 macOS 后台服务访问 `~/Documents` 时遇到权限限制。日常检查可运行 `rtk launchctl list | rtk rg 'com\\.autorecruit\\.(qdrant|embedding)'`、`rtk curl -sSf http://127.0.0.1:6333/collections` 和 `rtk curl -sSf http://127.0.0.1:8011/health`。

本地快速冒烟或未来 CI 环境可以运行 `npm run test:rag:offline`。它底层执行 `rag:baseline:offline -- --summary-only true`，默认写入临时数据目录，使用内存向量库和确定性假 embedding，只跑召回回归，不调用 OpenAI、不连接 Qdrant、不做答案生成评测。默认只打印紧凑摘要；需要完整 JSON 时运行 `npm run rag:baseline:offline -- --output-file tmp/rag-baseline-offline.json`，需要保留离线运行产物时再加 `--data-dir /path/to/debug-data`。当前仓库未提交 GitHub Actions workflow；提交前建议本地手动执行 `npm run typecheck`、`npm run test:scoring`、`npm run test:rag:offline`、`npm run build` 和 `git diff --check`。

RAG 日常运营流程见 `docs/rag运营手册.md`。仓库还提供了可复制模板：`fixtures/rag/rag-review-jobs.example.json`、`fixtures/rag/rag-metrics-policy.example.json` 和 `fixtures/rag/conversation-import.example.jsonl`。

默认 `--search-source saved` 会使用平台已保存的订阅/快捷搜索入口。需要临时主动输入筛选条件、不依赖保存标签时，使用 direct 搜索来源：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>" --jd-file ./fixtures/jd.txt --search-source direct --application-filter-input-file ./filter-input.json
```

direct 模式会打开平台搜索页、清理旧筛选、输入本次关键词、应用 `data/<platform>/filter-catalog/application-filter-options.latest.json` 支持的应用层筛选条件，然后继续同一套候选抽取、详情打开、评分、报告和邮件流程。`--application-filter-input-file` 只在 `--search-source direct` 下有效；筛选条件出现 `skipped` 或 `failed` 时会中止本次抓取，避免从半应用条件里抓候选人。

默认情况下，主流程会按平台默认状态设置已查看筛选。需要切换到包含已查看候选人的复跑时，显式打开 `--include-viewed`：

```bash
npm run dev -- --platform <platform> --keyword "<关键词>" --include-viewed true
```

平台行为：

- `51job`：默认勾选搜索页的 `我已看`；`--include-viewed true` 时取消它。
- `liepin`：默认勾选搜索页的 `隐藏已查看`；`--include-viewed true` 时取消它。
- `zhilian`：默认勾选可见的 `未看过`；`--include-viewed true` 时只取消可见的 `未看过`，不会改动 `未聊过`。设置后会确认快捷搜索条件仍显示 `关键词：<关键词>`，必要时重新选择快捷搜索标签再提取候选人。

Liepin 可在新增候选人进入详情页后自动转发给常用联系人。显式传入联系人姓名才会启用；转发成功后才继续解析和标记 seen，转发失败会让该候选人保持可重试：

```bash
npm run dev -- --platform liepin --keyword "<关键词>" --liepin-forward-contact "<常用联系人姓名>"
```

`--platform all` 的单职位运行也可以传入 `--include-viewed true` 和 `--liepin-forward-contact "<常用联系人姓名>"`。`--include-viewed true` 会应用到三个平台；`--liepin-forward-contact` 只在 Liepin 阶段生效。

direct 单职位模式同样支持这些开关。Zhilian 仍会在新增候选人详情中复制 `转给同事` -> `链接转发` 分享链接；Liepin 仍只对新增候选人执行常用联系人转发。

### Boss 自动沟通模式

Boss 自动沟通只允许 `--platform boss`，且必须显式配置转发方式和收件目标。物业电工严格模式示例：

```bash
PLAYWRIGHT_HEADLESS=false npm run dev -- \
  --platform boss \
  --boss-auto-chat true \
  --boss-chat-require-all true \
  --boss-forward-mode email \
  --boss-forward-recipient "resume@example.com" \
  --boss-chat-summary-email "summary@example.com" \
  --boss-chat-summary-cc "audit@example.com"
```

参数说明：

- `--boss-auto-chat true`：进入独立未读会话审查模式。
- `--boss-chat-require-all true`：使用当前已配置岗位的全部硬性条件；物业电工会走五项明确证据判断。
- `--boss-chat-score-threshold 0..100`：不启用严格模式时使用的模型评分阈值，默认 70。
- `--boss-forward-mode colleague|email`：符合者的转发方式。
- `--boss-forward-recipient`：站内同事姓名或简历收件邮箱。
- `--boss-chat-summary-email`：整批完成后的总结邮件收件人。
- `--boss-chat-summary-cc`：可选逗号分隔抄送列表，必须与总结收件人一起使用。

该模式不能和 `--keyword`、`--jobs-file`、`--jd`、`--jd-file`、普通 `--email`/`--cc`、`--include-viewed`、搜索来源、搜索订阅或 JD/RAG 问答参数组合。运行前应先确认岗位 JD 已保存在 Boss 平台目录；例如 `data/boss/jobs/物业电工/jd.json`。

总结邮件列出符合与不符合候选人的姓名、ID、理由和动作状态。若符合者已成功转发但后续消息或换电话失败，会明确显示“已转发，但联系动作未完成”。SMTP 发送失败不会删除已落盘的审查结果，但命令会失败，避免误报邮件已送达。

### 全平台模式

`--platform all` 会按固定顺序串行运行：

1. `51job`
2. `liepin`
3. `zhilian`

每个平台使用独立浏览器会话和平台隔离存储。任一平台失败时，整体运行停止并抛出该平台错误。

示例：

```bash
npm run dev -- --platform all --keyword "<关键词>" --jd "<JD 文本>"
```

全平台普通简历抓取支持 `--include-viewed true` 和 `--liepin-forward-contact "<常用联系人姓名>"`；转发开关只在 Liepin 阶段使用。

全平台也支持 direct 搜索来源：

```bash
npm run dev -- --platform all --keyword "<关键词>" --jd-file ./fixtures/jd.txt --search-source direct --application-filter-input-file ./filter-input.json
```

此时三个平台仍按 `51job -> liepin -> zhilian` 串行运行。每个平台会用自己的 application options catalog 解释同一个筛选输入文件；输入文件必须只包含目标平台支持的字段。任一平台筛选应用、候选抽取、转发、评分、导出或邮件失败时，`--platform all` 立即停止并抛出该平台错误。

### 批量模式

`--jobs-file` 接收 JSON 数组，每个条目至少包含 `keyword`，并可选包含 `jd`、`jdFile`、`email`、`cc`、`searchSource`、`applicationFilterInputFile`。

```json
[
  {
    "keyword": "东南亚 销售",
    "jdFile": "./fixtures/jd.txt",
    "email": "user@example.com",
    "cc": ["manager@example.com"],
    "searchSource": "direct",
    "applicationFilterInputFile": "./filter-input.json"
  }
]
```

运行：

```bash
npm run dev -- --platform <platform|all> --jobs-file ./jobs.json
```

批量模式不能与单职位参数 `--keyword`、`--jd`、`--jd-file` 混用。与 `--platform all` 组合时，外层顺序是 jobs 文件顺序，内层顺序是 `51job -> liepin -> zhilian`。

批量模式同样支持在 CLI 上传入 `--include-viewed true`，该开关会应用到本次 jobs 文件内的所有职位和平台。批量普通简历抓取也支持 `--liepin-forward-contact "<常用联系人姓名>"`，但只允许与 `--platform liepin` 或 `--platform all` 组合；`--platform all --jobs-file` 时只在每个职位的 Liepin 阶段转发，51job 和 Zhilian 不使用该开关。

批量模式可在 CLI 上传入 `--search-source direct --application-filter-input-file ./filter-input.json` 作为所有职位的默认 direct 筛选输入；jobs 文件内的 `searchSource` 和 `applicationFilterInputFile` 会覆盖 CLI 默认值。jobs 文件里的相对 `applicationFilterInputFile` 路径按 jobs 文件所在目录解析。

## 运行要求

### 必需条件

- Node.js 24 LTS 或 Node.js 26。仓库 `.nvmrc` 固定为 `24`，`package.json` 声明 `>=24 <27`。
- 已安装依赖：`npm install`
- 可用模型 API 凭据：`OPENAI_API_KEY`
- 可复用的平台登录态：通过 `npm run login:session -- --platform <platform>` 生成或刷新。
- 使用内部控制台前，先运行 `npm run web:build` 生成 `frontend/dist`，再启动 `npm run api`；开发前端时可改用 `npm run web:dev`。

### 可选配置

- `DATA_DIR`：调整本地数据目录，默认是 `data/`。
- `OPENAI_BASE_URL`：使用 OpenAI 兼容模型服务。
- `OPENAI_MODEL`：JD 解析、评分和 RAG 问答的默认模型。
- `JD_PARSING_MODEL`：覆盖 JD 解析模型。
- `SCORING_MODEL`：覆盖评分模型。
- `RAG_MODEL`：可选覆盖 JD/RAG 问答模型；未设置时使用 `OPENAI_MODEL`。当前本机配置通常只需要设置 `OPENAI_MODEL`。
- `RAG_EMBEDDING_PROVIDER`：RAG embedding provider，默认 `local-http`，使用本地 embedding HTTP 服务；也可显式设为 `openai`。
- `RAG_EMBEDDING_LOCAL_URL`：`local-http` provider 的服务地址，默认 `http://127.0.0.1:8011`，服务需提供兼容 `/embeddings` 的 HTTP 接口。
- `RAG_EMBEDDING_LOCAL_API_KEY`：`local-http` provider 的可选 bearer token。
- `RAG_EMBEDDING_MODEL`：覆盖 RAG embedding 模型，默认 `BAAI/bge-small-zh-v1.5`。
- 本地 embedding 服务可用 `npm run rag:embedding:local` 启动；本机常驻运行使用 LaunchAgent `com.autorecruit.embedding`，日志在 `~/.local/var/log/autorecruit/`。
- `RAG_TOP_K`：覆盖 RAG 检索返回片段数，默认 `8`。
- `RAG_RETRIEVAL_MODE`：RAG 检索模式，默认 `hybrid`；设为 `dense` 可临时只使用 Qdrant 向量检索。
- `RAG_DENSE_TOP_K`：覆盖 hybrid 模式下 Qdrant dense 召回候选数，默认是 `RAG_TOP_K * 4`。
- `RAG_KEYWORD_TOP_K`：覆盖 hybrid 模式下本地 BM25 关键词召回候选数，默认是 `RAG_TOP_K * 4`。
- `RAG_RERANK_CANDIDATE_K`：覆盖融合后进入轻量重新排序的候选数，默认是 `RAG_TOP_K * 3`。
- `RAG_MIN_CONFIDENCE_SCORE`：最低可信召回分，默认 `0.08`；低于该值时返回未说明。hybrid 模式下这是排序分，不是概率。
- `QDRANT_URL`：Qdrant 服务地址；已保存职位的向量 RAG 需要配置。
- `QDRANT_API_KEY`：Qdrant API key，可选。
- `RAG_VECTOR_COLLECTION`：Qdrant collection 名称，默认 `autorecruit_rag_chunks`。
- `AUTORECRUIT_CONSOLE_HOST` / `AUTORECRUIT_CONSOLE_PORT`：内部控制台 API 监听地址和端口，默认 `127.0.0.1:4180`。
- `AUTORECRUIT_CONSOLE_API_KEY`：内部控制台 API 的可选 bearer token；它是本地内部入口的轻量保护，不是完整认证网关。
- `AUTORECRUIT_CONSOLE_MAX_BODY_BYTES`：内部控制台 API 请求体上限，默认 2 MiB。
- `AUTORECRUIT_CONSOLE_FRONTEND_DIR`：`npm run api` 提供静态前端时读取的目录，默认 `frontend/dist`。
- `VITE_API_BASE_URL`：前端请求的 API 基址，默认 `/api`。
- `.venv/bin/python` + `crawl4ai`：启用 Crawl4AI 增强提取；不可用时会自动回退到内置解析。
- SMTP 环境变量：`SMTP_HOST`、`SMTP_PORT`、`SMTP_USER`、`SMTP_PASS`、`SMTP_FROM`。

### 浏览器配置

- `BROWSER_ENGINE=cloakbrowser|playwright`：浏览器启动引擎，默认 `cloakbrowser`。如需回退到 Playwright 自带 Chromium，设置为 `playwright`。
- `PLAYWRIGHT_HEADLESS=true`：启用 headless。默认是 headed；Liepin 始终强制使用有头浏览器，此变量对 Liepin 不生效。
- `PLAYWRIGHT_SEARCH_PAGE_TIMEOUT_MS`：搜索入口到候选列表 ready 的总等待预算，默认 `20000`。
- `PLAYWRIGHT_EMPTY_RESULTS_STABLE_MS`：51job 空列表稳定窗口，默认 `2000`。
- `PLAYWRIGHT_API_FALLBACK_TIMEOUT_MS`：Liepin/Zhilian API fallback 短等待上限，默认 `3000`。
- `PLAYWRIGHT_RESUME_DETAIL_TIMEOUT_MS`：简历详情打开/就绪总等待预算，默认 `20000`。
- `PLAYWRIGHT_ACTION_DELAY_MIN_MS` / `PLAYWRIGHT_ACTION_DELAY_MAX_MS`：全平台点击/输入动作前的随机等待默认值。51job/Zhilian/Boss 默认 `0-0`，Liepin 默认 `2000-3000`。
- `PLAYWRIGHT_CANDIDATE_DELAY_MIN_MS` / `PLAYWRIGHT_CANDIDATE_DELAY_MAX_MS`：全平台候选人之间的随机等待默认值。51job/Zhilian/Boss 默认 `0-0`，Liepin 默认 `2000-3000`。
- `PLAYWRIGHT_<PLATFORM>_ACTION_DELAY_MIN_MS` / `PLAYWRIGHT_<PLATFORM>_ACTION_DELAY_MAX_MS`：覆盖单个平台动作等待；`<PLATFORM>` 可为 `51JOB`、`LIEPIN`、`ZHILIAN`、`BOSS`。
- `PLAYWRIGHT_<PLATFORM>_CANDIDATE_DELAY_MIN_MS` / `PLAYWRIGHT_<PLATFORM>_CANDIDATE_DELAY_MAX_MS`：覆盖单个平台候选人间隔；平台枚举同上。
- `PLAYWRIGHT_REUSE_BROWSER=true|false`：全平台可复用浏览器默认开关。默认不启用；51job、Liepin、Zhilian 和 Boss 有平台默认值，会默认启用。
- `PLAYWRIGHT_51JOB_REUSE_BROWSER=false` / `PLAYWRIGHT_ZHILIAN_REUSE_BROWSER=false` / `PLAYWRIGHT_BOSS_REUSE_BROWSER=false`：关闭对应平台可复用浏览器。
- `PLAYWRIGHT_LIEPIN_REUSE_BROWSER=false`：关闭 Liepin 可复用浏览器；默认开启。开启时流程结束会关闭最后的详情页、停留在人才搜索页并保留浏览器，下次 Liepin 运行会优先连接这个窗口。
- `PLAYWRIGHT_51JOB_REUSE_CDP_PORT` / `PLAYWRIGHT_LIEPIN_REUSE_CDP_PORT` / `PLAYWRIGHT_ZHILIAN_REUSE_CDP_PORT` / `PLAYWRIGHT_BOSS_REUSE_CDP_PORT`：各平台可复用浏览器的本地 CDP 端口，默认分别是 `19325`、`19327`、`19329`、`19331`。

## 数据模型与落盘结构

核心类型定义在 `src/types/job.ts`：

- `NormalizedJob`：结构化 JD。
- `JobRecord`：职位持久化记录，包含 `jobKey`、`platform`、`searchKeyword`、JD 原文、标准化 JD 和邮件收件人。
- `CandidateListItem`：搜索结果页候选人卡片。
- `CandidateResume`：结构化简历，包含基础信息、工作经历、项目经历、教育经历、技能、证书和可选 `candidateShareUrl`。
- `CandidateScoreArtifact`：评分产物，可能是成功评分，也可能是失败记录。
- `RunResult`：一次执行的轻量摘要。
- `BossChatReviewRun`：一次 Boss 未读会话审查的候选人判断、动作状态和错误摘要。
- `JobResultsMarkdownExport`：导出的 Markdown 结果视图。

默认数据目录是 `data/`，每个职位位于：

```text
data/<platform>/jobs/<jobKey>/
```

目录内容：

- `jd.json`：职位记录，也是 rerun 复用 JD 载荷和邮件收件人的来源。
- `seen-ids.json`：已成功处理的候选人 ID。
- `resumes/<candidateId>.json`：结构化简历。
- `snapshots/<candidateId>.txt`：原始页面文本快照。
- `snapshots-dom/<candidateId>.json`：DOM 抽取快照。
- `scores/<candidateId>.json`：评分产物。
- `results/<timestamp>.json`：轻量运行摘要。
- `exports/latest.md`：最新导出的 Markdown 报告。
- `rag/sources.jsonl`：RAG 事实源记录，包含 JD、历史对话等来源。
- `rag/chunks.jsonl`：RAG chunk 事实源；Qdrant 只是可重建索引。
- `rag/embeddings.jsonl`：RAG embedding 缓存，按 provider、model 和内容 hash 复用向量。
- `rag/conversations/<conversationId>.jsonl`：导入的职位历史对话，按 turn id 去重合并。
- `rag/index-manifest.json`：最近一次 RAG 索引构建摘要。
- `rag/answer-logs.jsonl`：已保存职位的 RAG 问答审计日志，记录 `logId`、问题、答案、confidence、拒答原因、引用 sources 和人工反馈；错误反馈可包含 `errorType`；临时 JD 问答和离线评测不写入。
- `exports/resumes/<candidateId>-<name>.docx`：按 Word 模板离线导出的候选人 DOCX 简历。

Boss 自动沟通另有独立运行状态：

- `data/boss/chat-review/reviewed-conversation-ids.json`：已完成处理或已成功转发的会话 ID，防止重复动作。
- `data/boss/chat-review/runs/<timestamp>.json`：每次审查的候选人 ID、岗位、硬性条件证据、匹配结论、转发/聊天/换电话状态和错误。最新状态仍为 `failed` 且尚未转发的会话可作为重试来源。

历史平台化改造前的数据可通过 `npm run migrate:platform-storage` 从 `data/jobs/` 迁移到 `data/51job/jobs/`，并回填缺失的 `platform` 字段。迁移脚本会拒绝覆盖已存在的目标目录。

## 邮件与失败语义

收件人和抄送列表按职位维度持久化：

- 首次运行传入 `--email` 或 `--cc` 时，会写入对应 `jd.json`。
- 后续同一 `jobKey` 重跑时，如果未显式传入，会复用已保存配置。
- 再次显式传入时，以本次参数为准并覆盖保存值。

失败处理：

- 评分失败：保存失败评分产物，候选人仍保持已处理。
- 上游详情打开或提取失败：不写入 seen，下次继续重试。
- 导出失败：不让整次运行失败，错误记录到 summary。
- 邮件失败：不让整次运行失败，错误记录到 summary。
- 空结果运行：属于成功运行，可发送“无新增候选人”通知邮件。
- Zhilian 有新增候选人的报告邮件要求当前运行评分产物都带唯一 `candidateShareUrl`；缺失或重复会导致邮件发送失败并记录到 summary。
- Zhilian 搜索页卡片可能不暴露可用详情链接，候选人 ID 提取会优先读取卡片 Vue props 中的 `userMasterId`、`resumeNumber` 等字段，再使用 API fallback。
- Boss 简历打开或解析失败且尚未转发：记录失败并保留重试来源；即使红点已消失，也可从历史运行恢复当前列表中仍可见的会话。
- Boss 已转发但联系失败：保留 `forwarded: true`，不自动重复转发；总结邮件明确列出联系未完成原因。
- Boss 总结邮件失败：审查运行结果已经落盘，但命令返回失败，避免把未送达误报为成功。

## 维护建议

- 修改 `src/browser/resume-detail.ts` 这类启发式模块时，优先使用已保存快照离线重解析验证，再考虑 live 流程。
- 修改平台搜索入口、列表提取或详情解析时，保留多重 DOM fallback，避免假设页面结构稳定。
- 修改全平台或批量语义时，同时验证单平台、`--platform all` 和 `--jobs-file` 的顺序与错误传播。
- 修改报告或邮件逻辑时，覆盖有新增候选人、无新增候选人、评分失败和 Zhilian 分享链接校验路径。
- 修改 DOCX 简历导出时，保留模板正文、页眉和头像关系的兼容性；头像只能来自候选人本人详情页证据，无法确认时应省略头像而不是使用默认头像、相似人才头像或模板示例头像。
