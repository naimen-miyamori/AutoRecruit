# 项目说明文档

## 项目概述

AutoRecruit 是一个 TypeScript CLI，用于自动化招聘筛选流程。当前生产可用的浏览器抓取主链路已覆盖 51job、Liepin 和 Zhilian。平台适配器边界仍然保留，这三个平台当前都属于正式 CLI 支持范围。

Liepin 和 Zhilian 都已经接入完整 CLI 主流程，当前按正式支持平台维护；相关平台专项回归仍通过独立命令执行。

当前实现已经覆盖从 JD 结构化、候选人抓取、简历解析、去重、评分，到 Markdown 报告导出与邮件发送的完整主链路。

当前运行方式支持两类入口：单次执行处理一个职位，或通过 `--jobs-file` 批量顺序处理多个职位。正式支持的平台路径包括 51job、Liepin 和 Zhilian。共享编排和平台适配器边界已经保留，这三个平台都通过同一套主流程接入：

1. 用户输入搜索关键词；首次运行某个新 `jobKey` 时，还需要提供职位描述来源（内联 `--jd` 或文件 `--jd-file`）。
2. 系统会先根据搜索关键词生成 `jobKey`；若命中已有职位记录，则直接复用 `jd.json` 中已保存的 JD 原文与标准化结果，并跳过 JD 文件读取与 JD 解析；若未命中，则把本次 JD 解析为标准化 JSON。
3. 使用已保存且通过校验的目标平台登录态打开对应的搜索入口页面。
4. 进入对应关键词的人才搜索页并提取候选人 ID；如果明确空结果文案出现，例如 51job 筛选条件导致的“没有搜索到相关的人才”，或 51job 的空列表容器在稳定窗口内保持无候选卡，则视为一次成功的 0 候选人运行，而不是异常失败。
5. 按 `jobKey` 读取历史 `seen-ids.json` 去重，只处理新增候选人。
6. 打开新增候选人的简历详情页，保存原始快照与结构化简历。
7. 对成功保存简历的候选人先写入 `seen-ids.json`；详情打开或提取失败的候选人保留重试机会。
8. 从本地已保存简历批量执行 JD 匹配评分并保存评分产物；模型评分失败会写入失败评分产物，不回滚去重状态。
9. 生成最新一次运行的结果汇总，并导出 Markdown 报告。
10. 如存在收件人邮箱，则发送本次运行报告邮件；Zhilian 有新增候选人的报告邮件会优先使用从“转给同事”里复制出来的简历分享链接。

主入口也支持 `--platform all`，用于按固定顺序依次执行 51job、Liepin、Zhilian。该模式不会并发抓取，也不会跨平台共享浏览器会话；它会对每个平台重复执行同一套单平台主流程，并继续使用 `data/<platform>/jobs/<jobKey>/` 的平台隔离存储。若某个平台执行失败，整体运行会在该平台停止并抛出原始错误。

批量入口通过 `--jobs-file` 启用。jobs 文件是 JSON 数组，每个条目包含 `keyword`，并可选填 `jd`、`jdFile`、`email`、`cc`。批量模式按文件顺序执行；如果同时使用 `--platform all`，则外层顺序是 jobs 文件，内层顺序是 `51job -> liepin -> zhilian`。`--jobs-file` 不能与单职位参数 `--keyword`、`--jd`、`--jd-file` 混用。

## 核心能力

### 1. JD 标准化

入口会把用户提供的自由文本 JD 转换为结构化 `NormalizedJob`，用于后续评分。当前解析逻辑已经切换为**模型驱动的结构化提取**：系统会调用 OpenAI 格式的模型接口，要求其直接返回一个符合约束的 JSON 对象文本，再经过本地 schema 校验与清洗后落盘。`JD_PARSING_MODEL` 会覆盖 `OPENAI_MODEL`。当前重点抽取：

- 职位标题
- 专业要求
- 语言要求
- 工作内容 / 职责
- 硬性要求 / 优先条件
- 地域偏好
- 行业标签
- 经验、年龄、薪资等可识别字段

同一职位的持久化目录由 `platform + jobKey` 共同标识。当前 `jobKey` 优先由搜索关键词构造，关键词为空时再退回职位标题。首次运行新 `jobKey` 时，系统会读取 `--jd` 或 `--jd-file` 并调用 JD 解析链路生成 `NormalizedJob`；当同一平台下的同一关键词命中已有 `jobKey` 时，系统会直接复用 `data/<platform>/jobs/<jobKey>/jd.json` 中已保存的 `rawText` 与 `normalizedJob`，并完全跳过 JD 文件读取与 JD 大模型解析。即使 rerun 时再次传入 `--jd` 或 `--jd-file`，命中已有 `jobKey` 后也不会覆盖已保存的 JD 载荷。

在 `--platform all` 模式下，同一份 `--jd` 或 `--jd-file` 会作为本次所有缺失职位记录的平台的初始化 JD 来源；已经存在 `jd.json` 的平台仍然复用该平台自己的历史 JD 载荷。若不提供 JD 来源，则只有所有平台都已经存在对应 `jobKey` 时才能完整 rerun，否则会在第一个缺失职位记录的平台按常规缺失 JD 错误失败。

在 `--jobs-file` 批量模式下，每个条目独立解析自己的 JD 来源。条目里的 `jd` 与 `jdFile` 互斥；如果某个平台下对应 `jobKey` 已经存在，仍然直接复用该平台保存的 `jd.json`，可以不在条目中提供 JD 来源。条目级 `email` 和 `cc` 会覆盖命令行级默认收件人配置。

### 2. 浏览器自动化抓取

项目通过 Playwright 复用本地登录态访问当前已接通的平台页面。当前正式支持的平台包括 51job、Liepin 和 Zhilian，其中 51job 的入口方式如下：

- 打开订阅页 `https://ehire.51job.com/Revision/talent/subscribe`
- 悬浮对应订阅关键词
- 定位“搜索”入口，并由点击 helper 统一完成可见性与加载遮罩检查后立即点击
- 进入人才搜索页后从候选人卡片中提取 `candidateId`
- 如果页面出现明确空结果文案，或结果列表已可见但没有任何当前候选人卡片并满足稳定窗口，则按 0 候选人成功处理，并继续后续运行结果写入、报告导出与邮件分支
- 打开候选人详情页并抓取简历原始文本与 DOM 信息

各平台都使用各自默认的 storage-state 文件保存登录态。51job 默认使用 `storage-state.json`，Liepin 默认使用 `storage-state.liepin.json`，Zhilian 默认使用 `storage-state.zhilian.json`。系统启动时会复用保存态，并通过平台已鉴权入口校验登录态是否仍然有效；因此失败场景不仅包括文件不存在，也包括文件已存在但登录态过期或失效。`npm run login:session -- --platform 51job` 会打开一个不复用旧保存态的新浏览器会话，等待人工完成登录、保存 51job 的 storage state，然后再启动一个新的浏览器会话重新加载该保存态并验证可复用性，只有 fresh-session 验证通过后才会报告成功。正常多平台运行时应保持 `STORAGE_STATE_PATH` 未设置，让代码按平台自动选择默认文件；如果为了单次调用显式设置 `STORAGE_STATE_PATH`，它也必须指向当前平台专用的文件名，像共享 `storage-state.json` 这类会导致平台串用的路径会被代码直接拒绝。Zhilian 的登录入口是 `https://passport.zhaopin.com/org/login`，已鉴权搜索页入口是 `https://rd6.zhaopin.com/app/search`；相关逻辑当前属于正式 CLI 主流程支持。

`--platform all` 会按平台顺序分别创建和关闭浏览器会话，每个平台都使用自己的默认 storage state 文件。该模式下更应保持 `STORAGE_STATE_PATH` 未设置，避免把一个平台的登录态强行套用到其他平台。

运行时方面，仓库当前推荐使用 Node 24 LTS。根目录 `.nvmrc` 固定为 `24`，`package.json` 声明了 `engines.node` 为 `>=24 <27`。Node 26 也属于支持范围。源码运行脚本已经从 `tsx` 切换为 `node --import ./scripts/node-ts-hooks.mjs`，通过 `module.registerHooks()` 和 TypeScript 单文件转译执行 `.ts` 源码，避免了旧版 `tsx -> module.register()` 路径触发的 `DEP0205` 警告。

Zhilian 的搜索进入方式与 51job、Liepin 不同：

- 必须在 `https://rd6.zhaopin.com/app/search` 页面点击一个已保存的快捷搜索标签
- 标签文本必须包含这次运行传入的原始 `--keyword`
- 不能把 `https://rd6.zhaopin.com/desktop` 当作搜索页
- 点击候选人后，简历详情会以同页悬浮层的形式覆盖在搜索页之上，而不是作为新的标签页或独立详情页打开
- 解析简历详情时，会点击“转给同事”，选择“链接转发”，复制生成的安全分享链接，并保存为候选人的 `candidateShareUrl`

三个平台的搜索等待机制使用同一套外部 contract：主流程在打开平台搜索入口前创建一个总 deadline，并把它同时传给 `openSubscribeSearch()` 和 `extractCandidateList()`。deadline 耗尽后不会再叠加平台内部固定长等待；错误会带上页面诊断信息。

- 51job 的搜索结果等待 DOM 候选人卡片、明确空结果文案，或 `.virtual_list` 在稳定窗口内保持空列表；其中“没有搜索到相关的人才”这类由筛选条件造成的空结果也属于合法 0 候选状态。简历详情打开使用一个总详情页 deadline，并并行观察弹窗、新页面跳转和当前页内容。
- Liepin 的搜索页打开、搜索壳页、快捷搜索标签、DOM 候选和 `search-resumes` API fallback 都使用同一个搜索 deadline。进入快捷搜索结果后会先确保“隐藏已查看”筛选已勾选，再进入候选人 ID 提取；如果该筛选只在结果区出现，会先点击结果页“搜索”按钮让筛选项渲染出来。列表提取 DOM-first；DOM 候选完整时不等待 API，DOM 候选缺安全详情 URL 时只短等 API 补 URL。勾选“隐藏已查看”前的 `search-resumes` 响应会被缓存屏障丢弃，避免延迟返回的旧搜索结果被后续 ID 提取复用。
- Zhilian 的列表提取也是 DOM-first：如果 DOM 已经提取到候选人，就不会继续等待候选人 API；DOM 无候选时才在剩余 deadline 内等待 API fallback 或明确空结果。简历详情是同页 modal，解析前只做较短的就绪确认。

### 3. 增量去重与本地持久化

系统按 `jobKey` 保存职位维度的数据，避免重复处理历史候选人。候选人 ID 会写入 `seen-ids.json`，后续运行只处理新出现的 ID。

对于成功进入详情并保存简历快照/结构化简历的候选人，系统会先写入 `seen-ids.json`；对于上游页面打开/提取失败的候选人，会保留重试机会。评分失败不会阻止候选人被标记为已处理。

### 4. 简历解析与快照保存

每个新增候选人都会保存多份本地数据：

- 原始页面全文快照
- DOM 抽取快照
- 结构化简历 JSON

这使得后续可以离线重解析，而不必重新访问网站。

### 5. 评分、导出与邮件发送

系统会先保存所有成功抓取的新增简历，再从本地简历 JSON 执行 JD 匹配评分，并保存评分结果。评分产物既包括成功结果，也包括模型调用或校验失败时写入的失败产物。运行结束后：

- 导出本次最新运行对应的 Markdown 报告
- 如果配置了邮箱，则发送邮件

导出的 Markdown 报告会在头部元信息中标注本次结果的 `平台来源`。如果本次没有新增候选人，则对应的通知邮件正文也会包含相同的平台来源标识，便于区分不同平台运行结果。

Zhilian 的有候选人报告邮件会使用复制出来的同事转发分享链接展示候选人，而不是直接展示内部 `candidateId`。发送 Zhilian 报告前会校验本次运行的评分产物都有 `candidateShareUrl`，并且链接不能重复；缺失或重复会作为邮件发送错误记录。无新增候选人的 Zhilian 通知邮件不需要分享链接。

邮件发送支持两种情况：

1. **本次有新增候选人**：发送本次运行对应评分产物聚合出的结果邮件。
2. **本次无新增候选人**：发送“本次无新增候选人”的通知邮件；这也包括结果页正常加载但当前可见候选人为 0，或 51job 因筛选条件显示“没有搜索到相关的人才”的情况。

收件人邮箱按职位维度持久化：首次运行若通过 `--email` 传入，会写入对应 `jobKey` 的 `jd.json`；后续相同职位重跑时，即使不再显式传 `--email`，仍会复用已保存的收件人。抄送邮箱也支持通过 `--cc` 传入并按职位维度持久化；后续重跑时若未显式覆盖，会复用已保存的抄送列表。

## 项目整体架构理解

这个项目不是通用 ATS，也不是 Web 平台，而是围绕招聘网站搜索工作流定制的本地 CLI 工具。当前生产实现已覆盖 51job、Liepin 和 Zhilian，并通过平台适配器维持多平台抓取与筛选能力。它的目标不是做通用招聘系统，而是把一条可重复执行、可离线复盘的招聘筛选流水线稳定下来。

从职责划分上，项目可以分成 7 个层次：

1. **入口编排层**：负责解析参数、组织调用顺序、收口结果。
2. **浏览器交互层**：负责登录态复用、平台登录验证、订阅页/搜索页跳转、候选人列表抓取、详情页打开。
3. **平台适配层**：负责平台特有的登录入口、已鉴权首页、搜索页进入、候选人列表提取、简历详情解析。
4. **提取解析层**：负责从列表页和详情页提取结构化数据。
5. **评分层**：负责把结构化简历与结构化 JD 做匹配评分。
6. **持久化层**：负责本地 JSON、文本快照、DOM 快照、导出文件的写入和读取。
7. **报告分发层**：负责 Markdown 汇总导出与 SMTP 邮件发送。

它的关键特点有：

- **CLI-first**：主入口是命令行，不依赖前端界面。
- **本地优先**：数据保存在本地目录和 JSON 文件中。
- **模型化 JD 解析**：职位描述通过模型直接抽取为结构化 JSON，而不是依赖本地规则解析。
- **启发式简历解析**：简历与卡片解析仍依赖较多中文规则和 DOM 回退策略。
- **离线复盘友好**：原始文本和 DOM 快照会落盘，便于后续重解析和调试。
- **主链路闭环**：抓取、解析、评分、导出、邮件都已打通。

## 目录与模块说明

### 入口与公共模块

- `src/index.ts`：CLI 主入口与总编排层
- `src/config.ts`：环境变量读取与运行时配置，包括 JD 解析模型、评分模型、详情页超时与 SMTP 配置
- `src/types/job.ts`：JD、简历、评分、运行结果等共享类型
- `src/parsers/jd-parser.ts`：基于模型调用、JSON 文本解析与 schema 校验的 JD 标准化

### 浏览器相关

- `src/browser/session.ts`：浏览器启动、平台登录态加载、复用校验与关闭
- `src/browser/subscribe-search.ts`：51job 订阅页进入人才搜索页
- `src/browser/candidate-list.ts`：51job 结果列表候选人信息与 ID 提取，也负责将明确空结果文案（包括“没有搜索到相关的人才”这类筛选后空态）或稳定空列表识别为合法的 0 候选人就绪态，并在列表长期未渲染时输出诊断信息
- `src/browser/resume-detail.ts`：51job 简历详情页打开与解析；详情打开使用单个总 deadline 并并行观察弹窗、当前页跳转和当前页内容

### 平台适配

- `src/platforms/types.ts`：平台适配器接口与 `SupportedPlatform`
- `src/platforms/registry.ts`：平台注册、默认平台解析与 `--platform` 参数校验
- `src/platforms/51job-adapter.ts`：51job 平台适配器
- `src/platforms/liepin-adapter.ts`：Liepin 平台适配器，属于正式支持的平台实现；搜索页与详情页打开都使用总 deadline，候选列表提取 DOM-first，并以 `search-resumes` API 作为短 fallback
- `src/platforms/zhilian-adapter.ts`：Zhilian 平台适配器，属于正式支持的平台实现；列表提取 DOM-first，详情解析面向同页 modal

### 提取与评分

- `src/extraction/`：生产提取边界与可选 Crawl4AI 适配
- `src/scoring/score-resume.ts`：简历与 JD 的评分逻辑
- `src/scoring/score-prompt.ts`：评分提示词构造
- `src/scoring/score-schema.ts`：评分结构化输出校验与转换

### 存储与报告

- `src/storage/job-store.ts`：本地文件持久化
- `src/reporting/aggregate-results.ts`：结果聚合与 Markdown 渲染
- `src/reporting/mailer.ts`：SMTP 邮件发送与邮件标题/正文构建
- `src/scripts/export-job-results.ts`：导出最新结果报告
- `src/scripts/send-job-report-email.ts`：发送邮件报告

### 离线工具与辅助脚本

- `src/scripts/reparse-resumes.ts`：基于已保存快照重新解析简历
- `src/scripts/score-stored-resumes.ts`：对本地已保存简历重新评分
- `src/scripts/validate-resume-extraction.ts`：验证简历提取质量
- `src/scripts/capture-resume-dom-snapshot.ts`：补抓 DOM 快照
- `src/scripts/login-and-save-session.ts`：手动登录、保存会话并验证 fresh-session 可复用性
- `src/scripts/migrate-platform-storage.ts`：一次性回填历史职位记录和运行结果中的 `platform` 字段
- `src/scripts/debug-work-lines.ts`：调试工作经历文本切分
- `src/scripts/debug-work-boundaries.ts`：调试工作经历边界识别
- `src/scripts/debug-work-header-checks.ts`：调试工作经历头部识别

## 核心数据模型

项目中的核心数据结构定义在 `src/types/job.ts`，主要包含以下几类：

### 1. 职位数据

- `NormalizedJob`：结构化 JD
- `JobRecord`：职位持久化记录

`NormalizedJob` 主要包含：

- 标题、地点、部门
- 学历、专业、语言要求
- 职责、硬性要求、优先条件
- 最低经验年限
- 地域偏好、行业标签
- 年龄和薪资范围

`JobRecord` 在此基础上补充：

- `jobKey`
- `platform`
- `searchKeyword`
- `recipientEmail`
- `rawText`
- `createdAt`

### 2. 候选人列表数据

- `CandidateListItem`

它表示搜索结果页上的一张候选人卡片，包含：

- `candidateId`
- `resumeUrl`
- `name`
- `currentCompany`
- `currentTitle`
- `cardText`
- `sourceText`

### 3. 简历数据

- `CandidateResume`
- `WorkExperience`
- `ProjectExperience`
- `EducationExperience`
- `LanguageSkill`
- `ResumeDomSnapshot`

`CandidateResume` 是结构化简历主对象，当前字段包括：

- 基础信息：`candidateId`、`resumeUrl`、`candidateShareUrl`、`name`、`age`、`education`
- 地域与求职偏好：`regions`、`pr`
- 结构化经历：`workExperiences`、`projectExperiences`、`educationExperiences`
- 技能语言：`skill`
- 证书：`certificates`

`ResumeDomSnapshot` 主要用于保存从页面 DOM 中提取的工作经历辅助信息：

- `workLines`
- `workBlocks`
- `workNodes`

### 4. 评分与导出数据

- `CandidateScore`
- `CandidateScoreArtifact`
- `RunResult`
- `JobResultsMarkdownExport`

其中：

- `CandidateScore` 是一次成功评分的业务结果。
- `CandidateScoreArtifact` 是评分产物，既可以是成功记录，也可以是失败记录。
- Zhilian 的 `CandidateScoreArtifact` 会携带从简历详情复制出的 `candidateShareUrl`，用于邮件报告展示与校验。
- `RunResult` 表示一次执行的运行摘要，当前只保留轻量元数据与候选人 ID 列表，不再嵌入完整候选人卡片快照。
- `JobResultsMarkdownExport` 表示导出的 Markdown 结果视图。

## 本地数据结构

默认数据目录为 `data/`，每个职位的数据保存在：

`data/<platform>/jobs/<jobKey>/`

当前实际持久化结构包括：

- `jd.json`：职位原始文本、标准化 JD、平台标识、收件人邮箱等职位记录，也是后续 rerun 复用 JD 载荷的来源
- `seen-ids.json`：已处理候选人 ID 列表
- `resumes/<candidateId>.json`：结构化简历
- `snapshots/<candidateId>.txt`：原始页面文本快照
- `snapshots-dom/<candidateId>.json`：DOM 抽取快照
- `scores/<candidateId>.json`：候选人评分产物
- `results/<timestamp>.json`：单次运行结果摘要，包含平台、候选人 ID 列表和运行计数
- `exports/latest.md`：最新一次导出的 Markdown 报告，头部会带平台来源标识

对于早于平台化改造落盘的历史数据，可以执行一次 `npm run migrate:platform-storage`，它会先把旧的顶层 `data/jobs/` 目录项迁移到 `data/51job/jobs/`，再把缺失的 `platform` 字段统一回填为 `51job`；如果目标目录已存在，则会拒绝覆盖。

## 数据流说明

从运行过程来看，这个项目的数据生命周期是非常清晰的。一次执行大致会经过如下数据流：

`CLI 参数 / jobs 文件` → `NormalizedJob` → `JobRecord` → `候选人列表` → `新增候选人` → `结构化简历/原始快照/DOM 快照` → `seen-ids.json` → `评分产物` → `RunResult` → `Markdown 报告` → `邮件发送`

### 1. 输入层

入口参数来自 `src/index.ts`：

- `--platform`
- `--keyword`
- `--jd`
- `--jd-file`
- `--jobs-file`
- `--email`（可选）
- `--cc`（可选）

其中 JD 来源要求为：

- `--platform` 用于选择平台；当前正式支持的平台路径包括 `51job`、`liepin` 和 `zhilian`，也可传入 `all` 按 `51job -> liepin -> zhilian` 顺序串行执行所有平台。
- `--keyword` 始终必填。
- `--jd` 与 `--jd-file` 互斥，不能同时提供。
- **首次运行新 `jobKey`** 时，必须且只能提供一个 JD 来源。
- **命中已有 `jobKey` 的 rerun** 时，可以两个都不提供，系统会直接复用 `jd.json` 中已保存的 JD 原文与标准化结果。
- `--jd` 适合直接传入较短文本。
- `--jd-file` 适合从本地文件读取较长 JD，避免命令行引号与转义问题。
- `--jobs-file` 适合批量执行多个职位，文件格式为 JSON 数组；每个条目至少包含 `keyword`，可选 `jd`、`jdFile`、`email`、`cc`。
- `--jobs-file` 不能和 `--keyword`、`--jd`、`--jd-file` 同时使用。

系统会先根据搜索词生成 `jobKey` 并检查是否已有对应职位记录。若命中已有记录，则直接复用 `JobRecord.rawText` 与 `normalizedJob`，跳过 JD 文件读取和 `parseJobDescription()`；若未命中，再读取本次 JD 原文并解析成 `NormalizedJob`。`JobRecord.rawText` 持久化保存的是 JD 原文内容，而不是文件路径。

当 `--platform all` 被使用时，入口层会先展开支持平台列表，再逐个平台执行上述输入检查和职位记录处理。最终控制台输出是一个数组，每一项包含平台名和该平台原有的运行摘要。

当 `--jobs-file` 被使用时，入口层会先读取并校验 jobs JSON 数组，再按条目顺序执行。若 `--platform all` 同时被使用，控制台输出是一个扁平数组，顺序为第一个 job 的 51job、Liepin、Zhilian，再到第二个 job 的 51job、Liepin、Zhilian，依此类推。

### 2. 职位记录流

系统会读取已有职位记录，构造新的 `JobRecord`，并保存到：

- `data/<platform>/jobs/<jobKey>/jd.json`

这个文件是职位维度的核心元数据，后续重跑时不仅会复用历史平台、收件人邮箱和抄送列表，也会复用已保存的 `rawText` 与 `normalizedJob`。因此 rerun 会先检查 `jd.json` 是否存在，再决定是否需要读取本次 JD 输入并调用解析器。

### 3. 浏览器会话流

系统启动 Chromium，尝试读取当前平台对应的 storage state 文件，打开已鉴权首页，并校验当前登录态是否仍然有效。如果登录态缺失、过期或失效，普通 headed CLI 运行会自动打开对应平台的手动登录窗口，等待人工完成登录，保存新的 storage state，再用独立 fresh headless 会话复验可复用性；验证通过后会重新创建认证会话并继续当前任务。headless 模式无法交互登录，因此会直接失败并提示使用 `PLAYWRIGHT_HEADLESS=false` 重跑。`npm run login:session -- --platform 51job` 会在 `PLAYWRIGHT_HEADLESS=false` 下启动一个新的浏览器上下文，等待人工完成登录，保存 51job 的 storage state，然后再启动一个新的浏览器会话重新加载该保存态并验证是否仍可访问已鉴权入口；只有这一步成功后，命令才会输出成功信息。Liepin 与 Zhilian 的 `login:session` 也遵循同样的“先手动登录保存、再用独立 fresh session 复验”的模式，其中 Zhilian 默认使用 `storage-state.zhilian.json`，对应示例职位数据目录为 `data/zhilian/jobs/优衣库/`；首次运行该平台下的新 `jobKey` 仍然要求提供 `--jd` 或 `--jd-file`，而 rerun 会直接复用 `data/zhilian/jobs/优衣库/jd.json` 并跳过 JD 解析。若显式传入 `STORAGE_STATE_PATH`，代码会校验该路径是否与当前平台匹配，不允许把非 51job 平台指向共享 `storage-state.json`，也不允许把某个平台指向另一个平台命名的状态文件。

### 4. 搜索结果流

进入人才搜索页后，系统提取全部候选人卡片，得到 `CandidateListItem[]`。随后读取：

- `seen-ids.json`

只保留历史未处理过的候选人，形成当前运行的新增候选人集合。

### 5. 新增候选人抓取流

对每个新增候选人，系统会：

1. 打开简历详情页。
2. 提取结构化简历。
3. 读取页面正文文本。
4. 保存简历 JSON、正文快照、DOM 快照。

因此，一个候选人一旦详情提取成功，就会在本地留下：

- `resumes/<candidateId>.json`
- `snapshots/<candidateId>.txt`
- `snapshots-dom/<candidateId>.json`

所有新增候选人的抓取阶段结束后，系统会把成功保存简历的候选人写入 `seen-ids.json`，然后再从本地简历 JSON 进入评分阶段。

### 6. 去重更新语义

项目不会把所有抓到的候选人都立即写进 `seen-ids.json`，而是区分处理结果：

- **详情打开成功且简历已保存**：记为已处理。
- **详情打开失败或上游提取失败**：不记为已处理，下次仍可重试。

因此：

- **评分失败** 不会阻止候选人被标记为已处理。
- **提取失败** 会保留重试机会。

### 7. 运行结果流

成功抓取的候选人被写入 `seen-ids.json` 后，系统会从本地 `resumes/<candidateId>.json` 执行评分，并把每个候选人的评分产物写入：

- `scores/<candidateId>.json`

评分成功时产物状态为 `success`，评分失败时产物状态为 `failed` 并记录错误信息。随后系统会汇总生成 `RunResult` 并写入：

- `results/<timestamp>.json`

这个文件是一次运行级别的轻量摘要，而不是单个候选人的详细档案。当前会保存抓取时间、候选人总数、新增候选人 ID 列表、成功评分候选人 ID 列表，以及抓取失败或评分失败的候选人与错误信息。

### 8. 导出与邮件流

主流程结束后，系统会尝试：

1. 聚合评分结果并导出 Markdown 报告。
2. 如存在收件人邮箱，则发送报告邮件。

Markdown 报告写入：

- `exports/latest.md`

邮件发送则消费聚合后的导出结果，而不是重新计算整套评分。

## 调用链说明

### 1. 顶层入口

顶层入口是 `src/index.ts` 中的 `main()`。它负责串联整个流程：

1. 解析参数。
2. 如果传入 `--jobs-file`，读取 jobs JSON 数组并展开为一组待运行职位；否则按单职位参数运行。
3. 生成 `jobKey` 并检查是否存在已保存职位记录。
4. 仅在首次运行时读取并解析 JD；若命中已有 `jobKey`，则直接复用 `jd.json` 中已保存的 JD 载荷。
5. 读取和保存职位记录。
6. 建立已登录浏览器会话。
7. 运行抓取流程，保存新增简历并更新成功抓取候选人的去重状态。
8. 从本地简历执行评分并保存评分产物。
9. 导出 Markdown 报告。
10. 发送邮件。

### 2. 初始化调用链

初始化阶段的调用链可以概括为：

`main()`
→ `parseArgs()` / `loadBatchJobInputs()`
→ `buildJobKey()`
→ `new JobStore()`
→ `JobStore.readJobRecordIfExists()`
→ 若已有记录：直接复用 `rawText` / `normalizedJob`
→ 若无记录：读取 JD 原文（内联参数或 `--jd-file`）
→ `parseJobDescription()`
→ 模型返回 JSON 文本
→ 本地解析 JSON
→ schema 校验与字段清洗
→ `JobStore.saveJobRecord()`

这一段负责确定“这次要处理哪个职位”和“职位元数据如何持久化”。当前实现会优先按关键词查已有 `jobKey`，只有首次运行时才会读取 JD 输入并转换为可用于后续评分的标准化结构。

### 3. 浏览器会话调用链

浏览器初始化的核心入口是 `ensureAuthenticatedBrowserSession()`。其调用逻辑是：

`ensureAuthenticatedBrowserSession()`
→ `createBrowserSession()`
→ `chromium.launch()`
→ `createBrowserContext()`
→ 读取当前平台对应的 storage state（默认路径分别为 51job 的 `storage-state.json`、Liepin 的 `storage-state.liepin.json`、Zhilian 的 `storage-state.zhilian.json`）
→ `context.newPage()`
→ `openAuthenticatedSubscribePage()`
→ `assertAuthenticatedPage()`

这里不仅仅是启动浏览器，更重要的是验证保存态是否仍可用于实际抓取。若保存态已经过期，headed 模式会自动打开手动登录窗口，等待登录完成后保存新的 storage state，并用 fresh headless 会话验证可复用性，然后重新创建认证会话继续当前任务；headless 模式无法交互登录，会直接提示使用 `PLAYWRIGHT_HEADLESS=false` 重跑。

在 `--platform all` 模式下，上述浏览器会话调用链会对 51job、Liepin、Zhilian 依次执行；前一个平台的会话关闭后，才会开始下一个平台。

### 4. 订阅搜索页调用链

进入订阅搜索页后，系统通过 `src/browser/subscribe-search.ts` 完成以下工作：

- 等待订阅卡片加载完成。
- 根据搜索关键词查找对应订阅卡片。
- 支持精确匹配与前缀匹配。
- 悬浮卡片并定位搜索按钮。
- 兼容多种按钮和链接选择器。
- 由点击 helper 负责搜索按钮的 attached、visible 与加载遮罩检查，避免打开搜索页前重复等待。
- 点击后等待新页面弹出或当前页跳转。

因此这一层的职责不是“简单点击一个按钮”，而是把 51job 订阅页的非稳定页面结构收敛成一个稳定的“打开人才搜索页”动作。

### 5. 候选人列表调用链

结果页列表抓取主要由 `src/browser/candidate-list.ts` 完成：

- 等待候选人卡片可见、明确空结果文案出现，或空列表容器稳定超过配置窗口；51job 的“没有搜索到相关的人才”按明确空结果文案处理。
- 从 `div[id^="no_interested_"]` 等结构中收集候选人卡片。
- 从元素 ID、HTML、文本中用多组正则回退提取 `candidateId`。
- 尝试从卡片文本中推断姓名、当前公司、当前职位。
- 返回 `CandidateListItem[]`。

这一层说明项目对 51job 列表页结构做了多重兼容，而不是依赖单一稳定选择器。

### 6. 简历详情与评分调用链

新增候选人的详情抓取主逻辑位于 `captureCandidateResume()`。它会：

`captureCandidateResume()`
→ `openResumeDetail()`
→ `extractResumeFromPage()`
→ 读取 `body.innerText()`
→ `saveCandidateResume()`
→ 关闭详情页

整批新增候选人抓取结束后，`runResumeCaptureFlow()` 会把成功抓取的候选人写入 `seen-ids.json`，再调用 `scoreCapturedResumes()`：

`scoreCapturedResumes()`
→ 读取本地 `resumes/<candidateId>.json`
→ `scoreResumeAgainstJob()`
→ `saveCandidateScoreArtifact()`

其中 `src/browser/resume-detail.ts` 是项目中最启发式、最复杂的模块之一。它会同时结合：

- 页面对文本的整体切片
- DOM 中工作经历相关节点
- 中文字段与分节启发式规则

去恢复结构化简历。

### 7. 评分调用链

评分入口在 `src/scoring/score-resume.ts`。调用过程大致是：

`scoreResumeAgainstJob()`
→ `buildScorePrompt()`
→ 调用 OpenAI Responses 接口
→ 读取返回的 JSON 文本
→ 用 schema 校验解析结果
→ 转为 `CandidateScore`

当前评分逻辑依赖 OpenAI SDK 与本地 schema 校验，不是自由文本打分，也不是从普通文本里再二次提取 JSON。与 JD 解析类似，它同样依赖 OpenAI 格式的模型端点；运行时需要 `OPENAI_API_KEY`，`SCORING_MODEL` 会覆盖 `OPENAI_MODEL`，并可通过环境变量配置 base URL。

### 8. 导出调用链

导出逻辑通过 `src/scripts/export-job-results.ts` 调起，底层依赖：

- 读取职位记录
- 读取最新一次运行结果
- 读取评分产物
- 过滤出本次运行涉及候选人的评分产物
- `src/reporting/aggregate-results.ts` 聚合本次运行评分
- 渲染 Markdown
- 写入 `exports/latest.md`

`aggregate-results.ts` 会对同一个候选人的多次评分按时间取最新结果，再按评分高低排序，最终产出面向人阅读的报告。

Zhilian 邮件报告会传入“优先展示分享链接”的渲染选项；聚合数据会保留评分产物上的 `candidateShareUrl`，渲染时用它替代内部候选人 ID。

### 9. 邮件调用链

邮件发送逻辑通过 `src/scripts/send-job-report-email.ts` 调起，底层由 `src/reporting/mailer.ts` 提供：

- SMTP 配置校验
- transport 创建
- 邮件主题生成
- 文本正文发送

系统当前已支持：

- 正常评分报告邮件
- 无新增候选人的通知邮件

当本次存在新增候选人时，邮件脚本会重新读取最新运行结果并过滤出本次运行涉及候选人的评分产物，再聚合为邮件正文。

对于 Zhilian，本次运行的评分产物必须包含唯一的 `candidateShareUrl`。如果某个当前运行候选人缺失复制链接，或两个候选人拿到了同一个复制链接，邮件脚本会报错并让主流程在 summary 中记录邮件失败。

## 模块职责与稳定性理解

从当前实现来看，模块大致可以分为两类：

### 1. 相对稳定的核心层

这些模块的职责边界较清晰，结构相对稳定：

- `src/index.ts`
- `src/storage/job-store.ts`
- `src/types/job.ts`
- `src/reporting/aggregate-results.ts`
- `src/reporting/mailer.ts`

它们主要承担编排、数据契约、持久化、导出和邮件发送，不太受页面结构波动影响。

### 2. 高风险启发式层

这些模块更容易随着页面变化或样本变化而失稳：

- `src/browser/subscribe-search.ts`
- `src/browser/candidate-list.ts`
- `src/browser/resume-detail.ts`
- `src/extraction/`

其中 `resume-detail.ts` 是最复杂、最值得谨慎修改的部分。改动这类模块时，优先建议：

- 先使用本地快照验证
- 优先离线重解析验证效果
- 尽量保留现有多重回退逻辑
- 不要过早假设页面结构稳定

## 运行要求

### 必需条件

- Node.js 24 LTS 或 Node.js 26 环境；如需推荐组合，可使用仓库根目录 `.nvmrc` 切换到 Node 24
- 已安装依赖：`npm install`
- 可用的目标平台 storage state：51job 默认使用 `storage-state.json`，Liepin 默认使用 `storage-state.liepin.json`，Zhilian 默认使用 `storage-state.zhilian.json`；推荐通过 `npm run login:session -- --platform <platform>` 生成或刷新
- 可用的模型 API 凭据：`OPENAI_API_KEY`

### 运行时版本

- 推荐使用 Node 24 LTS
- 根目录 `.nvmrc` 固定为 `24`
- `package.json` 声明支持范围为 `>=24 <27`
- Node 26 也受支持；源码脚本通过 `node --import ./scripts/node-ts-hooks.mjs` 执行，不再依赖会触发 `DEP0205` 的 `tsx` 源码运行路径

### 可选条件

- 若要启用 Crawl4AI 增强提取，需要仓库内 `.venv/bin/python` 可用，且已安装 `crawl4ai`
- 若不可用，系统会回退到内置解析路径，不阻塞主流程

## 邮件与运行语义说明

### 收件人持久化

- 首次运行传入 `--email` 时，会把收件人保存到对应职位的 `jd.json`
- 后续同一 `jobKey` 重跑时，如未再传 `--email`，会自动复用已保存收件人
- 如果显式再次传入 `--email`，则以本次传入值为准并覆盖保存值

### 抄送持久化

- 首次运行传入 `--cc` 时，会把抄送列表保存到对应职位的 `jd.json`
- 后续同一 `jobKey` 重跑时，如未再传 `--cc`，会自动复用已保存抄送列表
- 如果显式再次传入 `--cc`，则以本次传入值为准并覆盖保存值

### 失败处理语义

当前主流程对不同失败场景做了区分：

- **评分失败**：候选人仍会保存简历与失败评分产物，并计入失败列表
- **上游提取失败**：候选人不会被标记为已处理，以便下次继续重试
- **导出失败**：不会让整次运行失败，错误会记录到最终 summary
- **邮件失败**：不会让整次运行失败，错误会记录到最终 summary
- **Zhilian 分享链接缺失或重复**：有新增候选人的 Zhilian 报告邮件会失败并记录到 summary；候选人抓取、评分和运行结果仍按已完成状态保留
- **空结果运行**：如果本次没有新增候选人，或 51job 因筛选条件显示“没有搜索到相关的人才”，仍可发送专门的“无新增候选人”通知邮件

## 当前项目定位

这是一个 CLI-first 的 MVP 工程，重点在于把 51job、Liepin 和 Zhilian 招聘搜索下的半自动筛选流程落成可运行、可验证、可离线复查的本地工具。当前代码已经不再停留在设计阶段，而是具备可执行的抓取、解析、评分、导出和邮件发送能力。
